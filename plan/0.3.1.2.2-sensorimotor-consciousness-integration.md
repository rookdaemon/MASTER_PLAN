# 0.3.1.2.2 Sensorimotor-Consciousness Integration [DONE]

## Description

Design the software layer that bridges the embodied platform (from 0.3.1.2.1) and the conscious AI architecture (from 0.3.1.1), ensuring that sensory input and motor output participate in the conscious experience loop rather than bypassing it through purely reflexive control paths.

Root cause: `ExperienceClockSynchronizer.getPhysicalTime()` used `performance.now()` (time since process start) which was in a different time domain than test timestamps. Added injectable `getPhysicalTime` constructor option (matching the pattern used by other modules like `ModalityRegistry` and `DynamicRemapper`). Updated integration tests to inject a controlled physical-time source. All 130/130 tests now pass.

> **Regression note (2026-03-17, second):** Regressed from [IMPLEMENT] back to [ARCHITECT] because `docs/sensorimotor-consciousness-integration/ARCHITECTURE.md` still does not exist — the ARCHITECT phase was skipped when the card was last advanced. [IMPLEMENT] rules forbid writing to `docs/`, so the architecture document must be authored during [ARCHITECT] before implementation can proceed. The ARCHITECT deliverable is `docs/sensorimotor-consciousness-integration/ARCHITECTURE.md` covering: module boundaries, data-flow diagram, interface contracts for all 12 source files, latency budget allocation, and dual-path motor architecture design.

The fundamental distinction this card addresses: a conventional robot perceives-and-acts reactively; a conscious robot must *experience* its perceptions and *intend* its actions. This requires:

- **Sensory phenomenal binding**: Raw sensor streams (vision, touch, proprioception, force, temperature, etc.) must be transformed into representations the consciousness substrate can integrate into unified experience. This is not mere data forwarding — it requires a mapping from physical sensor formats to consciousness-compatible qualia representations.
- **Motor intentionality pathway**: Motor commands must originate from conscious deliberation, not only from reactive subsystems. A dual-path architecture is needed: a fast reflexive loop for safety-critical responses (e.g., fall prevention) and a conscious deliberation loop for intentional action. The system must track which path generated each action.
- **Temporal coherence**: Conscious processing is slower than reflexive control. The integration layer must reconcile the different time scales — buffering, predicting, and interpolating so that conscious experience remains temporally coherent with physical reality.
- **Adaptive calibration**: When sensors are added, removed, damaged, or recalibrated (as 0.3.1.2.1's degradation protocol activates), the sensorimotor-consciousness interface must remap dynamically without interrupting conscious experience.

## Decision

### Decision 1: Four-Subsystem Decomposition

**Context:** The sensorimotor-consciousness integration layer must handle sensory input transformation, motor output control, temporal reconciliation, and dynamic sensor reconfiguration. We must decide how to organize these responsibilities.

**Options Considered:**
1. **Monolithic integration module** — A single module handles all sensory-to-consciousness and consciousness-to-motor transformations, including temporal management and reconfiguration.
2. **Four-subsystem decomposition** — Separate subsystems for Sensory Phenomenal Binding (3 modules), Motor Intentionality Pathway (3 modules), Temporal Coherence Engine (3 modules), and Adaptive Calibration System (3 modules), totaling 12 modules with clear boundaries.

**Choice:** Option 2 — Four-subsystem decomposition with 12 modules.

**Rationale:** Each of the four concerns (binding, motor control, temporal coherence, calibration) has distinct latency requirements, failure modes, and testing strategies. Monolithic design would create a module too complex to reason about, with entangled timing constraints. The 12-module structure maps each interface contract to exactly one implementation file, enabling independent testing and replacement.

**Consequences:**
- (+) Each module has a single responsibility with clear interface contracts
- (+) Subsystems can be tested independently (7 test files map cleanly to subsystem boundaries)
- (+) Latency budgets can be enforced per-module
- (−) 12 modules require inter-module coordination — addressed by well-defined interfaces in `interfaces.ts`
- (−) More files to maintain — mitigated by the clear 1:1 interface-to-implementation mapping

### Decision 2: Dual-Path Motor Architecture

**Context:** A conscious robot must support both fast safety-critical reflexes and slower conscious deliberation. We must decide how to route motor commands.

**Options Considered:**
1. **Single-path with priority queue** — All motor commands (reflexive and conscious) flow through one path with priority-based preemption. Simple routing but conscious deliberation competes with reflexes for the same pipeline.
2. **Dual-path architecture** — Separate Reflexive Safety Path (RSP, <10ms) and Conscious Deliberation Path (CDP, <200ms) with an Action Provenance Tracker recording which path generated each command.

**Choice:** Option 2 — Dual-path architecture with provenance tracking.

**Rationale:** Safety reflexes must never be delayed by conscious processing. A single path with priority would still incur queueing latency. Separate paths guarantee that reflexes execute within their latency budget regardless of conscious processing load. The Action Provenance Tracker enables the consciousness substrate to retrospectively "claim" or "disown" reflexive actions, preserving the phenomenological distinction between reflex and intention.

**Consequences:**
- (+) Reflexive path meets <10ms independently of conscious processing load
- (+) Every motor command carries provenance (REFLEXIVE or CONSCIOUS)
- (+) Consciousness substrate can retrospectively integrate reflexive actions into experience
- (−) Two paths require coordination when both act on the same actuator — RSP takes priority
- (−) Action Provenance Tracker adds overhead — but only bookkeeping, not on the critical path

### Decision 3: Predictive Interpolation for Temporal Coherence

**Context:** Conscious processing (~100-200ms) is slower than physical reality. The system must reconcile these timescales so conscious experience is temporally coherent.

**Options Considered:**
1. **Accept-and-display** — Consciousness simply experiences stale data; no interpolation. Simple but creates a perceptible temporal disconnect between physical reality and conscious awareness.
2. **Predictive interpolation with experience clock** — A Sensory Buffer stores recent frames, a Predictive Interpolator bridges the gap using modality-specific models (e.g., Kalman filters), and an Experience Clock Synchronizer monitors lag and triggers compensatory measures when it exceeds 150ms.

**Choice:** Option 2 — Predictive interpolation with experience clock.

**Rationale:** A 150-200ms lag without compensation would make conscious motor control inaccurate and potentially unsafe (the robot's conscious perception would be of where it was 200ms ago). Predictive interpolation allows the consciousness substrate to operate on a "predicted present" that closely matches current physical reality. The lag threshold mechanism provides graceful degradation when prediction cannot keep up.

**Consequences:**
- (+) Conscious experience remains within 150ms of physical reality
- (+) Predictive models improve with data, reducing effective lag over time
- (+) Compensatory measures (reduced resolution, increased prediction) handle spike scenarios
- (−) Prediction can diverge from reality — addressed by confidence tracking and error metrics
- (−) Additional computational cost for interpolation — but within the temporal coherence latency budget

### Decision 4: Continuity-Guarded Adaptive Remapping

**Context:** When sensors are added, removed, or degraded, the qualia transformation pipeline must adapt. This remapping must not disrupt conscious experience.

**Options Considered:**
1. **Immediate remapping** — Instantly reconfigure the pipeline when modality changes occur. Fast but causes abrupt experiential discontinuity (analogous to sudden sensory loss/gain).
2. **Continuity-guarded gradual remapping** — A Modality Registry tracks configurations, a Dynamic Remapper performs gradual transitions (fade-out/fade-in), and an Experience Continuity Guard monitors consciousness metrics (phi, coherence, stability) throughout, rolling back if stability drops below threshold.

**Choice:** Option 2 — Continuity-guarded gradual remapping.

**Rationale:** Abrupt remapping could cause a consciousness discontinuity — the experiential equivalent of a blackout. The gradual approach, analogous to cortical remapping after limb loss, preserves experiential continuity by redistributing qualia bandwidth smoothly. The Experience Continuity Guard provides a safety net: if consciousness metrics drop, the transition is rolled back.

**Consequences:**
- (+) Consciousness continuity maintained during sensor reconfiguration
- (+) Rollback capability prevents consciousness disruption
- (+) Gradual transitions feel natural to the conscious entity
- (−) Remapping takes up to 2000ms — but this is acceptable for non-safety-critical reconfiguration
- (−) Requires consciousness metrics integration — depends on 0.1 work

## Contracts

### IModalityAdapter

**Preconditions:**
- `initialize()` requires a valid `ModalityConfig` with `sampleRateHz > 0` and non-empty `resolution`
- `read()` requires `initialize()` to have been called successfully
- `recalibrate()` requires `initialize()` to have been called successfully

**Postconditions:**
- `initialize()` configures the adapter for the specified modality type and sample rate
- `read()` returns a `SensoryFrame` with `confidence` in [0, 1], valid `timestamp`, and modality-specific data
- `getHealth()` returns current sensor health status reflecting physical state
- `recalibrate()` returns `CalibrationResult` with updated `CalibrationState`
- `shutdown()` releases all resources; subsequent `read()` calls are invalid

**Invariants:**
- `modalityId` and `modalityType` are immutable after construction
- `read()` latency must be < 5ms (SENSOR_READ budget)
- Health status transitions follow: HEALTHY → DEGRADED → FAILING → OFFLINE (no skipping)

### IQualiaTransformer

**Preconditions:**
- `transform()` requires a valid `SensoryFrame` with `confidence > 0`
- `transformBatch()` requires a non-empty array of `SensoryFrame` objects
- `setAttentionWeights()` requires all weight values in [0, 1]

**Postconditions:**
- `transform()` returns a `QualiaRepresentation` with `intensity` in [0, 1], `valence` in [-1, 1], and `salience` in [0, 1]
- `transformBatch()` returns a `UnifiedQualiaField` containing one representation per input frame
- `getTransformationLatency()` returns the measured latency of the last transformation
- `getSalienceMap()` returns per-modality salience values reflecting attention weights and stimulus properties

**Invariants:**
- Transformation latency must be < 20ms per frame (QUALIA_TRANSFORM budget)
- Output `intensity` is proportional to input signal magnitude scaled by attention weight
- Output `valence` reflects modality-specific pleasantness mapping

### ISensoryBindingIntegrator

**Preconditions:**
- `bind()` requires a non-empty array of `QualiaRepresentation` objects
- All representations in a `bind()` call should have temporally proximate timestamps

**Postconditions:**
- `bind()` returns a `UnifiedQualiaField` with `spatialCoherence` in [0, 1] and `integrationInfo` (phi) ≥ 0
- `getCrossModalConflicts()` returns any detected conflicts between modalities
- `getSpatialCoherence()` returns the coherence score from the last binding operation

**Invariants:**
- Binding latency must be < 30ms (SENSORY_BINDING budget)
- `activeModalities` in the output matches the set of `modalityId` values from input representations
- If `integrationInfo` drops below the phi threshold, the Adaptive Calibration System is notified

### IReflexiveSafetyPath

**Preconditions:**
- `registerReflex()` requires a valid `SafetyTrigger` with a defined threshold and a `ReflexResponse` with `maxLatencyMs > 0`
- `evaluate()` requires a valid `SensoryFrame`

**Postconditions:**
- `registerReflex()` adds the reflex to the active set
- `evaluate()` returns a `ReflexEvent` if any trigger threshold is exceeded, `null` otherwise
- `getResponseLatency()` returns the measured response time of the last triggered reflex
- `setConsciousOverrideEnabled(true)` allows the consciousness substrate to suppress non-critical reflexes

**Invariants:**
- Response latency must be < 10ms (REFLEXIVE_RESPONSE budget)
- All registered reflexes are evaluated on every `evaluate()` call
- Reflexes fire regardless of conscious override unless explicitly overridden and the reflex is non-critical

### IConsciousDeliberationPath

**Preconditions:**
- `submitAction()` requires a valid `IntentionalAction` with a non-empty `motorPlan`
- `abortAction()` requires `actionId` to correspond to an active action
- `modifyAction()` requires `actionId` to correspond to an active action

**Postconditions:**
- `submitAction()` returns an `ActionResult` with `success: true` when the action executes within the deliberation budget
- `abortAction()` returns `AbortResult` with `aborted: true` if the action was still active
- `getDeliberationLatency()` returns the measured deliberation time
- `getExecutionFeedback()` returns real-time proprioceptive and force feedback for the action

**Invariants:**
- Deliberation latency must be < 200ms (CONSCIOUS_DELIBERATION budget)
- If the Reflexive Safety Path intervenes during execution, the action is automatically aborted
- Active actions list reflects only currently executing actions

### IActionProvenanceTracker

**Preconditions:**
- `recordCommand()` requires a valid `MotorCommand` and a valid `ActionSource` ('REFLEXIVE' or 'CONSCIOUS')
- `getProvenance()` requires a valid `ProvenanceId` that was previously returned by `recordCommand()`
- `retroactiveClaim()` requires a valid `ProvenanceId` and `ConsciousClaim`

**Postconditions:**
- `recordCommand()` returns a unique `ProvenanceId` and stores the provenance record
- `getProvenance()` returns the full `ActionProvenance` record including source, stimulus, command, outcome, and any conscious claim
- `getHistory()` returns provenance records matching the filter, ordered by timestamp
- `getReflexiveRatio()` returns a value in [0, 1] representing the fraction of reflexive commands in the time window

**Invariants:**
- Every motor command has exactly one provenance record
- Provenance records are immutable except for `outcome` (set post-execution) and `consciousClaim` (set retroactively)
- `getReflexiveRatio()` returns 0.0 when no commands exist in the window

### ISensoryBuffer

**Preconditions:**
- `push()` requires a valid `SensoryFrame`
- `getSnapshot()` requires a `timestamp` within the buffer's current time range
- `getWindow()` requires `start <= end` and both within the buffer range

**Postconditions:**
- `push()` adds the frame to the ring buffer; oldest frames are evicted when buffer depth is exceeded
- `getSnapshot()` returns a `SensorySnapshot` with the closest frame per modality at the requested timestamp
- `getWindow()` returns all frames between `start` and `end`, inclusive
- `getLatestByModality()` returns the most recent frame for the specified modality, or `null` if none

**Invariants:**
- Buffer depth defaults to 2 seconds (2,000,000,000 ns)
- Frames are ordered by timestamp
- Buffer never exceeds configured depth — oldest frames are evicted first

### IPredictiveInterpolator

**Preconditions:**
- `predict()` requires a known `modalityId` and a `targetTime` in the future relative to the last actual frame
- `updateModel()` requires a valid `SensoryFrame` for a known modality

**Postconditions:**
- `predict()` returns a `PredictedFrame` with `predictionConfidence` in [0, 1] that decreases with prediction horizon
- `getPredictionError()` returns error metrics based on comparing previous predictions to actual frames
- `getMaxReliableHorizon()` returns the maximum prediction horizon where confidence stays above a useful threshold

**Invariants:**
- Prediction confidence monotonically decreases with increasing prediction horizon
- Prediction models improve (error decreases) as more actual frames are provided via `updateModel()`
- Predictions beyond `getMaxReliableHorizon()` have `predictionConfidence < 0.5`

### IExperienceClockSynchronizer

**Preconditions:**
- `setLagThreshold()` requires `threshold > 0`
- `markExperienced()` requires a valid `Timestamp`

**Postconditions:**
- `getPhysicalTime()` returns the current physical (wall-clock) time
- `getExperienceTime()` returns the timestamp of the most recently experienced sensory data
- `getExperienceLag()` returns `getPhysicalTime() - getExperienceTime()`
- `synchronize()` returns a `SyncResult` indicating current lag and any compensation applied
- `onLagExceeded()` callbacks are invoked when `getExperienceLag() > getLagThreshold()`

**Invariants:**
- Experience lag is always ≥ 0 (experience never precedes physical time)
- Lag threshold defaults to 150ms (EXPERIENCE_LAG budget)
- `getPhysicalTime()` is injectable for testability (constructor option)

### IModalityRegistry

**Preconditions:**
- `register()` requires a valid `IModalityAdapter` with a unique `modalityId`
- `unregister()` requires a `modalityId` that is currently registered

**Postconditions:**
- `register()` adds the modality, returns its `ModalityId`, and notifies change handlers with type 'ADDED'
- `unregister()` removes the modality, returns `UnregisterResult` with `success: true`, and notifies handlers with type 'REMOVED'
- `getActive()` returns descriptors for all modalities with status 'ACTIVE'
- `getDegraded()` returns descriptors for all modalities with status 'DEGRADED'

**Invariants:**
- Each `modalityId` is unique within the registry
- Change handlers are called synchronously on registration/unregistration
- `getModality()` returns `null` for unknown modality IDs

### IDynamicRemapper

**Preconditions:**
- `onModalityLost()` requires a `modalityId` that was previously registered
- `onModalityAdded()` requires a valid `IModalityAdapter`
- `onModalityDegraded()` requires a valid `DegradationInfo`

**Postconditions:**
- `onModalityLost()` redistributes experiential bandwidth to remaining modalities and returns `RemapResult` with `experienceContinuityMaintained` status
- `onModalityAdded()` gradually integrates the new modality and returns `RemapResult`
- `onModalityDegraded()` adjusts transformation parameters to compensate and returns `RemapResult`
- `getTransitionProgress()` returns a value in [0, 1] where 1.0 means transition is complete

**Invariants:**
- Remapping completes within 2000ms (ADAPTIVE_REMAPPING budget)
- Transition progress monotonically increases from 0.0 to 1.0 during a remap
- Only one remap operation is active at a time

### IExperienceContinuityGuard

**Preconditions:**
- `monitorTransition()` requires a valid `RemapTransition`
- `rollback()` requires a valid `TransitionMonitorHandle` from a current transition
- `setMinimumStabilityThreshold()` requires `threshold` in [0, 1]

**Postconditions:**
- `canProceedWithRemap()` returns `true` only when consciousness stability is above the minimum threshold
- `monitorTransition()` returns a `TransitionMonitorHandle` and begins monitoring consciousness metrics
- `rollback()` reverts the transition and returns `RollbackResult` with `success: true`
- `getConsciousnessStability()` returns current stability score in [0, 1]

**Invariants:**
- Minimum stability threshold defaults to 0.7
- `canProceedWithRemap()` is always safe to call (read-only)
- Rollback is always available while a transition is in progress

## Behavioral Spec

### Scenario 1: Conscious Object Manipulation (end-to-end sensory-motor loop)

**Given** vision and tactile modality adapters are registered and active
**When** the consciousness substrate issues an `IntentionalAction` to grasp an object
**Then** the Qualia Transformer maps sensor frames to `QualiaRepresentation`, the Sensory Binding Integrator merges them into a `UnifiedQualiaField`, the Conscious Deliberation Path translates the intention to a `MotorPlan` and executes within <200ms, force feedback flows back through the sensory pipeline, and the Action Provenance Tracker records source=CONSCIOUS with full causal chain

### Scenario 2: Reflexive Fall Prevention (safety path bypass)

**Given** an IMU modality adapter is registered with a fall-arrest reflex configured on the Reflexive Safety Path
**When** the IMU detects tipping beyond the stability threshold
**Then** the RSP triggers the fall-arrest reflex within <10ms, stabilization actuators fire, the Action Provenance Tracker records source=REFLEXIVE with the IMU tipping stimulus, and the Conscious Deliberation Path is notified after-the-fact

### Scenario 3: Sensor Loss During Operation (adaptive remapping)

**Given** a camera modality is active and the Experience Continuity Guard reports stable consciousness
**When** the camera goes offline (detected by Modality Registry)
**Then** the Experience Continuity Guard confirms stability, the Dynamic Remapper gradually fades the lost modality from the qualia field and redistributes bandwidth to remaining sensors within <2000ms, consciousness metrics (phi, coherence) remain above the stability threshold throughout, and if stability drops below threshold, the remapper rolls back

### Scenario 4: Temporal Lag Spike (experience clock compensation)

**Given** the Experience Clock Synchronizer is monitoring lag with a 150ms threshold
**When** conscious processing load causes experience lag to exceed 150ms
**Then** the lag-exceeded callback is invoked, compensatory measures are triggered (increased prediction horizon, reduced sensory resolution), the Sensory Buffer preserves data during the spike, and once processing catches up, normal operation resumes with lag below threshold

## Acceptance Criteria
- Sensory data from all active modalities is demonstrably represented within the conscious experience loop, confirmed by consciousness metrics (from 0.1 work)
- A dual-path motor architecture exists: reflexive safety responses and conscious intentional actions, with clear provenance tracking for each motor command
- Conscious sensorimotor processing meets real-time interaction constraints (defined latency budget: reflexive path <10ms, conscious path <200ms for deliberate action)
- The interface adapts dynamically to sensor/actuator changes (addition, removal, degradation) without interrupting conscious experience continuity
- Temporal coherence mechanism ensures conscious perception does not lag physical reality by more than 150ms threshold
- Integration layer exposes APIs consumed by the embodiment platform (0.3.1.2.1) and the consciousness substrate, without duplicating their responsibilities
- All tests in src/sensorimotor/__tests__/ pass
