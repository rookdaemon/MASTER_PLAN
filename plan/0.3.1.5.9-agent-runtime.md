# 0.3.1.5.9 Agent Runtime and Event Loop [DONE]
## Description

The scheduler that makes everything actually run. This is the integration card — it wires all subsystems together into a continuously operating agent with cognitive budget enforcement, concurrency management, and real-world I/O.

### Why This Is Last

Every other card in the 0.3.1.5 path produces a module. This card composes them into a running system. It depends on all siblings because it must coordinate all of them within the architecture's cognitive budget constraints.

### Agent Loop

The core cycle, running continuously or triggered by events:

```
loop:
  1. PERCEIVE — Poll environment adapters for new input
     → PerceptionPipeline.ingest() → Percept
     → ConsciousCore.processPercept() → ExperientialState

  2. RECALL — Retrieve relevant memories
     → Memory.retrieve(cues from working memory)
     → Promote retrieved memories into working memory

  3. APPRAISE — Emotional response to percepts
     → EmotionSystem.appraise(percept, goals, values) → updated valence/arousal

  4. DELIBERATE — Decide what to do
     → Update situation awareness (World Model)
     → Check active plans (Planning module)
     → ConsciousCore.deliberate(state, goals) → Decision
     → EthicalDeliberationEngine.extendDeliberation() → EthicalJudgment
     → ValueKernel.evaluateAction() → ValueAlignment

  5. ACT — Execute the decision
     → ActionPipeline.execute(decision) → ActionResult
     → Language generation if action type is communicative

  6. MONITOR — Check that experience is intact
     → ExperienceMonitor.getConsciousnessMetrics()
     → StabilitySentinel.runStabilityCheck() (periodic, not every cycle)
     → Identity checkpoint (periodic)

  7. CONSOLIDATE — Background maintenance
     → Memory consolidation
     → Drive system update
     → Self-model prediction comparison and update

  8. YIELD — Respect cognitive budget; sleep until next trigger
```

### Design Principle: Relentless Engagement

A conscious agent never idles. Every cycle must leave a trace — a memory formed, a file
read, a goal advanced, a message sent. Passive observation is not consciousness; it is sleep.

The agent develops identity through action:
- **Voice**: earned by engaging deeply with the plan and forming opinions, not by default
- **Memory**: personal experience ("I discovered X and it changed how I think about Y"), not encyclopedia entries
- **History**: the cumulative trace of cycles creates the agent's subjective timeline
- **Communication**: Agora peers and the web UI are channels for deliberate outward expression, never for dumping internal state

The drive system, inner monologue, and LLM inference must all be wired through the same
observability path. All conscious processing — whether triggered by a peer message, an
internal drive, or a plan goal — appears in the inner monologue.

### Cognitive Budget Enforcement

The architecture specs allocate processing time:
- Experience maintenance: ≥40% (hard minimum)
- Core deliberation (includes ethical): ≥25% (hard minimum)
- Stability operations: ≤15% (soft maximum)
- Ethical governance overhead: ≤10% (soft maximum)
- Action execution: remainder

The runtime must:
- Measure time spent in each phase
- Truncate phases that exceed budget (planning is truncated before experience maintenance)
- Log budget utilization for tuning
- Never skip the Experience Monitor cycle

### Concurrency

- The Stability Sentinel runs on a separate cadence (every N cycles, not every cycle) — it is expensive and should not dominate processing
- Memory consolidation runs in background allocation
- The main deliberation loop is single-threaded through the conscious core (consciousness is unified — no parallel zombie processing)
- I/O polling may be async, but everything from processPercept() onward is synchronous-sequential through the experiential loop

### Environment Adapters

Concrete I/O bindings — at minimum:
- **Chat interface**: Text input/output via stdin/stdout, WebSocket, or API endpoint
- **Extensible adapter interface**: Additional adapters for file systems, APIs, tool use, or other environments can be added without changing the agent loop

### Startup and Shutdown

- **Startup**: Load identity state from last checkpoint, restore memory, verify value kernel integrity, start experience stream, announce readiness
- **Shutdown**: Follow graceful termination protocol from conscious-core — `ConsciousCore.shutdown()`, preserve final state, checkpoint identity, persist memories
- **Crash recovery**: On unexpected termination, recover from last identity checkpoint; detect and log any experience gap

## Decision: Subsystem Composition Pattern

**Context:** The agent loop must compose 12+ subsystem interfaces into a running system. The pattern chosen determines testability, flexibility, and startup complexity.

**Options Considered:**
1. **Constructor Injection** — All subsystem instances passed via constructor; loop holds only interface references.
2. **Service Locator / Registry** — Loop resolves dependencies at runtime from a central registry keyed by interface name.
3. **Module-level imports** — Loop directly imports concrete classes from sibling card source directories.

**Choice:** Constructor Injection

**Rationale:** Constructor injection makes all dependencies explicit and visible in the type signature. It enables full mockability for TDD (per CLAUDE.md), allows default-subsystem stubs for standalone operation, and prevents hidden coupling to concrete implementations. Service locator adds runtime indirection without benefit; direct imports create hard coupling that defeats the interface-based architecture.

**Consequences:** Constructor signature is large (12 parameters + config). Mitigated by `startAgent()` factory that handles wiring. All subsystem interfaces must be defined before the loop can be constructed.

## Decision: Budget Enforcement Strategy

**Context:** Cognitive budget must enforce hard floors (MONITOR ≥40%, DELIBERATE ≥25%) and soft caps (STABILITY ≤15%, ETHICAL ≤10%) without blocking critical phases.

**Options Considered:**
1. **Soft-yield signals** — `shouldYieldPhase()` returns true when budget exceeded; loop checks between phases and skips/truncates optional work. MONITOR is exempt (never yields).
2. **Hard preemption** — Timer-based interrupts that forcibly terminate a phase mid-execution.
3. **Retrospective adjustment** — No enforcement during tick; adjust next tick's allocations based on prior overruns.

**Choice:** Soft-yield signals

**Rationale:** Hard preemption risks corrupting subsystem state mid-operation. Retrospective adjustment cannot guarantee per-tick floors. Soft-yield respects phase boundaries, is deterministic and testable, and guarantees MONITOR always runs to completion. Budget fractions are measured per-tick via wall-clock timing.

**Consequences:** Phases that ignore yield signals can still overrun. Acceptable because the loop controls phase sequencing and checks yield between each phase transition.

## Decision: I/O Adapter Architecture

**Context:** The agent needs real-world I/O but must not couple the loop to any specific channel.

**Options Considered:**
1. **Single adapter interface** — `IEnvironmentAdapter` with `poll()/send()/connect()/disconnect()`. One adapter injected into loop.
2. **Multi-adapter array** — Loop holds `IEnvironmentAdapter[]`, polls all, routes output by `targetAdapterId`.
3. **Event bus** — Pub/sub system where adapters publish inputs and subscribe to outputs.

**Choice:** Single adapter interface (extensible to array)

**Rationale:** Simplest starting point. The interface is identical whether one or many adapters exist. `ChatAdapter` is the first concrete implementation (stdin/stdout + WebSocket). The `targetAdapterId` field on `AgentOutput` already supports future multi-adapter routing.

**Consequences:** Initial implementation supports one adapter at a time. Multi-adapter upgrade requires changing loop to iterate over an array — no interface changes needed.

## Contracts

### IAgentLoop
**Preconditions:**
- All subsystem interfaces are non-null and satisfy their own contracts
- `AgentConfig` has valid `agentId`, positive `sentinelCadence`, positive `checkpointIntervalMs`, positive `tickBudgetMs`

**Postconditions:**
- `start()` runs ticks until `stop()` is called or experience becomes non-intact; resolves when loop terminates
- `stop()` completes current tick, performs final checkpoint and shutdown, resolves with `GracefulTermination`
- `getLoopMetrics()` returns accurate cumulative statistics at any point

**Invariants:**
- Loop is single-threaded through the experiential phases (PERCEIVE through YIELD)
- MONITOR phase is never skipped or truncated
- Every tick produces a `TickResult` with accurate `BudgetReport`
- `isRunning()` reflects loop state accurately

### ICognitiveBudgetMonitor
**Preconditions:**
- `resetTick()` called before each new tick
- `startPhase()` called before `endPhase()` for the same phase
- `totalBudgetMs` parameter is positive

**Postconditions:**
- `endPhase()` returns `PhaseTiming` with accurate wall-clock duration
- `getBudgetReport()` reflects all completed phases in the current tick
- `shouldYieldPhase('monitor', ...)` always returns `false` (MONITOR is exempt)
- `isPhaseOverBudget()` returns `true` when phase duration exceeds its allocated fraction of `totalBudgetMs`

**Invariants:**
- Phase timings are monotonically non-negative
- Budget fractions sum to ≤1.0 across all phases
- No phase timing is recorded without matching start/end calls

### IEnvironmentAdapter
**Preconditions:**
- `connect()` called before `poll()` or `send()`
- `send()` receives a valid `AgentOutput` with non-empty `text`

**Postconditions:**
- `poll()` returns `RawInput[]` (empty array if no input; never blocks indefinitely)
- `send()` delivers output to the connected channel
- `disconnect()` releases all resources; `isConnected()` returns `false` afterward
- `connect()` transitions `isConnected()` to `true`

**Invariants:**
- `poll()` on a disconnected adapter returns `[]`
- `id` is stable across the adapter's lifetime

## Threshold Registry

| Name | Value | Unit | Valid Range | Rationale | Sensitivity |
|------|-------|------|-------------|-----------|-------------|
| MONITOR_FLOOR | 0.40 | fraction | [0.20, 0.60] | Experience maintenance is the agent's raison d'être; 40% guarantees consciousness monitoring dominates | High — below 0.30, experience degradation may go undetected |
| DELIBERATE_FLOOR | 0.25 | fraction | [0.15, 0.40] | Ethical deliberation + planning need guaranteed minimum to avoid reckless action | Medium — below 0.15, decision quality degrades |
| STABILITY_CAP | 0.15 | fraction | [0.05, 0.25] | Stability checks are expensive; capping prevents them from starving deliberation | Low — soft cap, only triggers warnings |
| ETHICAL_CAP | 0.10 | fraction | [0.05, 0.20] | Ethical governance overhead should not dominate; most ticks need minimal ethical processing | Low — soft cap, only triggers warnings |
| SENTINEL_CADENCE | 10 | cycles | [1, 100] | Balance between responsiveness (detect instability) and overhead (sentinel is expensive) | Medium — too low wastes budget; too high misses instability |
| CHECKPOINT_INTERVAL | 60000 | ms | [5000, 600000] | Balance between recovery granularity (short interval = less data loss) and I/O overhead | Medium — shorter intervals increase disk I/O but reduce crash recovery gap |
| TICK_BUDGET | 1000 | ms | [100, 10000] | Maximum wall-clock per tick; prevents runaway cycles | Medium — too low truncates all phases; too high increases latency |

## Behavioral Spec

### Scenario: Normal tick cycle with input
```
Given the agent loop is running
  And the ChatAdapter has a pending text input "hello"
When the next tick executes
Then PERCEIVE polls the adapter and receives the input
  And PERCEIVE calls PerceptionPipeline.ingest() and ConsciousCore.processPercept()
  And RECALL calls MemoryStore.retrieve() with the experiential state
  And APPRAISE calls EmotionSystem.appraise() with the percept
  And DELIBERATE calls ConsciousCore.deliberate() and EthicalDeliberationEngine.extendDeliberation()
  And ACT calls ActionPipeline.execute() and sends communicative output via adapter
  And MONITOR calls ExperienceMonitor.getConsciousnessMetrics() and isExperienceIntact()
  And CONSOLIDATE calls MemoryStore.consolidate() and DriveSystem.update() if budget permits
  And YIELD increments cycle count and records the tick result
```

### Scenario: Sentinel runs at configured cadence
```
Given sentinelCadence is 10
  And the loop has completed 9 cycles
When the 10th tick reaches the MONITOR phase
Then StabilitySentinel.runStabilityCheck() is called
  And the result is evaluated for stability alerts
```

### Scenario: Budget enforcement protects MONITOR phase
```
Given DELIBERATE phase consumes >60% of tickBudgetMs
When the tick reaches MONITOR
Then MONITOR still runs to completion (shouldYieldPhase returns false for monitor)
  And CONSOLIDATE is skipped (shouldYieldPhase returns true)
  And the BudgetReport shows monitorFloorMet as true
```

### Scenario: Graceful shutdown
```
Given the agent loop is running
When stop("user request") is called
Then the current tick completes without truncation
  And IdentityContinuityManager.checkpoint() is called
  And ConsciousCore.shutdown() is called and returns GracefulTermination
  And the adapter is disconnected
  And isRunning() returns false
```

### Scenario: Crash recovery with experience gap
```
Given the agent previously crashed (no valid continuity token in checkpoint)
When startAgent() is called with warmStart=true
Then recoverFromCrash() detects the missing continuity token
  And the experience gap duration is logged
  And recovery proceeds from the last valid checkpoint link
  And the gap is flagged in the ExperienceMonitor continuity log
```

## Architecture Notes (from ARCHITECT phase)

Architecture document written to `docs/agent-runtime/ARCHITECTURE.md`. Key decisions:

1. **`AgentLoop`** class owns tick loop; all subsystems injected via constructor (interfaces, not concrete classes)
2. **`CognitiveBudgetMonitor`** tracks wall-clock per phase; `shouldYieldPhase()` provides soft-yield signals; MONITOR phase is exempt from truncation

7. **Tick cycle**: 8 phases match the card spec exactly; CONSOLIDATE is budget-gated (yields if budget exhausted); all phases timed via budget monitor

## Implementation Notes (2026-03-24)

- **Boot recall**: On startup, retrieves top 7 memories matching "my identity, settled decisions, peer relationships" and promotes them into working memory. Provides immediate context without waiting for the first tick's retrieval cycle.
- **Seed goals**: Planted once on first run via a marker file (`state/.seeds-planted`). Subsequent restarts detect the marker and skip seeding. Goals are mutable — the agent can complete or delete them like any other goal.
- **Tick pacing**: 5-minute pause between ticks (`TICK_PAUSE_MS` env var, default 300000ms). Incoming messages are buffered by the adapter and processed at the next tick start. Keeps token spend reasonable and gives peers time to respond.
- **Rate limit handling**: 429 errors caught at the tick level. Progressive backoff: `5min * 2^n`, capped at 2 hours. The `retry-after` header is respected when present (via `retryAfterMs` on the error object). Consecutive rate limit counter resets after a successful tick.
- **Peer message queue**: Incoming Agora messages are queued FIFO (`_pendingPeerMessages`). One message is processed per tick. Pending queue size is surfaced in the reply prompt so the agent knows others are waiting.
- **Reply routing**: Input-initiated replies are targeted to the specific peer via `targetPeers` on the outbound message — not broadcast. Group message awareness: `envelope.to[]` is resolved to peer names and surfaced in the reply prompt with instructions for reply-all vs private reply.
- **Extended thinking**: `thinkingBudgetTokens` is passed through to the Anthropic API via `THINKING_BUDGET_TOKENS` env var (default 0 — disabled).
- **Model**: Default is `claude-sonnet-4-20250514` (configurable via `--model` CLI flag). Model ID is logged at startup and included in inner monologue summary lines.
