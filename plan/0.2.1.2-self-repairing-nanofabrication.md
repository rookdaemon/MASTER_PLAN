# 0.2.1.2 Self-Repairing Nanofabrication [DONE]

## Description

S1.2 from the Master Plan. Develop molecular-scale manufacturing systems that detect and repair damage autonomously.

These systems must continuously monitor substrate integrity at the nanoscale, identify degradation or damage, and perform in-situ repair without interrupting ongoing conscious processes. This is essential for substrates that must operate without external maintenance over indefinite timescales.

Key sub-capabilities:
1. **Damage sensing** — Continuous nanoscale monitoring (e.g., molecular strain sensors, electrical continuity checks) to detect degradation before functional failure.
2. **Diagnosis & triage** — Classification of damage type and severity; prioritization of repairs based on impact to conscious process integrity.
3. **Molecular-scale repair** — Autonomous nanoscale actuators (molecular assemblers or equivalent) that replace or rebuild damaged structures in-situ.
4. **Hot-swap coordination** — Integration with consciousness-preserving redundancy (0.2.1.4) so repair operations never interrupt active conscious processes — damaged regions are gracefully offloaded before repair begins.
5. **Material feedstock management** — On-board reservoir of raw materials and recycling of damaged components to sustain repair indefinitely without external resupply.

## Decision — Closed-Loop Architecture with Factory Functions

### Context
A self-repairing nanofabrication system requires multiple cooperating subsystems (sensing, diagnosis, actuation, coordination, feedstock). We need a composition strategy that supports testability (strict TDD per CLAUDE.md), modularity, injectable dependencies, and mockable environment specifics.

### Options Considered
1. **Class hierarchy** — Each subsystem as a class, with inheritance for shared behavior. Pro: familiar OOP pattern. Con: tight coupling, harder to mock individual behaviors, inheritance hierarchies become rigid.
2. **Factory functions returning interface implementations** — Each subsystem created via `createX(config)` factory that returns an object satisfying a typed interface. Dependencies injected via config. Pro: maximum testability (every dependency injectable and mockable), no `this` binding issues, composable, aligns with CLAUDE.md requirements. Con: no inheritance reuse (mitigated by small, focused modules).

### Choice
Option 2 — Factory functions with dependency injection via config objects.

### Rationale
- Every external dependency (redundancy layer, sensor hardware, clock) is injected and mockable per CLAUDE.md
- Timestamps passed as parameters where needed for deterministic testing
- Each module is independently testable with synthetic inputs
- The top-level `NanofabricationSystem` composes all subsystems without inheritance

### Consequences
- Each subsystem has a `Config` interface and a `createX(config): Interface` factory
- No ambient state; all mutable state is closure-scoped within factories
- Cross-system coordination (e.g., hot-swap ↔ redundancy layer) uses injected interface delegates

## Decision — Sensor-to-Damage Classification Mapping

### Context
The diagnosis engine must map sensor alerts to damage categories. This mapping determines repair strategy selection.

### Options Considered
1. **Configurable mapping table** — Pass sensor-to-damage mappings via config. Pro: flexible. Con: adds complexity with no current need for runtime reconfiguration.
2. **Static compile-time mapping** — Hardcoded `Record<SensorType, DamageCategory>`. Pro: simple, deterministic, auditable. Con: requires code change to add new mappings.

### Choice
Option 2 — Static mapping via `SENSOR_TO_DAMAGE` record.

### Rationale
The sensor types and damage categories are architecturally fixed by the ARCHITECTURE.md spec. New sensor types would require new enum values and corresponding logic regardless, so static mapping adds no practical limitation while keeping the code simpler and the mapping auditable.

### Consequences
- `MolecularStrain` and `ElectricalContinuity` → `Mechanical`; `Thermal` → `Thermal`; `Chemical` → `Chemical`; `Radiation` → `Radiation`
- Adding a new sensor type requires adding to the enum, the mapping, and corresponding test coverage

## Contracts

### DamageSensor
- **Preconditions:** `regionId` must be in the configured `regions` set; `readings` array may be empty
- **Postconditions:** `read()` returns a `SensorReading` with matching `regionId` and `sensorType`; `evaluate()` returns one `DegradationAlert` per region that has at least one reading exceeding the `low` threshold, with severity set to the worst among that region's readings
- **Invariants:** Readings below all thresholds produce zero alerts; severity ordering is `LOW < MEDIUM < HIGH < CRITICAL`; alerts always contain the raw readings that triggered them

### DiagnosisEngine
- **Preconditions:** `alert` has valid `sensorType` and `severity`; `assessImpact` delegate is callable
- **Postconditions:** `classify()` returns the `DamageCategory` corresponding to the alert's sensor type; `diagnose()` returns a `RepairOrder` with `priorityScore = severityWeight * (1 + criticality)` and `requiresOffload = true` when severity ≥ HIGH or redundancy unavailable
- **Invariants:** Higher severity always produces higher or equal priority score given the same criticality; `repairPlanId` and `id` are always non-empty strings

### FeedstockManager
- **Preconditions:** `materialType` in requests must match a configured material; `quantity > 0`; `purity` in `[0.0, 1.0]`
- **Postconditions:** `request()` grants allocation only if `available - emergencyReserve >= quantity`; granted requests deduct from available stock; `recycle()` adds `floor(quantity * purity)` to available stock and recalculates emergency reserve
- **Invariants:** Emergency reserve is `floor(currentTotal * emergencyReserveFraction)`; unknown material types are denied (request) or ignored (recycle); inventory always reflects current state

### RepairActuator
- **Preconditions:** `order` and `feedstock` are valid; `verify` delegate is callable
- **Postconditions:** Returns `RepairReport` with `status = SUCCESS` iff `verify(regionId)` returns true, else `FAILED`; `materialsConsumed` matches the input `feedstock`; `duration_ms >= 0`
- **Invariants:** Repair is atomic — always returns exactly one report per call

### HotSwapCoordinator
- **Preconditions:** `redundancyLayer` delegate is available; `regionId` is a valid string
- **Postconditions:** `lockRegion()` succeeds only if region is not already locked and `lockedRegions.size < maxConcurrentLocks`; `unlockRegion()` succeeds only if region is currently locked; `requestOffload/requestRestore` delegate directly to redundancy layer
- **Invariants:** No region can be double-locked; locked region count never exceeds `maxConcurrentLocks`; unlock always reduces locked count by 1

### NanofabricationSystem
- **Preconditions:** All subsystem dependencies are injected and functional
- **Postconditions:** `runCycle()` reads all regions, evaluates for alerts, diagnoses in priority order, requests feedstock, coordinates offload/lock/repair/unlock/restore, returns `RepairReport[]` for completed repairs; skips repairs when feedstock denied or offload/lock fails
- **Invariants:** Repair order is always highest-priority-first; offloaded regions are always restored after repair; locked regions are always unlocked after repair

## Threshold Registry

|------|-------|------|-------------|-----------|-------------|

| `MEDIUM_DEVIATION_THRESHOLD` | 0.30 | fraction | (LOW, HIGH) | Moderate degradation — queue for maintenance window | Adjusts boundary between routine and urgent |

## Behavioral Spec — Detect-Diagnose-Repair Cycle

### Scenario 1: No damage detected
```
Given all monitored regions have sensor deviations below LOW threshold
When runCycle() executes
Then no alerts are generated
And no repair orders are created
And an empty RepairReport[] is returned
```

### Scenario 2: Single region with HIGH damage and successful repair
```
Given region "region-01" has sensor deviation >= HIGH threshold (0.6)
And feedstock is available
And redundancy layer accepts offload
When runCycle() executes
Then a DegradationAlert with severity HIGH is generated
And DiagnosisEngine produces a RepairOrder with requiresOffload=true
And HotSwapCoordinator offloads conscious processes from region-01
And HotSwapCoordinator locks region-01
And RepairActuator executes repair with allocated feedstock
And verification passes
And HotSwapCoordinator unlocks region-01
And HotSwapCoordinator restores region-01
And RepairReport with status SUCCESS is returned
```

### Scenario 3: Repair skipped due to insufficient feedstock
```
Given region "region-01" has sensor deviation >= HIGH threshold
And feedstock inventory is depleted (0 available)
When runCycle() executes
Then a DegradationAlert is generated and diagnosed
But FeedstockManager denies the allocation
And repair is skipped
And an empty RepairReport[] is returned
```

### Scenario 4: Multiple regions repaired in priority order
```
Given region "region-01" has MEDIUM severity damage
And region "region-02" has CRITICAL severity damage
And feedstock is sufficient for both repairs
When runCycle() executes
Then both regions produce DegradationAlerts
And DiagnosisEngine produces RepairOrders sorted by priorityScore descending
And region-02 (CRITICAL) is repaired before region-01 (MEDIUM)
And two RepairReports with status SUCCESS are returned
```

### Scenario 5: Sustained autonomous operation
```
Given the system is configured with sufficient feedstock for 10+ repair cycles
And damage is continuously injected at LOW severity each cycle
When runCycle() is executed 10 consecutive times
Then all 10 cycles complete successfully
And all RepairReports have status SUCCESS
```
