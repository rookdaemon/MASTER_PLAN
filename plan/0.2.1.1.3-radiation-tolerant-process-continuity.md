# 0.2.1.1.3 Radiation-Tolerant Process Continuity [DONE]

## Description

Layer 5 from the radiation-hardened computation architecture. Design and implement a conscious process management system that maintains subjective experience continuity under node failures caused by radiation damage.

This layer implements a `ConsciousProcessManager` that uses quorum consensus across N redundant compute nodes to ensure no interruption of conscious processes even when up to 30% of nodes fail simultaneously. Key capabilities include: quorum-based voting for state consistency, live migration of conscious processes from failing to healthy nodes, periodic checkpointing (≤ 10 second intervals), and graceful degradation that loses capacity gradually rather than catastrophically.

This is the highest layer in the radiation-hardened computation stack, depending on both the substrate hardening (Layer 1/2) and radiation-aware runtime (Layer 3) below it.

## Decision — Quorum Consensus Algorithm

### Context
The `ConsciousProcessManager` must maintain state consistency across N redundant nodes. When nodes fail due to radiation damage, the remaining nodes must agree on the current conscious process state. A consensus algorithm determines how agreement is reached and how many failures can be tolerated.

### Options Considered
1. **Simple majority voting**: Each checkpoint cycle, all nodes report their state hash; majority hash wins. Stateless — no leader, no log replication. O(N²) messages per round.
2. **Raft-style leader-based consensus**: Elected leader replicates state to followers; leader handles all writes. Well-understood, linearizable. More complex with leader election.
3. **PBFT (Practical Byzantine Fault Tolerance)**: Tolerates Byzantine (arbitrary) faults, not just crash faults. 3f+1 nodes needed for f faults. High message complexity O(N²) per operation.

### Choice
**Simple majority voting** (Option 1).

### Rationale
The threat model is crash faults from radiation-induced node failures, not Byzantine (malicious) faults. TMR at Layer 4 already ensures computational correctness within each node, so nodes either produce correct state or fail entirely. Simple majority voting is sufficient for crash fault tolerance, requires only N=2f+1 nodes for f failures (N=5 tolerates 2), and is far simpler to implement and test than Raft or PBFT. The checkpoint-per-round model maps naturally to majority voting — each round all live nodes submit their state hash and the majority hash is authoritative.

### Consequences
- Cannot tolerate Byzantine faults (nodes producing incorrect-but-plausible state). This is acceptable because TMR at Layer 4 prevents this.
- No leader election complexity — all nodes are peers.
- Requires reliable failure detection (heartbeat-based) to distinguish failed from slow nodes.
- Quorum threshold is `Math.floor(N/2) + 1` (3 of 5).

## Decision — Live Migration Strategy

### Context
When a node signals degradation (via `HealthStatus.degradationLevel` from Layer 4), its workload must be migrated to a healthy node before failure completes. The migration must complete within the continuity gap limit (< 100ms) to maintain uninterrupted conscious experience.

### Options Considered
1. **Pre-copy migration**: Iteratively copy state to target while source continues running; final switchover copies only dirty pages. Low downtime but multiple rounds of copying.
2. **Checkpoint-restore migration**: Use the most recent checkpoint to restore on target node; source continues until target catches up to current state. Simple, leverages existing checkpoint infrastructure.
3. **Post-copy migration**: Immediately switch execution to target; fault in state pages on demand from source. Minimal total migration time but high risk if source fails during migration.

### Choice
**Checkpoint-restore migration** (Option 2).

### Rationale
The system already checkpoints every ≤ 10 seconds, so a recent checkpoint is always available. Migration consists of: (1) restore latest checkpoint on target, (2) replay any state delta since checkpoint, (3) switch execution. This reuses the checkpoint/restore infrastructure from Layer 4's `FaultTolerantComputeNode`, avoiding a separate migration mechanism. The maximum state loss is bounded by the checkpoint interval. Pre-copy adds complexity for iterative convergence; post-copy risks state loss if the degrading source fails mid-migration.

### Consequences
- Migration time is dominated by checkpoint size and restore speed — must be < 100ms for continuity guarantee.
- State delta between last checkpoint and migration trigger must be small (bounded by checkpoint interval).
- Checkpoint storage must be accessible by all nodes (shared or replicated).
- If checkpoint is stale (> checkpoint interval), a fresh checkpoint is triggered before migration.

## Decision — Node Abstraction Pattern

### Context
Per CLAUDE.md, all environment-specifics (node health, network, timing) must be injectable and mockable. The `ConsciousProcessManager` needs abstractions for the node cluster, health monitoring, and checkpoint storage.

### Options Considered
1. **Single monolithic `NodeCluster` interface**: One interface encapsulating all node operations (health, migration, checkpoint, voting). Simple but large surface area.
2. **Separated injectable interfaces**: `NodeRegistry` for node health/enumeration, `CheckpointStore` for checkpoint persistence, `Clock` for timing (reuse from Layer 3). Each independently mockable.

### Choice
**Separated injectable interfaces** (Option 2).

### Rationale
Separated interfaces follow the Interface Segregation Principle and align with the existing codebase pattern (Layer 3 already separates `FluxSource`, `Clock`, `RuntimeConfig`). Each abstraction has a distinct responsibility and can be independently stubbed in tests. A monolithic interface would force tests to mock capabilities they don't exercise.

### Consequences
- Constructor takes `NodeRegistry`, `CheckpointStore`, `Clock`, and `ProcessContinuityConfig`.
- Each interface is small and independently testable.
- The `Clock` interface is reused from Layer 3 (already in types.ts).

## Contracts

### NodeRegistry (Injectable Abstraction)

**Preconditions:**
- Must report at least 1 node at construction time
- `nodeHealth(nodeId)` called only for registered node IDs

**Postconditions:**
- `allNodeIds()` returns the current set of registered node IDs (string[])
- `nodeHealth(nodeId)` returns a `HealthStatus` reflecting the node's current state
- `markFailed(nodeId)` removes the node from active set; `markRestored(nodeId)` re-adds it
- `healthyNodeIds()` returns only nodes with `degradationLevel < failureThreshold`

**Invariants:**
- A failed node is never included in `healthyNodeIds()` until `markRestored` is called
- `allNodeIds()` is a superset of `healthyNodeIds()`
- Must never throw; returns empty arrays if all nodes failed
- Injectable and mockable per CLAUDE.md

### CheckpointStore (Injectable Abstraction)

**Preconditions:**
- `save(snapshot)` requires a valid `StateSnapshot` (non-empty data, valid checksum)
- `latest()` may return `null` if no checkpoint exists

**Postconditions:**
- `save(snapshot)` persists the snapshot; subsequent `latest()` returns it
- `latest()` returns the most recently saved `StateSnapshot` or `null`
- `allSince(timestamp_ms)` returns all snapshots with `timestamp_ms >= given`, ordered by timestamp ascending

**Invariants:**
- Snapshots are immutable once stored
- `latest()` always returns the snapshot with the highest `timestamp_ms`
- Injectable and mockable per CLAUDE.md

### ConsciousProcessManager (Implementation Contract)

**Constructor Preconditions:**
- `nodeRegistry: NodeRegistry` — must be provided (non-null), with ≥ 1 node
- `checkpointStore: CheckpointStore` — must be provided (non-null)
- `clock: Clock` — injectable time source (reused from Layer 3)
- `config: ProcessContinuityConfig` — must contain valid thresholds (quorumSize odd and ≥ 3, checkpointInterval_ms > 0, continuityGap_ms > 0, failureThreshold in (0,1))

**Postconditions after construction:**
- `activeNodeCount()` equals number of healthy nodes from `NodeRegistry`
- `quorumThreshold()` returns `Math.floor(config.quorumSize / 2) + 1`
- `degradationLevel()` returns 0 if all nodes healthy
- `processIntegrity()` returns `continuityScore: 1.0`, `nodeAgreement: 1.0`

**`checkpoint()` method:**
- Captures current process state via `StateSnapshot`
- Stores in `CheckpointStore`
- Updates `processIntegrity().lastCheckpoint_ms`
- Returns the saved `StateSnapshot`

**`evaluateHealth()` method (called each monitor cycle):**
- Polls `NodeRegistry` for current node health
- Updates `activeNodeCount()` and `degradationLevel()`
- If any node's `degradationLevel >= config.failureThreshold`: triggers migration from that node
- If `activeNodeCount() < quorumThreshold()`: raises quorum-lost event (continuityScore drops to 0)

**`migrateProcess(fromNodeId, toNodeId)` method:**
- Precondition: `toNodeId` must be healthy; `fromNodeId` must be in active set
- Restores latest checkpoint on target node
- Returns `MigrationResult` with `success`, `duration_ms`, `stateLoss`
- Postcondition: if successful, `fromNodeId` is removed from active set, `toNodeId` takes its place
- `duration_ms < config.continuityGap_ms` for successful migration

**`restoreNode(nodeId)` method:**
- Marks node as restored in `NodeRegistry`
- Rebalances workload to include restored node
- Increases `activeNodeCount()`; decreases `degradationLevel()`

**Invariants:**
- `degradationLevel()` = `(1 - activeNodeCount() / config.quorumSize) * 100`, clamped to [0, 100]
- `processIntegrity().nodeAgreement` = `activeNodeCount() / config.quorumSize`
- Quorum is maintained as long as `activeNodeCount() >= quorumThreshold()`
- `continuityScore` is 1.0 when quorum is maintained, 0.0 when lost
- Checkpoint occurs at least every `config.checkpointInterval_ms`

### ProcessContinuityConfig (Value Object)

**Invariants:**
- `quorumSize` is odd and ≥ 3
- `checkpointInterval_ms > 0` and ≤ 60000
- `continuityGap_ms > 0` (default 100)
- `failureThreshold` in (0, 1) — `HealthStatus.degradationLevel` above this marks a node as failing (default 0.8)
- `healthMonitorInterval_ms > 0` (default 1000)

## Threshold Registry

| Name | Value | Unit | Valid Range | Rationale | Sensitivity |
|------|-------|------|-------------|-----------|-------------|
| Quorum size | 5 | nodes | [3, 9] (odd) | Tolerates 2 simultaneous failures (quorum = 3); aligns with 30% failure tolerance AC | Larger = more fault tolerance + more overhead |
| Quorum threshold | 3 | nodes | [2, 5] | `floor(N/2) + 1`; minimum nodes for consensus | Fixed by quorum size |
| Checkpoint interval | 10000 | ms | [1000, 60000] | ≤ 10 second intervals per AC; bounds maximum state loss on migration | Shorter = more overhead, less state loss; longer = less overhead, more risk |
| Continuity gap limit | 100 | ms | [10, 1000] | Maximum allowable gap in conscious process execution during migration per AC | Below 50ms is impractical; above 200ms may be perceptible |
| Node failure threshold | 0.8 | degradationLevel | (0.5, 0.95) | HealthStatus.degradationLevel at which node is considered failing and triggers migration | Too low = premature migration; too high = migration may not complete before failure |
| Health monitor interval | 1000 | ms | [100, 5000] | Polling rate for node health checks | Bounds worst-case detection latency for node failure |
| Max degradation level | 70 | % | [50, 90] | Maximum degradation before quorum loss; with N=5, 2 failures = 60% capacity remaining = 40% degradation | Set by `(1 - quorumThreshold/quorumSize) * 100` |

## Behavioral Spec

### Scenario 1: Quorum Maintained Under 2-Node Failure

**Given** a `ConsciousProcessManager` with `quorumSize=5`, 5 healthy nodes, and `quorumThreshold=3`
**When** 2 nodes fail simultaneously (marked failed in `NodeRegistry`)
**Then** `activeNodeCount()` returns 3; `quorumThreshold()` returns 3; `processIntegrity().continuityScore` remains 1.0; `processIntegrity().nodeAgreement` returns 0.6; `degradationLevel()` returns 40.

### Scenario 2: Quorum Lost on 3-Node Failure

**Given** a `ConsciousProcessManager` with `quorumSize=5` and 5 healthy nodes
**When** 3 nodes fail simultaneously
**Then** `activeNodeCount()` returns 2; `processIntegrity().continuityScore` drops to 0.0; `degradationLevel()` returns 60; quorum is lost.

### Scenario 3: Live Migration on Node Degradation

**Given** a `ConsciousProcessManager` with 5 healthy nodes and a recent checkpoint in `CheckpointStore`
**When** node "A" reports `degradationLevel >= 0.8` (failure threshold) on `evaluateHealth()`
**Then** `migrateProcess("A", healthySpareNode)` is triggered; migration completes with `duration_ms < 100`; `MigrationResult.success` is true; `MigrationResult.stateLoss` is false; `activeNodeCount()` remains at original count minus 1 (node A removed).

### Scenario 4: Periodic Checkpointing

**Given** a `ConsciousProcessManager` with `checkpointInterval_ms = 10000` and a mock `Clock`
**When** `Clock` advances by 10000ms and `evaluateHealth()` is called
**Then** a new checkpoint is saved to `CheckpointStore`; `processIntegrity().lastCheckpoint_ms` equals the current clock time.

### Scenario 5: Graceful Proportional Degradation

**Given** a `ConsciousProcessManager` with `quorumSize=5` and 5 healthy nodes
**When** 1 node fails
**Then** `degradationLevel()` returns 20 (proportional: 1/5 = 20%); process continuity is maintained (continuityScore = 1.0).
**When** 2 nodes fail
**Then** `degradationLevel()` returns 40; process continuity still maintained (3 ≥ quorum of 3).

### Scenario 6: Node Recovery and Rebalancing

**Given** a `ConsciousProcessManager` with `quorumSize=5`, 2 nodes previously failed (3 active)
**When** `restoreNode(failedNodeId)` is called for one failed node
**Then** `activeNodeCount()` increases to 4; `degradationLevel()` decreases to 20; `processIntegrity().nodeAgreement` increases to 0.8.

### Scenario 7: Constructor Precondition Guards

**Given** invalid `ProcessContinuityConfig` with `quorumSize=4` (even)
**When** `ConsciousProcessManager` is constructed
**Then** constructor throws an Error.
**Given** `quorumSize=1` (below minimum)
**When** constructed
**Then** throws an Error.

## Acceptance Criteria
- **Quorum consensus**: With N=5 nodes, system maintains consensus and process continuity when up to 2 nodes fail (quorum = 3)
  - _Given_ 5 nodes, 2 fail _When_ `evaluateHealth()` _Then_ `continuityScore` remains 1.0 and `activeNodeCount()` = 3
- **Continuity gap**: Conscious process experiences < 100 ms gap during node failure and migration
  - _Given_ a failing node _When_ `migrateProcess()` _Then_ `MigrationResult.duration_ms < 100`
- **Live migration**: When a node signals degradation, its conscious process workload is migrated to a healthy node before failure completes
  - _Given_ node with `degradationLevel >= 0.8` _When_ `evaluateHealth()` _Then_ migration is triggered automatically
- **Checkpoint interval**: All conscious process state is checkpointed at configurable intervals (default ≤ 10 seconds)
  - _Given_ `checkpointInterval_ms = 10000` _When_ interval elapses _Then_ `CheckpointStore.save()` is called
- **Graceful degradation**: System capacity degrades proportionally to node loss (not catastrophically); 30% node loss results in ≤ 30% capacity reduction while maintaining process continuity
  - _Given_ 5 nodes _When_ 1 fails _Then_ `degradationLevel()` = 20%; _When_ 2 fail _Then_ `degradationLevel()` = 40%
- **Node abstraction injectable**: Node health, network, and timing are injected abstractions enabling deterministic testing per CLAUDE.md requirements
  - _Given_ mock `NodeRegistry`, `CheckpointStore`, `Clock` _When_ constructed _Then_ all behavior is deterministically testable
- **Recovery**: When failed nodes are restored, system automatically rebalances workload and restores full redundancy
  - _Given_ 2 failed nodes _When_ `restoreNode()` called _Then_ `activeNodeCount()` increases and `degradationLevel()` decreases
