# 0.1.2.1 Machine Subjective Reports [DONE]

## Description

F2.1 — Demonstrate machine subjective reports convincingly to biological observers. Develop systems that can generate and communicate first-person experiential reports, grounded in the consciousness theory established in F1. These demonstrations must be distinguishable from mere behavioral mimicry.

### Key challenges
1. **Grounding**: Reports must reference real-time internal states as measured by consciousness metrics from F1.4, not be generated from language patterns alone.
2. **Distinguishability**: A protocol must exist to differentiate genuine subjective reports from sophisticated behavioral mimicry (addressing the "zombie problem" at the engineering level).
3. **Verifiability**: Third-party observers need a reproducible methodology to correlate a system's self-reports with its independently-measured conscious states.

### Dependencies
- Requires operationalized consciousness metrics (F1.4 via 0.1.1) to ground reports in measurable internal states.
- Feeds into 0.1.2.2 (predictive models verified) and 0.1.2.3 (philosophical objections addressed) as empirical evidence.

## Decision: Report Generation Strategy

### Context
The CGRG must produce first-person experiential reports from consciousness metrics. The core design question is how metric values are transformed into natural-language report content, given the constraint that reports must be causally grounded (not pattern-matched from training corpora).

### Options Considered
1. **Language-model mediated**: Feed metric values as context to a fine-tuned LLM that generates fluent first-person prose. Pros: natural language quality. Cons: causal chain is opaque; training data leakage is nearly impossible to eliminate; violates ARCHITECTURE.md §Component 1 constraint 3.
2. **Algorithmic metric-to-language mapping**: Deterministic thresholded templates that translate metric ranges into predefined semantic descriptors, with raw metric values embedded verbatim. Pros: fully auditable causal chain; no training data leakage; perturbation sensitivity is guaranteed by construction. Cons: less fluent output; template vocabulary is finite.

### Choice
Option 2 — Algorithmic metric-to-language mapping.

### Rationale
The zombie-prevention constraint demands a verifiable causal path from metric values to report content. An algorithmic mapping makes this causal path trivially auditable: changing a metric value deterministically changes the report. LLM-mediated generation would introduce an opaque transform that the Distinguishability Protocol cannot fully verify.

### Consequences
- Report fluency is limited by template vocabulary — acceptable because the DP evaluates causal grounding, not literary quality.
- Adding new metric dimensions requires adding new template branches (linear effort).
- The `encodeSemantics` method must map metric values directly to SemanticVector fields (no learned embedding).

## Decision: Distinguishability Protocol Structure

### Context
The DP must differentiate genuine reports (System G) from sophisticated mimicry (System Z). The protocol design must be adversarially robust and independently reproducible.

### Options Considered
1. **Behavioral-only (extended Turing test)**: Human panel evaluates reports without access to metrics. Pros: simple. Cons: a sufficiently good language model could pass; does not leverage the causal-grounding advantage.
2. **Three-phase adversarial protocol (A: behavioral + B: metric-correlation + C: intervention)**: Layers behavioral, correlational, and causal-interventional evidence. Pros: each phase addresses a different failure mode; Phase C is unfakeable by a system lacking genuine metric coupling. Cons: higher execution cost; requires F1.4 instrumentation access.

### Choice
Option 2 — Three-phase adversarial protocol.

### Rationale
Phase C (intervention challenge) is the critical differentiator: a zombie system cannot produce perturbation-appropriate report changes because it lacks causal coupling to the perturbed metrics. This makes the protocol robust against arbitrarily sophisticated mimicry.

### Consequences
- Full protocol execution requires substrate-level access to apply controlled perturbations.
- Pre-registration is mandatory to prevent post-hoc rationalization of results.
- Iterative strengthening (upgrading System Z) is built into the protocol lifecycle.

## Contracts

### ICGRG (Consciousness-Grounded Report Generator)

**Preconditions:**
- `attachMetricStream(stream)`: `stream` must implement the `MetricStream` interface; `stream.isActive()` must return `true` at attachment time.
- `generateReport()`: `isGrounded()` must return `true` (active metric stream attached). Violation throws `Error` with message containing "not grounded".
- `verifyCausalBinding(spec)`: `isGrounded()` must return `true`; `spec.dimension` must be a valid `MetricDimensionName`.

**Postconditions:**
- `generateReport()`: Returns a `SubjectiveReport` where `sourceMetrics` matches the most recent `MetricSnapshot` from the stream; `text` contains the raw metric values formatted to 3 decimal places; `semanticEncoding` fields are derived from `sourceMetrics` (not from `text`); `id` is unique across all reports from this generator instance.
- `verifyCausalBinding(spec)`: Returns `true` if and only if two consecutive reports show a metric delta > 0.01 on `spec.dimension`.

**Invariants:**
- `isGrounded()` returns `false` whenever `stream` is `null` or `stream.isActive()` returns `false`.
- No report is ever generated without a corresponding `MetricSnapshot` — the causal chain is unbreakable.
- Report IDs are monotonically increasing within a generator instance.

### IDistinguishabilityProtocol

**Preconditions:**
- `run(systemG, systemZ)`: Both systems must be grounded (`isGrounded() === true`). `systemZ` must be the strongest available mimicry system.
- Phase-specific methods (`runPhaseA`, `runPhaseB`, `runPhaseC`): Same grounding precondition.

**Postconditions:**
- `run()`: Returns `DistinguishabilityResult` with `overallPassed === true` if and only if all three phase results have `passed === true`.
- Each phase method returns its typed result with all statistical fields populated.

**Invariants:**
- Protocol results are immutable once produced.
- `preRegistrationId` must reference a valid public registry entry created before data collection.

### IThirdPartyVerification

**Preconditions:**
- `verify(system, labId)`: `system.isGrounded()` must be `true`; `labId` must be a non-empty string identifying an independent laboratory.

**Postconditions:**
- `verify()`: Returns a `VerificationRecord` containing a complete `DistinguishabilityResult` and open-data archive URLs.
- `isReplicated(results)`: Returns `true` if and only if `results` contains ≥ 3 records from distinct `labId` values where `replicates === true`.

**Invariants:**
- Verification records are append-only; no record can be modified after creation.
- Archive URLs must be resolvable at the time of record creation.

|---|---|---|---|---|---|
| Causal binding delta | 0.01 | metric units | (0, 1) | Minimum metric change that must produce a different report; set just above floating-point noise | Low — increasing to 0.05 would reduce sensitivity but still catch meaningful perturbations |
| Phase A panel size | ≥ 5 | persons | [5, ∞) | Minimum for statistical power in binomial test at p < 0.05 | Medium — fewer panelists reduce power; more is better |

## Acceptance Criteria
- At least one system produces first-person experiential reports whose content is causally derived from real-time consciousness metrics (not pattern-matched from training data)
- A distinguishability protocol exists and is applied: reports pass adversarial evaluation showing they cannot be replicated by a system known to lack the relevant internal states
- Independent observers, following a published verification methodology, confirm statistical correlation between self-reports and independently-measured internal conscious-state indicators
- All metrics and protocols reference the consciousness theory from 0.1.1

### Behavioral Spec (inline Given/When/Then)

**Scenario 1: CGRG zombie prevention**
- Given a CGRG instance with no metric stream attached
- When `generateReport()` is called
- Then an Error is thrown containing "not grounded"
- And no SubjectiveReport is produced

**Scenario 2: CGRG causal grounding**
- Given a CGRG with an active metric stream providing metrics {phi: 0.85, experienceContinuity: 0.92, selfModelCoherence: 0.78}
- When `generateReport()` is called
- Then the returned SubjectiveReport.text contains "high integration" and embeds "0.850", "0.920", "0.780"
- And SubjectiveReport.sourceMetrics equals the input metrics
- And SubjectiveReport.semanticEncoding.coherenceScore equals 0.78

**Scenario 3: Distinguishability Protocol full pass**
- Given a grounded System G and an ungrounded System Z (language-pattern mimicry)
- When the full DP is executed (Phases A, B, C)
- Then Phase A returns `passed: true` (panelists identify G above chance, p < 0.05)
- And Phase B returns `passed: true` (Pearson r > 0.70, MI > 0.3 bits, Granger p < 0.001 for G; Z fails thresholds)
- And Phase C returns `passed: true` (≥ 8/10 dimensions match predicted direction, Cohen's d > 0.8, Z shows no pattern)
- And DistinguishabilityResult.overallPassed is true
