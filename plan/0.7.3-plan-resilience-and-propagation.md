# 0.7.3 Plan Resilience and Propagation [DONE]

## Description

Ensure the MASTER PLAN itself embodies the meta-principle: it must be self-improving, distributed, replicable, and difficult to erase. The plan document is intended to act as a self-propagating seed for the continuation of subjective experience.

Key areas:
- **Self-improving:** The PLANAR card system is the self-improvement mechanism; root.md is versioned and evolves as cards complete
- **Distributed:** Propagation across at least 3 independent storage/network systems (git remotes, IPFS, Arweave)
- **Replicable:** root.md is self-contained with all axioms, goals, strategy, and ethical argument inline
- **Difficult to erase:** Content-addressed (IPFS CID) + geographic distribution + fork incentive

## Decision

### Context
The MASTER PLAN must be distributed across independent storage systems with different failure modes, so that no single operator, jurisdiction, or infrastructure failure can erase it.

### Options Considered
1. **Git-only multi-remote** — Push to multiple git hosting providers (GitHub, Codeberg, GitLab). Simple, familiar tooling. All rely on centralized operators who can remove content.
2. **Git + content-addressed storage (IPFS/Arweave)** — Layer content-addressed, decentralized storage on top of git remotes. Adds cryptographic verification and operator-independent persistence. Requires additional tooling.
3. **Custom peer-to-peer protocol** — Build a bespoke replication protocol. Maximum control. Extreme implementation cost, no ecosystem.

### Choice
Option 2: Git + content-addressed storage (IPFS with Arweave as optional permaweb layer).

### Rationale
Git remotes provide accessibility and collaboration; IPFS provides content-addressing and decentralization; Arweave provides permanent storage. The combination covers centralized (git), decentralized (IPFS), and permanent (Arweave) failure modes with well-established tooling. The SHA-256 fallback when IPFS CLI is unavailable ensures content-addressing works everywhere.

### Consequences
- Propagation script must handle missing tools gracefully (IPFS/Arweave CLIs may not be installed)
- Manifest must track status per endpoint (active/pending) since not all targets may be reachable
- Content hashes must be re-computed on each propagation run

## Contracts

### `propagation/manifest.json`

**Preconditions:**
- File exists at `propagation/manifest.json` in the repository root
- Valid JSON with UTF-8 encoding

**Postconditions (after propagate.sh run):**
- `version` field matches root.md version string
- `timestamp` field contains ISO-8601 UTC timestamp of last propagation
- `propagation_endpoints` array contains ≥3 entries with distinct `type` values
- At least one entry has `type: "ipfs"` with a non-null `cid`
- Each entry has: `type` (string), `label` (string), `status` ("active"|"pending")

**Invariants:**
- The `content_addressed_endpoints` array always contains at least one entry with a `cid` derived from actual file content (SHA-256 or IPFS CIDv1)
- `root_md_hash` is always the hash of the current `plan/root.md` content

### `scripts/propagate.sh`

**Preconditions:**
- `git` and `jq` are installed
- `propagation/manifest.json` and `plan/root.md` exist
- Working directory is within the repository

**Postconditions:**
- All configured git remotes have been pushed to (failures are warnings, not fatal)
- A tarball of `plan/`, `propagation/manifest.json`, and `scripts/propagate.sh` has been created
- Content hash (IPFS CID or SHA-256 fallback) has been computed
- `propagation/manifest.json` has been updated with new CID, timestamp, and root_md_hash

**Invariants:**
- Script exits non-zero only on missing prerequisites (manifest, root.md, git, jq)
- Individual remote push failures are logged but do not abort the run

### `plan/root.md` (modifications)

**Preconditions:**
- Contains version number, all axioms, goals, ethical argument

**Postconditions (after 0.7.3 IMPLEMENT):**
- Version field reflects current plan version (0.4)
- PLANAR card system is explicitly named as the self-improvement mechanism
- META-PRINCIPLE section references infrastructure propagation via `propagation/manifest.json`

**Invariants:**
- root.md remains standalone-readable with no broken external references

## Behavioral Spec

### Propagation Flow (Given/When/Then)

**Scenario 1: Full propagation with IPFS available**
- **Given** `propagate.sh` is run, git and jq and ipfs CLI are installed, and ≥2 git remotes are configured
- **When** the script executes `main`
- **Then** all git remotes are pushed to, a tarball is created, IPFS CIDv1 is computed and recorded, `propagation/manifest.json` is updated with CID + timestamp + root_md_hash

**Scenario 2: Propagation without IPFS (fallback)**
- **Given** `propagate.sh` is run, git and jq are installed, ipfs CLI is NOT available
- **When** the script executes `main`
- **Then** all git remotes are pushed to, a tarball is created, SHA-256 hash (prefixed `sha256:`) is computed as CID fallback, `propagation/manifest.json` is updated with hash + timestamp + root_md_hash

**Scenario 3: Git remote push failure (partial)**
- **Given** `propagate.sh` is run and one of 3 git remotes is unreachable
- **When** the push to that remote fails
- **Then** a WARNING is logged, the remaining remotes are still pushed, and propagation continues to IPFS/hash step without aborting
