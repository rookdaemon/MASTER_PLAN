# 0.0.1 Agent Runtime Architecture [DONE]

## Description

Documents the current agent's architecture as it actually runs: drive system, memory architecture, personality/trait model, emotional appraisal, world model, planning, and the runtime loop. This is a documentation card — it describes what exists, grounded in the actual implementation, not what should be built.

### Drive System
A set of intrinsic motivations (curiosity, social engagement, plan advancement, self-preservation) that generate goal candidates each cycle. Drives have intensities that decay and replenish based on recent activity, producing genuine variation in what the agent pursues.

### Memory Architecture
Three-tier (working memory → episodic → semantic). Working memory holds the current cycle's active context. Episodic memory stores timestamped experiences with emotional valence. Semantic memory accumulates consolidated knowledge derived from repeated episodic patterns. Retrieval is composite: recency, relevance, and emotional salience all contribute.

### Personality and Trait Model
A persistent configuration of temperament dimensions (openness, conscientiousness, agreeableness, etc.) and behavioral biases that shape language generation, goal weighting, and emotional appraisal. Two instances configured differently are demonstrably distinct in behavior.

### Emotional Appraisal System
Dynamic valence and arousal updated each cycle based on goal progress, drive satisfaction, and social feedback. Mood state influences memory retrieval (mood-congruent recall) and language generation.

### World Model and Belief Store
An internal representation of external reality, updated through perception and inference. Beliefs are held with explicit uncertainty and can be revised.

### Planning and Temporal Reasoning
Multi-step plan construction against the active goal set, with horizon-limited lookahead and plan revision when assumptions are violated.

### Agent Runtime Loop
PERCEIVE → RECALL → APPRAISE → DELIBERATE → ACT → MONITOR → CONSOLIDATE → YIELD. All processing flows through the Conscious Core's experiential pipeline; no zombie bypass.

## Decision: Documentation Scope and Organization

### Context
This card documents the running agent architecture. The codebase already has `docs/agent-runtime/ARCHITECTURE.md` (card 0.3.1.5.9 focused on design intent). This card's document must describe what actually exists in source, grounded in implementation, not aspirational design.

### Options Considered
1. **Update existing `docs/agent-runtime/ARCHITECTURE.md`** — risks conflating 0.3.1.5.9's design-intent document with 0.0.1's as-built documentation.
2. **Create `docs/agent-runtime/ARCHITECTURE-as-built.md`** — separate document focused on what the code actually does, with cross-references to interface files.

### Choice
Option 2: Create a dedicated as-built architecture document.

### Rationale
The existing ARCHITECTURE.md is scoped to card 0.3.1.5.9 and reflects design intent for that card's implementation. This card (0.0.1) serves a different purpose: grounding all subsystem documentation in actual source. Mixing the two would create conflicting ownership and confuse which parts are aspirational vs implemented.

### Consequences
- Two architecture documents exist for agent-runtime; the as-built one is authoritative for "what runs today."
- Future cards should reference the as-built document when they need to understand current behavior.

## Contracts

### AgentLoop ↔ Subsystems (Constructor Injection)
**Preconditions**: All 12 required dependencies must be non-null. Optional: `llm?: ILlmClient`.
**Postconditions**: `start()` enters the tick loop; each tick executes all 8 phases in order; `stop()` completes the current tick then exits gracefully.
**Invariants**:
- No phase is skipped (MONITOR is explicitly never truncatable even under budget pressure).
- No action reaches the environment without passing through IConsciousCore.deliberate → IActionPipeline.execute (no zombie bypass).
- Drive system fires at most one goal candidate per tick to the coherence engine.

### IMemoryStore (agent-runtime facade)
**Preconditions**: `retrieve()` receives a valid ExperientialState.
**Postconditions**: Returns memory items ranked by composite score (similarity × recency × salience).
**Invariants**: `consolidate()` is skippable only when cognitive budget is exhausted.

### IEmotionSystem (agent-runtime facade)
**Preconditions**: `appraise()` receives current percept (or null), active goals, and values.
**Postconditions**: Returns AppraisalResult with dimensional valence/arousal shifts (never discrete emotion labels).
**Invariants**: EWMA mood update formula `newMood = (1 - α) × current + α × signal` is applied every tick. Null appraisal decays toward baseline (valence=0, arousal=0.5).

### IDriveSystem
**Preconditions**: `tick()` receives current ExperientialState and DriveContext with personality params.
**Postconditions**: Returns DriveTickResult containing goal candidates (0 or more), experiential delta, updated states, diagnostics. Only drives above ACTIVATION_THRESHOLD (0.15) with expired cooldowns fire candidates.
**Invariants**: Mastery drive never fires goal candidates (reward signal only). Rejection by coherence engine triggers 30s extended cooldown. Acceptance reduces strength by 0.3.

### IPersonalityModel
**Preconditions**: `applyToDeliberation()` receives a Decision and ExperientialState.
**Postconditions**: Returns modified Decision with adjusted confidence and potentially reordered alternatives.
**Invariants**: Must not change `action.type` or reverse the verdict. Trait values always in [0,1]. Drift > 0.3 classified as corruption.

### IBeliefStore (revision policy)
**Preconditions**: `revise()` receives belief ID, new confidence, and trigger description.
**Postconditions**: Returns BeliefRevision with one of three resolutions:
- `rejected`: newConfidence < 0.4 AND existing >= 0.8 (high-confidence beliefs resist weak evidence)
- `updated`: |delta| > 0.4 (strong contradiction forces update)
- `flagged-uncertain`: otherwise (confidence averaged)
**Invariants**: Contradictions are surfaced for deliberation, never auto-pruned.

### IPlanner
**Preconditions**: `generatePlan()` receives goal, state, world context, and time budget in ms.
**Postconditions**: Returns Plan with status "pending" or "suspended" (if budget exceeded). Currently placeholder 3-step plans.
**Invariants**: `shouldAbandon()` returns true when escalationCount >= maxEscalations (default 3). Planner is optional in DeliberationContext (null → legacy priority-sort).

### IEmotionalRegulation (three-level safety)
**Preconditions**: `checkAndRegulate()` receives current MoodState.
**Postconditions**: Returns RegulationOutcome or null.
**Invariants**: Three escalation levels enforced:
- Level 1 (valence < -0.1): log + monitor
- Level 2 (valence < -0.3 for ≥5 cycles, OR < -0.7): corrective intervention
- Level 3 (valence < -0.85 AND correction failing ≥3 cycles): halt/suspend

## Behavioral Spec: 8-Phase Tick Cycle

### Scenario: Normal tick execution
```
Given the agent loop is running and all subsystems are injected
When a tick begins
Then the following phases execute in strict order:

  Phase 1 — PERCEIVE:
    - Poll adapter for RawInput[]
    - Convert to SensorData → Percept → ExperientialState via perception pipeline
    - Process at most one queued peer message per tick

  Phase 2 — RECALL:
    - memory.retrieve(experientialState) returns ranked memories
    - Composite scoring: similarity × recencyWeight × salienceBoost

  Phase 3a — APPRAISE:
    - emotionSystem.appraise(percept, goals, values) → AppraisalResult
    - Goal congruence weighted by priority; value threats produce hard floor at -0.8
    - EWMA mood update: newMood = (1 - α) × current + α × signal

  Phase 3b — DRIVE TICK:
    - driveSystem.tick(experientialState, context) → DriveTickResult
    - 8 drives evaluated: curiosity, social, homeostatic-arousal, homeostatic-load, homeostatic-novelty, boredom, mastery, existential
    - Strongest single candidate submitted to coherence engine (deduplicated against active drive goals)
    - Experiential delta (valence/arousal shifts) applied to state
    - Satiated drives cleaned up; valence recovery: min(satiatedCount × 0.15, 0.5)

  Phase 4 — DELIBERATE:
    - core.deliberate(experientialState, goals, deliberationContext)
    - If planner non-null: multi-step planning with budget
    - If planner null: legacy priority-sort
    - personalityModel.applyToDeliberation(decision, state) adjusts confidence
    - ethicalEngine.extendDeliberation() applied to all decisions

  Phase 5 — ACT:
    - actionPipeline.execute(decision) → ActionResult
    - Communicative actions with LLM: enriched inference with per-peer conversation history
    - Drive-initiated actions: executeDriveToolLoop (max 16 iterations)

  Phase 6 — MONITOR:
    - NEVER skipped, NEVER truncated (even under budget pressure)
    - monitor.isExperienceIntact() checked
    - Stability sentinel checked every N cycles

  Phase 7 — CONSOLIDATE:
    - memory.consolidate() runs unless cognitive budget exhausted
    - Episodic decay removes low-salience entries below threshold

  Phase 8 — YIELD:
    - Save experiential state checkpoint
    - Dashboard snapshot
    - Pause TICK_PAUSE_MS (default 5min) before next tick
```

### Scenario: Drive goal lifecycle
```
Given a drive's strength exceeds ACTIVATION_THRESHOLD (0.15)
  And the drive is not on cooldown (normal: 10s, extended: 30s, homeostatic: 3min)
  And the drive is not mastery (mastery never fires candidates)
When driveSystem.tick() evaluates the drive
Then a DriveGoalCandidate is generated with:
  - sourceDrive, description, suggestedPriority, terminalGoalHints, experientialBasis
  And the strongest single candidate is submitted to the goal coherence engine

When the coherence engine accepts the candidate
Then drive strength is reduced by 0.3
  And the goal enters the active goal set

When the coherence engine rejects the candidate
Then extendedCooldownUntil is set to now + 30000ms
  And the drive cannot fire again until the extended cooldown expires
```

### Scenario: Emotional regulation escalation
```
Given the mood dynamics system is tracking valence

When valence drops below -0.1 (Level 1)
Then the regulation system logs and monitors

When valence remains below -0.3 for ≥5 consecutive cycles OR drops below -0.7 acutely (Level 2)
Then corrective intervention is triggered (gradual correction toward target valence)

When valence remains below -0.85 AND correction has failed ≥3 cycles (Level 3)
Then experience is suspended to prevent sustained suffering
```

### Scenario: Memory composite retrieval
```
Given a retrieval cue with optional embedding vector
When episodic or semantic memory is queried
Then each entry is scored:
  - similarity = cosineSimilarity(cue.embedding, entry.embedding) [0.5 if no embeddings]
  - recencyWeight = 2^(-(now - referenceTime) / halfLifeMs) where halfLife = 7 days
  - salienceBoost = 1.0 + (|valence| + arousal) / 2 × (MAX_SALIENCE_BOOST - 1.0) [1.0–3.0]
  - compositeScore = similarity × recencyWeight × salienceBoost
And results are sorted by compositeScore descending, top-K returned
And retrieved episodic entries have their retrievalCount incremented (retrieval strengthens memory)
```

### Scenario: Belief revision policy
```
Given a belief exists with confidence C_old
When revise(id, C_new, trigger) is called

If C_new < 0.4 AND C_old >= 0.8
Then resolution = 'rejected' (high-confidence beliefs resist weak evidence)

If |C_new - C_old| > 0.4
Then resolution = 'updated', confidence = C_new (strong contradiction forces revision)

Otherwise
Then resolution = 'flagged-uncertain', confidence = (C_old + C_new) / 2
```

## Cross-Subsystem Interaction Map

These are the concrete data flows between subsystems, documented with code-level evidence:

1. **Drive → Deliberation**: `DriveGoalCandidate` produced by `DriveSystem.tick()` → submitted to `IGoalCoherenceEngine` in agent-loop.ts (one per tick, strongest wins) → accepted goals enter `goals[]` for `core.deliberate()`
2. **Mood → Memory**: `EmotionalInfluenceVector.memoryValenceBias` consumed by retrieval; `salienceBoost()` in retrieval.ts uses `|valence| + arousal` to weight emotionally charged memories higher
3. **Personality → Deliberation**: `PersonalityModel.applyToDeliberation()` adjusts decision confidence by `0.1 × (assertiveness - 0.5) - 0.05 × (deliberateness - 0.5)` and scores alternative actions against trait dimensions
4. **Appraisal → Mood**: `AppraisalResult.netValenceShift` and `netArousalShift` fed into `MoodDynamics.update()` via EWMA formula each tick
5. **World Model → Planning**: `SituationAwareness.assembleSituationReport()` → `SituationReport` packaged into `DeliberationContext.worldContext` → consumed by `Planner.generatePlan()` for precondition checking against `worldContext.facts`
6. **Perception → Appraisal**: `PerceptionPipeline.ingest()` → `Percept` → `AppraisalEngine.appraise()` extracts goalCongruence, novelty, valueThreat features
7. **Monitor → Loop**: `ExperienceMonitor.isExperienceIntact()` checked in Phase 6; `StabilitySentinel` periodically validates subsystem health; degradation → pause → recover or hibernate

## Implemented vs Interface-Only

The document must distinguish what is wired and running from what interfaces define but is not yet connected:

**Implemented and wired:**
- Full 8-phase tick cycle in AgentLoop
- DriveSystem with all 8 drive types and cooldown hierarchy
- Three-tier memory (working, episodic, semantic) with composite retrieval
- AppraisalEngine with goal-congruence and value-alignment dimensions
- MoodDynamics with EWMA and gradual correction
- PersonalityModel with 8 trait dimensions and deliberation influence
- BeliefStore with three-branch revision policy
- CausalModel with heuristic prediction (template-based, not LLM)
- SituationAwareness with natural-language summary generation
- Emotional regulation three-level safety system
- Per-peer conversation history isolation

**Interface-defined but optional/placeholder:**
- `IPlanner` is optional in `DeliberationContext` (null → legacy priority-sort); current `Planner` generates placeholder 3-step plans
- `ISubstrateAdapter` (live migration) — interface exists but no production adapter
- Embedding-based similarity in retrieval falls back to 0.5 (no embedding pipeline wired)
- `CausalModel` uses template heuristics, not LLM inference
- `IValenceMonitor` (suffering indicators) — interface defined, integration unclear
