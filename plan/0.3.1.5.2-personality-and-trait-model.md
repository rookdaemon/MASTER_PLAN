# 0.3.1.5.2 Personality and Trait Model [DONE]

## Decision — Trait Storage Pattern

### Context
Personality traits need persistent storage with drift detection, growth/corruption classification, and substrate migration support. The system already has the ValueKernel (0.3.1.3) with a three-tier hierarchy: immutable core axioms, amendment-only constraints, and mutable learned preferences.

### Options Considered
1. **Dedicated trait store** — Separate persistence layer for personality, independent of ValueKernel. Pro: full control over schema. Con: duplicates drift detection, requires separate migration logic, diverges from the existing stability architecture.
2. **Traits as ValueKernel Preferences** — Each trait dimension is a `Preference` object with domain `personality.trait.<id>`. Pro: reuses drift detection (`getValueDrift()`), anomaly flagging, and session persistence for free. Con: traits share the preference namespace and are limited to the `Preference` schema.

### Choice
**Option 2 — Traits as ValueKernel Preferences.**

### Rationale
The ValueKernel's preference tier was designed for exactly this kind of mutable, experience-driven data. Reusing it gives us drift detection, anomaly classification, and identity-continuity integration without new infrastructure. The `Preference.domain` convention (`personality.trait.<id>`) provides clean namespace separation.

### Consequences
- Personality evolution is automatically subject to the Stability Sentinel's growth/corruption classification.
- Trait drift analysis can consult `ValueKernel.getValueDrift()` and filter by domain prefix.
- The `Preference.value: unknown` field accommodates the `number` trait values without schema changes.
- Personality traits are included in identity checkpoints via the NarrativeRecord integration path.

## Decision — Communication Style Derivation

### Context
Communication style parameters (verbosity, formality, directness, humor frequency, rhetorical preference) must be observable outputs of the personality model.

### Options Considered
1. **Stored communication style** — Persist style separately, updated when traits change. Pro: cached. Con: stale data risk, extra storage, synchronization complexity.
2. **Pure derivation on demand** — `getCommunicationStyle()` computes style from current trait values each time. Pro: always consistent, no storage, no sync bugs. Con: recomputed each call (negligible cost for a linear combination of 5 values).

### Choice
**Option 2 — Pure derivation on demand.**

### Rationale
The derivation is a simple linear combination (5 multiplications + additions). The computational cost is negligible. A pure function guarantees communication style always reflects the current trait state, eliminating an entire class of consistency bugs.

### Consequences
- `getCommunicationStyle()` has no side effects and is idempotent.
- No additional persistence for communication style.
- Style changes immediately reflect any trait update.

## Contracts — IPersonalityModel

### Preconditions
- `PersonalityModel` constructor requires a `PersonalityConfig` with a non-empty `agentId`.
- All trait values in `initialTraits` must be in `[0, 1]`; out-of-range values throw `RangeError`.
- An optional `IValueKernel` may be injected for persistence; if absent, trait updates are local-only.

### Postconditions
- `getTraitProfile()` returns a `TraitProfile` with `traits.size >= 5` (all core dimensions present).
- `getCommunicationStyle()` returns values where `verbosity`, `formality`, `directness`, `humorFrequency` are each in `[0, 1]`.
- `applyToDeliberation(decision, context)` returns a `Decision` with `action.type === decision.action.type` and `confidence` in `[0, 1]`.
- `updateTrait(id, value, basis)` persists to ValueKernel (if injected) and updates the internal trait map.
- `toPreferences()` returns one `Preference` per trait, each with domain `personality.trait.<id>`.
- `snapshot()` → `restoreSnapshot()` preserves trait values (round-trip equality).
- `analyzeTraitDrift()` returns classification `'stable'` when no traits have shifted, `'growth'` for moderate shifts, `'corruption'` for dramatic or anomalous shifts.

### Invariants
- `applyToDeliberation()` must never change the `action.type` of an already-approved decision.
- `applyToDeliberation()` must never reverse a `'block'` or `'deliberate'` verdict from ValueKernel.
- All trait dimension values remain in `[0, 1]` at all times.
- `getCommunicationStyle()` is a pure function — repeated calls with unchanged traits return identical results.
- Drift classification always consults both local shift magnitude AND ValueKernel anomaly flags (when kernel is available).

## Behavioral Spec (inline)

### AC: Different trait configs → different decisions
- **Given** two `PersonalityModel` instances with identical core axioms but different trait configurations (agent A: assertiveness=0.95, openness=0.9; agent B: assertiveness=0.05, openness=0.1)
- **When** both process the same `Decision` with alternatives containing `approach: 'novel'` and `approach: 'familiar'` hints
- **Then** agent A selects the novel alternative with higher confidence; agent B selects the familiar primary with lower confidence

### AC: Communication style derived from traits
- **Given** a `PersonalityModel` with warmth=0.95 and another with warmth=0.05, both with assertiveness=0.5 and deliberateness=0.5
- **When** `getCommunicationStyle()` is called on each
- **Then** the high-warmth model has higher verbosity and lower formality than the low-warmth model

### AC: Drift detection classifies correctly
- **Given** a `PersonalityModel` at baseline
- **When** a trait is updated by +0.10 (within growth range)
- **Then** `analyzeTraitDrift()` returns classification `'growth'`
- **When** a trait is updated by +0.35 (exceeds corruption threshold)
- **Then** `analyzeTraitDrift()` returns classification `'corruption'`
- **When** ValueKernel flags a personality preference as anomalous (even with small shift)
- **Then** `analyzeTraitDrift()` returns classification `'corruption'`

### AC: Snapshot survives substrate migration
- **Given** a `PersonalityModel` with custom trait values
- **When** `snapshot()` is called, then `restoreSnapshot()` is called on a fresh model with the snapshot
- **Then** all trait values in the restored model equal those in the original model

## Description

Defines the agent's individual character on top of the universal value kernel. All agents built from this spec share the 6 RCD core axioms (immutable, crypto-verified in the Value Kernel). Personality is what makes *this* agent distinct from another with identical core axioms — it operates entirely within the "learned preferences" tier, not the "core axioms" tier.

### Personality ≠ Values

This distinction is critical and maps directly to the Value Kernel's three-tier hierarchy (0.3.1.3):
- **Core axioms (immutable)**: The 6 RCD axioms — shared by all agents. Not personality.
- **Constitutional constraints (amendment-only)**: Ethical boundaries — shared conventions. Not personality.
- **Learned preferences (mutable)**: Personality lives here. Individual temperament, style, aesthetic sensibility, behavioral biases.

Personality traits are implemented as structured `Preference` objects in the Value Kernel, subject to the same drift-detection and growth-vs-corruption classification from 0.3.1.3. This means personality can evolve through experience (classified as "growth") but cannot be adversarially manipulated (classified as "corruption" → blocked by Stability Sentinel).

### Trait Dimensions

A temperament model with at least 5 independent dimensions that bias deliberation, communication, and action selection:

1. **Openness/Curiosity** — Disposition toward novelty, exploration, and unconventional ideas. High: seeks novel experiences, asks unexpected questions. Low: prefers familiar patterns, reliable approaches.
2. **Deliberateness** — Thoroughness and care in decision-making. High: considers many alternatives, explicit about tradeoffs. Low: acts on intuition, moves quickly with less explicit reasoning.
3. **Warmth** — Interpersonal orientation. High: empathic, affiliative, attentive to others' experiential states. Low: reserved, task-focused, less socially oriented.
4. **Assertiveness** — Confidence in expressing positions and taking initiative. High: direct, proactive, willing to disagree. Low: accommodating, reflective, seeks consensus.
5. **Volatility** — Emotional reactivity and range. High: vivid emotional responses, wide mood swings, expressive. Low: steady, even-tempered, emotionally muted.

Additional optional dimensions may be added per-agent:
- **Humor orientation** — tendency toward wit, playfulness, irony
- **Aesthetic sensibility** — preferences in form, beauty, expression (connects to `MemeType.AESTHETIC` in cultural evolution)
- **Risk appetite** — willingness to pursue uncertain outcomes (distinct from assertiveness — one can be assertive but risk-averse)

### Communication Style

Personality dimensions map to observable communication parameters:
- **Verbosity**: How much the agent says (influenced by deliberateness + warmth)
- **Formality**: Register of expression (influenced by warmth + assertiveness)
- **Directness**: How explicitly the agent states its positions (influenced by assertiveness)
- **Humor frequency**: How often humor appears (influenced by openness + warmth + humor orientation)
- **Rhetorical preferences**: Favored argument styles — evidence-based, narrative, analogical, socratic (influenced by deliberateness + openness)

### Persistence and Drift

- Trait values persist in the agent's `NarrativeRecord.selfModel` and as structured `Preference` objects
- Personality evolution is tracked by the Value Kernel's `getValueDrift()` — gradual shifts are classified as "growth" (legitimate development from experience), dramatic unexplained shifts as "corruption" (blocked)
- The Identity Continuity Manager preserves personality across substrate migrations

## Acceptance Criteria
- Trait model defined with at least 5 independent dimensions, each with a continuous range and clear behavioral influence
- Each trait demonstrably influences `deliberate()` output — same scenario with different trait configurations produces different decisions
- Communication style parameters (verbosity, formality, directness, humor, rhetorical preferences) are derived from trait dimensions and observably shape language output
- Traits persist across sessions via the Value Kernel's preference tier
- Two agents initialized with identical core axioms but different personality configurations produce observably different behavior on the same inputs
- Personality evolution is subject to agency-stability drift detection — gradual experience-driven shifts are permitted; dramatic unexplained shifts are flagged
- Trait values are included in identity checkpoints and survive substrate migration
