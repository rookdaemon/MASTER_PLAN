# 0.3.1.5.11 Task Journal and Goal Decomposition [DONE]
## Description

Persistent task decomposition that bridges the gap between abstract drive-generated goals and concrete executable work. The TaskJournal gives the agent cross-tick continuity: a high-level goal breaks into ordered subtasks, each with a completion criterion and an output slot. The journal survives restarts and provides the agent's "what am I doing right now?" context.

### Why This Matters

Without task decomposition, the agent receives a drive goal ("explore world model uncertainty") but has no mechanism to break it into steps, track partial progress, or resume after a restart. The TaskJournal is the bridge between motivation (drives) and execution (tool calls). It also feeds the AgentDigest's "Current Focus" section, so the LLM always knows what it should be working on.

### Design

- Tasks contain ordered subtasks, each with a description, completion criterion, and optional output slot
- Only one task is active at a time; others are queued (FIFO promotion when the active task completes)
- `forceActive` option abandons the current task to promote a more urgent one
- Subtask progression: when a subtask completes, the next pending subtask auto-activates
- When all subtasks complete, the task itself completes and the next queued task is promoted
- Tasks link to source goal IDs for drive traceability
- Compact formatting for digest injection: shows progress fraction and next subtask criterion

## Decision: Single Active Task Constraint

**Context:** The agent could pursue multiple tasks in parallel or be restricted to a single active task at a time. Parallel pursuit risks context thrashing in the LLM and makes progress tracking ambiguous.

**Options Considered:**
1. **Parallel active tasks** — Multiple tasks active simultaneously; agent selects which to advance each tick.
2. **Single active task with queue** — Only one task active; others queued FIFO and promoted on completion.

**Choice:** Option 2 — Single active task with FIFO queue.

**Rationale:** The agent operates through a single LLM context window. Presenting multiple active tasks creates ambiguity about what to do next and splits attention. A single active task with clear "next subtask" gives the LLM unambiguous direction. The queue provides a backlog without cognitive overload. The `forceActive` escape hatch handles urgency.

**Consequences:** An urgent new goal must either wait in the queue or explicitly abandon the current task via `forceActive`. Acceptable because drive-generated goals already compete at the drive system level before reaching the task journal.

## Decision: Persistence Format

**Context:** Task state must survive restarts. Options range from a database to flat files.

**Options Considered:**
1. **SQLite** — Structured queries, ACID transactions.
2. **JSON file** — Simple, human-readable, no dependencies.

**Choice:** Option 2 — Single JSON file (`state/tasks.json`).

**Rationale:** Task volume is low (tens of tasks, not thousands). JSON is human-inspectable, requires no additional dependencies, and the entire task list fits comfortably in memory. Atomic write via `writeFileSync` is sufficient for a single-process agent.

**Consequences:** No concurrent access protection. Acceptable because the agent is single-threaded.

## Contracts

### TaskJournal

**Preconditions:**
- `workspacePath` exists and is writable
- `createTask()` receives at least one subtask with non-empty description and criterion
- `completeSubtask()` receives valid taskId and subtaskId

**Postconditions:**
- `createTask()` returns a Task with status `active` (if no other active task) or `queued` (if one exists)
- `completeSubtask()` marks the subtask done, activates the next pending subtask, and returns `{ taskDone: true }` when all subtasks are complete
- When a task completes, the next queued task is promoted to active and its first subtask activated
- `abandonTask()` sets status to `abandoned` with a completedAt timestamp
- All mutations persist to `tasks.json` immediately

**Invariants:**
- At most one task has status `active` at any time
- Task IDs are unique (timestamp-based)
- Subtask IDs are unique within their parent task
- `all()` returns a defensive copy

## Behavioral Spec

### Scenario: Task creation with no active task
```
Given the TaskJournal has no active task
When createTask() is called with title "Explore plan cards" and 3 subtasks
Then a new task is created with status "active"
  And the first subtask has status "pending"
  And the task's sourceGoalId links to the originating drive goal
```

### Scenario: Task creation while another is active
```
Given the TaskJournal has an active task "Explore plan cards"
When createTask() is called with title "Reply to peer message"
Then the new task is created with status "queued"
  And the original task remains "active"
```

### Scenario: Force-active abandons current task
```
Given the TaskJournal has an active task "Explore plan cards"
When createTask() is called with forceActive: true
Then "Explore plan cards" is marked "abandoned" with a completedAt timestamp
  And the new task becomes "active"
```

### Scenario: Subtask completion promotes next subtask
```
Given an active task with subtasks [A (active), B (pending), C (pending)]
When completeSubtask() is called for subtask A with output "found 3 cards"
Then subtask A status is "done" with output "found 3 cards"
  And subtask B status is "active"
  And completeSubtask returns { taskDone: false }
```

### Scenario: All subtasks done promotes queued task
```
Given an active task with one remaining subtask and a queued task behind it
When completeSubtask() marks the last subtask done
Then the active task status becomes "done" with completedAt
  And the queued task is promoted to "active"
  And the queued task's first subtask is activated
```

## Acceptance Criteria
- Tasks decompose into ordered subtasks with completion criteria
- Only one task is active at a time; others queue behind it
- Subtask completion auto-promotes the next pending subtask
- Task completion auto-promotes the next queued task
- Agent tools (task_create, task_update) are available for LLM-driven task management
- Tasks persist to disk and survive agent restarts
- Active task summary is injected into the agent digest for LLM context
- Source goal ID links tasks back to the drive that spawned them
