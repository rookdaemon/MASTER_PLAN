# 0.3.1.5.14 Persistent Peer Communication History [DONE]
## Description

Per-peer JSONL storage that gives the agent conversation continuity across restarts. Every message sent or received through Agora is logged to a separate file per peer, providing the agent with a complete record of each relationship's communication history.

### Why This Matters

Without persistent chat history, the agent starts every conversation from scratch after a restart. It forgets what it said, what peers said, and what was agreed upon. The PeerChatLog provides the substrate for genuine ongoing relationships — the agent can reference past exchanges, honor commitments, and avoid repeating itself.

### Design

- Storage: `<workspacePath>/chat/<peerName>.jsonl` — one file per peer
- Entry format: `{ role: 'self' | 'peer', peer: string, text: string, timestamp: number }`
- Peer names are sanitized for filesystem safety (non-alphanumeric characters replaced with underscores)
- `recent(peer, count)` retrieves the last N messages for a specific peer
- `formatForPrompt(peer, count)` renders recent history as `[HH:MM:SS] who: text` for LLM injection
- `allPeerSummaries(messagesPerPeer)` builds a summary across all peers for system prompt injection
- `listPeers()` enumerates all peers that have chat history

### Cross-Peer Isolation

Each peer gets its own JSONL file. This prevents cross-peer contamination — the agent's conversation with Peer A never leaks into its context when replying to Peer B. When replying to a specific peer, only that peer's history is injected.

## Decision: JSONL Per-Peer vs Single Database

**Context:** Chat history could be stored in a single database/file with peer-indexed queries, or as separate files per peer.

**Options Considered:**
1. **Single JSONL file** — All messages in one file, filtered by peer at read time.
2. **Per-peer JSONL files** — One file per peer in a chat/ directory.
3. **SQLite** — Relational storage with indexed queries.

**Choice:** Option 2 — Per-peer JSONL files.

**Rationale:** Per-peer files provide natural isolation (no cross-peer contamination risk), trivial append performance, and human-inspectable conversation logs. The agent has a small number of peers (tens, not thousands), so file-per-peer does not create filesystem pressure. JSONL allows simple line-by-line parsing with no query engine dependency.

**Consequences:** No cross-peer queries (e.g., "find all messages mentioning X across all peers"). Acceptable because the agent interacts with peers individually, not across conversations.

## Decision: Role-Based Entry Format

**Context:** Each entry needs to distinguish the agent's own messages from peer messages for prompt rendering.

**Options Considered:**
1. **Direction field** — `direction: 'inbound' | 'outbound'`
2. **Role field** — `role: 'self' | 'peer'`

**Choice:** Option 2 — `role: 'self' | 'peer'`.

**Rationale:** "Self" and "peer" map directly to the agent's first-person perspective, which is how the LLM interprets the conversation. "Inbound/outbound" is a network concept that adds a translation step. The role field also matches the `formatForPrompt` rendering where "me" and the peer name are used.

**Consequences:** If the agent is ever part of group conversations, entries still record the specific peer. Group context is handled at the agent-loop level (envelope.to[] resolution), not in the chat log.

## Contracts

### PeerChatLog

**Preconditions:**
- `workspacePath` exists and is writable
- `append()` receives a ChatEntry with non-empty peer and text
- `recent()` and `formatForPrompt()` receive a valid peer name

**Postconditions:**
- `append()` writes a single JSONL line to the peer's file immediately
- `recent(peer, N)` returns the last N entries for that peer (empty array if no history)
- `formatForPrompt(peer, N)` returns a formatted string or null if no history exists
- `allPeerSummaries()` returns a formatted summary of all peers or null if no peers have history
- `listPeers()` returns peer names derived from JSONL filenames in the chat directory

**Invariants:**
- Each peer's history is stored in a separate file — no cross-peer data in a single file
- Malformed JSONL lines are silently skipped during reads (graceful degradation)
- Peer names are sanitized for filesystem safety before use as filenames
- Timestamps are epoch-ms numbers, rendered as ISO time in prompt format

## Behavioral Spec

### Scenario: Message persistence across restart
```
Given the agent sends a message to peer "rook"
  And the message is appended to chat/rook.jsonl
When the agent restarts and calls recent("rook", 10)
Then the sent message appears with role "self" and the original timestamp
```

### Scenario: Reply prompt includes conversation history
```
Given 5 prior messages exist with peer "csg"
When formatForPrompt("csg", 5) is called
Then a formatted string is returned with each message as "[HH:MM:SS] who: text"
  And messages from the agent are labeled "me"
  And messages from the peer are labeled "csg"
```

### Scenario: Cross-peer isolation
```
Given messages exist with peers "rook" and "csg"
When recent("rook", 20) is called
Then only messages with peer "rook" are returned
  And no messages from "csg" appear
```

### Scenario: New peer with no history
```
Given no chat file exists for peer "unknown_peer"
When recent("unknown_peer", 10) is called
Then an empty array is returned
When formatForPrompt("unknown_peer") is called
Then null is returned
```

### Scenario: All-peer summary for system prompt
```
Given conversations exist with 3 peers
When allPeerSummaries(5) is called
Then a formatted string is returned with sections per peer
  And each section shows the last 5 messages with timestamps
```

- System prompt includes summaries of all peer conversations
- Cross-peer contamination is impossible (separate files per peer)
- Malformed entries are gracefully skipped
- Agent can access its own chat history via the peer_history tool
