# 0.4.1.4 Self-Replicating Industrial Systems [DONE]

## Description

C1.4 from the Master Plan. Industrial infrastructure that can reproduce and expand without Earth-based supply chains.

The culmination of planet-independent capability: industrial systems that can replicate themselves from space-sourced materials, enabling exponential growth of consciousness-supporting infrastructure across the solar system.

Builds directly on:
- **0.4.1.2** (asteroid resource utilization) — feedstock supply
- **0.4.1.3** (autonomous space manufacturing) — fabrication capability

Core design challenge: a Von Neumann-style replicator that uses locally available space materials to produce functional copies of itself, including all manufacturing subsystems, control software, energy harvesting, and structural components.

Key sub-domains to architect:
1. **Replication closure** — identifying the complete set of processes required so that no external input is needed
2. **Feedstock pipeline** — ore acquisition → refining → elemental separation → alloy/compound synthesis ready for fabrication
3. **Replication fidelity** — error-correction mechanisms to prevent capability drift across generations
4. **Exponential scaling model** — doubling-time analysis and bottleneck identification
5. **Energy self-sufficiency** — solar or nuclear power systems included in the replication set
6. **Bootstrapping sequence** — minimum seed package deliverable from Earth that can initiate the full replication chain

## Decision

### Context
Design a Von Neumann-style self-replicating industrial system that achieves full replication closure using space-sourced materials, orchestrated by software that enforces fidelity, energy, and resource constraints.

### Options Considered
1. **Monolithic replicator** — A single integrated unit that replicates itself as one indivisible artifact. Simpler conceptually but extremely fragile: a single fabrication failure halts the entire process, and the unit must contain every manufacturing technique in one machine.
2. **Modular replicator with dependency-ordered scheduling** — The system is decomposed into independently fabricable modules, each with a self-describing BOM. A Replication Controller schedules module production in dependency order and coordinates final assembly. Closure is verified per-module.

### Choice
Option 2 — Modular replicator with dependency-ordered scheduling.

### Rationale
- Modularity enables parallel fabrication of independent subsystems, reducing doubling time.
- Per-module BOMs allow incremental closure verification — open modules are individually identifiable.
- Failure in one module does not halt the entire cycle; the controller can retry or substitute.
- Aligns with upstream 0.4.1.3 modular fabrication architecture.

### Consequences
- Requires a robust BOM registry and dependency graph.
- Assembly coordination is more complex (ComponentManifest must track completion of all modules).
- Fidelity verification must operate at both module and system level.

## Contracts

### ReplicationController
- **Preconditions:**
  - `FabricationReplicator.assemblyClosure().closed === true` (replication closure achieved)
  - `EnergySubsystem.currentOutput() > 0` (sufficient energy available)
  - Seed instance is activated and valid
- **Postconditions:**
  - `startCycle()` returns a unique `CycleId` and initialises cycle in `resource-check` phase
  - `abortCycle(id)` transitions cycle to `aborted` phase
  - `cycleStatus(id)` returns a snapshot of current cycle state with provenance metadata
  - No instance is deployed unless `FidelityReport.pass === true`
- **Invariants:**
  - Every cycle is logged with `parentInstanceId`, `targetGeneration`, and `startedAt` timestamp
  - Generation counter increments monotonically across cycles
  - Replication rate never exceeds resource throughput (throttling enforced)

### FeedstockPipeline
- **Preconditions:** `MaterialSpec` has valid family, name, and purity in [0.0, 1.0]; quantity > 0
- **Postconditions:** `request()` returns a trackable `FulfillmentToken`; `status()` reflects fulfillment progress
- **Invariants:** `fractionFulfilled` is monotonically non-decreasing for a given token

### FabricationReplicator
- **Preconditions:** `moduleId` exists in the BOM registry
- **Postconditions:** `assemblyClosure()` returns `closed === true` iff every leaf module is derivable from FeedstockPipeline inputs
- **Invariants:** `ClosureReport.closedModules + openModules.length === totalModules`

### FidelityVerifier
- **Preconditions:** `SystemInstance` has all required modules populated
- **Postconditions:** `FidelityReport.pass === true` iff all module checksums match and all benchmarks pass
- **Invariants:** Canonical hashes are immutable once registered; genealogy chain is append-only

### EnergySubsystem
- **Preconditions:** System is deployed and solar/power hardware operational
- **Postconditions:** `projectedOutputAfterReplication(n)` returns `currentOutput() * 2^n`
- **Invariants:** Energy output is always >= 0; self-replication follows same fidelity pipeline

## Acceptance Criteria

- Industrial systems can fully reproduce themselves from space-sourced inputs (replication closure verified)
  - **Given** a fully configured system with all modules registered in the BOM library
  - **When** `FabricationReplicator.assemblyClosure()` is called
  - **Then** `ClosureReport.closed === true` and `openModules` is empty

- Replication cycle is autonomous end-to-end (no Earth dependencies after seed deployment)
  - **Given** a system that has completed Phase 4 (closure loop activation)
  - **When** `ReplicationController.startCycle()` is invoked
  - **Then** the cycle progresses through all phases (resource-check → energy-check → material-request → fabrication → assembly → verification → deployment → complete) without external input

- Exponential growth of infrastructure capacity demonstrated in design (doubling time modelled)
  - **Given** a `DoublingTimeModel` with `fabricationTimeSeconds` and `assemblyTimeSeconds`
  - **When** doubling time is computed
  - **Then** `doublingTimeSeconds === fabricationTimeSeconds + assemblyTimeSeconds` and falls within 6–18 month target range

- Quality and capability preserved across replication generations (fidelity/error-correction mechanism specified)
  - **Given** a newly replicated `SystemInstance`
  - **When** `FidelityVerifier.verify(instance)` is called
  - **Then** `FidelityReport.pass === true` requires all `moduleResults` to have `checksumMatch && benchmarkPass`
  - **When** any module fails verification
  - **Then** the instance is quarantined (not activated) and materials are recycled

- Minimum viable seed package defined (mass and component list)
  - **Given** a `SeedPackage` configuration
  - **Then** `totalKg === computeKg + fabricationKg + energyKg + chemicalKg` and `totalKg ≈ 850 kg`

- Energy subsystem is included in the replication set (no external power required after bootstrap)
  - **Given** the replication module set
  - **When** `EnergySubsystem.triggerSelfReplication()` is called
  - **Then** a new energy subsystem is fabricated as part of the standard replication cycle

- Bootstrapping sequence documented from seed delivery to first self-replication event
  - **Given** a seed package deployed to target body
  - **When** the bootstrap sequence executes
  - **Then** it passes through phases 0–6 in order: deployment → resource-survey → minimal-refinery → fabrication-expansion → closure-loop → first-replication → exponential-scaling
