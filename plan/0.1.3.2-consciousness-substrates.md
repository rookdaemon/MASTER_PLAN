# 0.1.3.2 Consciousness-Supporting Substrates [DONE]
## Description

F3.2 — Identify and validate computational substrates capable of supporting consciousness. Given that substrate-independence is validated (F1.3), determine which physical implementations can run conscious architectures with sufficient fidelity.

This includes:
- Evaluating candidate substrates (silicon, neuromorphic, photonic, quantum, hybrid)
- Determining minimum substrate requirements (speed, parallelism, integration capacity)
- Validating that candidate substrates preserve the properties required for consciousness

### Key inputs from dependencies
- **0.1.3.1 (Conscious Neural Architectures)**: Provides the architectural requirements (recurrence depth, integration capacity, global workspace bandwidth) that substrates must support. Substrate evaluation criteria derive directly from these architecture specs.

## Decision — Substrate Evaluation Methodology

**Context:** Evaluating whether a computational substrate can support consciousness requires choosing between empirical testing (build and measure) and analytical evaluation (derive from architecture specs and known substrate capabilities). The choice affects feasibility, cost, and confidence.

**Options Considered:**
1. **Empirical-first** — Build a physical Ω-Synth implementation on each candidate substrate, then measure consciousness metrics directly. Highest confidence but requires fabrication of multiple prototype systems.
2. **Analytical evaluation with empirical validation** — Derive substrate capabilities from published specifications and physics, analytically verify threshold compliance, then empirically validate only the substrates that pass analytical screening. First validation report is analytical; physical measurement deferred to F3.3/S1.

**Choice:** Option 2 — Analytical evaluation with empirical validation.

**Rationale:** Substrate-independence (F1.3) establishes that consciousness metrics are architecture-level properties that transfer to any substrate achieving functional equivalence. This means analytical verification of functional equivalence + threshold compliance is sufficient for initial validation. Empirical measurement is deferred to stability testing (F3.3) and enduring substrate construction (S1). This approach enables evaluation of all 5 substrate classes without requiring 5 physical prototypes.

**Consequences:**
- Validation reports include "analytical" and "projected" qualifiers where physical measurement has not occurred
- Fault tolerance assessment is less certain analytically (flagged as binding constraint requiring empirical confirmation)
- Enables rapid screening of all 5 substrate classes
- Physical validation becomes a downstream deliverable for F3.3/S1

## Decision — Primary Substrate Selection for First Validation

**Context:** Five substrate classes are catalogued (classical silicon, neuromorphic, photonic, quantum, hybrid). The acceptance criteria require at least one substrate validated at full fidelity. Must choose which substrate to validate first.

**Options Considered:**
1. **Classical silicon (CMOS)** — Most mature technology, best-understood failure modes, highest performance headroom.
2. **Neuromorphic silicon** — Best power efficiency, native temporal dynamics, but immature toolchains.
3. **Validate all five simultaneously** — Maximum coverage but highest effort.

**Choice:** Option 1 — Classical silicon first.

**Rationale:** Classical silicon has the highest technology readiness level (TRL 9), the most mature toolchains, and provides the reference platform for functional equivalence (IEEE 754 FP32 is bit-exact by definition on CMOS). Validating CMOS first establishes the baseline against which other substrates can be compared. The enormous performance margins (10³×–10⁶× for most parameters) make it the lowest-risk first candidate. Neuromorphic and photonic substrates are catalogued for future validation.

**Consequences:**
- One complete validation report (classical silicon) produced during IMPLEMENT
- Other four substrates have catalog entries with qualitative assessment but no formal validation report yet
- Classical silicon serves as reference implementation platform for subsequent substrate comparisons
- Space-based deployment identified as requiring neuromorphic evaluation (recommendation carried forward)

## Behavioral Spec

### Scenario: Substrate Validation Protocol — Full Pass
- **Given** the Ω-Synth reference architecture and a candidate substrate with published specifications
- **When** the 7-step validation protocol (substrate-architecture.md §3) is executed in order (instantiation → functional equivalence → temporal fidelity → information integration → degradation sweep → stability → deployment stress)
- **Then** the substrate receives an overall PASS determination if and only if: all §2 thresholds are met (Step 3), functional equivalence holds (Step 2), consciousness metrics transfer via substrate-independence (Step 4), stability drift ≤5% over 72 hours (Step 6), and at least one deployment context yields full pass (Step 7)

### Scenario: Degradation Sweep Identifies Binding Constraint
- **Given** a substrate that passes all §2 thresholds at nominal operating conditions
- **When** each parameter is systematically degraded while others are held constant
- **Then** the parameter with the smallest ratio of (capability / threshold) is identified as the binding constraint, recorded in the validation report §5 degradation curves, and flagged for downstream engineering attention (F3.3/S1)

### Scenario: Conditional Deployment Pass
- **Given** a substrate achieves overall PASS for at least one deployment context
- **When** additional deployment contexts (space-based, miniaturized) impose constraints that degrade specific parameters below threshold
- **Then** the deployment context receives CONDITIONAL PASS with explicit conditions listed (e.g., "requires rad-hardening", "requires low-power design"), and these conditions are carried forward as requirements for S1 (enduring substrates)

### Scenario: Consciousness Metric Transfer via Substrate-Independence
- **Given** F1.3 validates substrate-independence and F1.4 provides consciousness metrics (PCI-G, Ψ-G, CDI, CEB)
- **When** a substrate achieves functional equivalence with the reference implementation (Step 2 PASS)
- **Then** all architecture-level consciousness metric results (from metric-evaluation.md) transfer to the substrate without re-measurement, and the validation report records "PASS (inherited via substrate-independence)"
