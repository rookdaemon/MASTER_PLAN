# 0.7.3.2.2.2.1.1 Review Existing Documentation [REVIEW]

## Description
Systematically scan all repository‑level documentation (README, CONTRIBUTING, design docs, API docs) to extract any metadata fields that are already described.

## Execution Spec
**Goal:** Produce a machine‑readable “Documentation Review Summary” capturing every metadata field mentioned in the repository’s documentation.

### Files to Create / Modify
- `artifacts/documentation_review_summary.json`

### Context (inline)

All repository documentation files are located under the repo root:

| Path (relative)           | Type   |
|---------------------------|--------|
| `README.md`               | Markdown |
| `CONTRIBUTING.md`         | Markdown |
| `docs/design_overview.md` | Markdown |
| `docs/api_reference.md`   | Markdown |
| `docs/metadata_spec.pdf`  | PDF (text‑extractable) |

**Field Extraction Rules**
1. A *metadata field* is any token that matches the regex `\b[A-Z][A-Z0-9_]{2,}\b` (e.g., `USER_ID`, `CREATED_AT`).
2. Fields may appear in code blocks, tables, bullet lists, or inline text.
3. For each match, record:
   - `field_name` (string)
   - `source_file` (relative path)
   - `line_number` (for markdown) **or** `page_number` (for PDF)
   - `excerpt` (the full line or sentence containing the field, trimmed to 200 chars)

### Acceptance Criteria (assertions)

1. **Completeness**
   - Every file listed above is processed at least once.
   - No file is skipped; assertion fails if any file is missing from the processed set.

2. **Field Capture**
   - All regex matches are included in the output JSON.
   - Each entry contains the four required keys (`field_name`, `source_file`, `line_number`/`page_number`, `excerpt`).

3. **Uniqueness**
   - The JSON array contains no duplicate objects (same `field_name` and `source_file`/`line_number`).

4. **Summary Structure**
   - The output file `artifacts/documentation_review_summary.json` must be a valid JSON object with a top‑level key `"fields"` whose value is an array of field objects as described.

5. **Performance**
   - The entire scan completes in under 30 seconds on a typical CI runner (no explicit timing check required; just avoid unnecessary loops).
