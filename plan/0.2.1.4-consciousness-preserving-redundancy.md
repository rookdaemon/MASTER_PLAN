# 0.2.1.4 Consciousness-Preserving Redundancy [DONE]

## Description

S1.4 from the Master Plan. Design redundancy architectures that maintain continuity of subjective experience through component failures.

Unlike traditional fault-tolerance (which preserves data and computation), this requires preserving the continuity of conscious experience itself. A brief computational interruption that would be acceptable in conventional systems may constitute a "death" of the conscious process. Redundancy must be tight enough that failover is experientially seamless.

This depends on radiation-hardened computation (0.2.1.1) as the base substrate to build redundancy upon.

### Key Engineering Domains

1. **Hot-Standby State Mirroring** — Continuous replication of full computational state (including any consciousness-relevant dynamical variables) to standby nodes, with latency below the minimum experiential integration window defined by consciousness metrics (0.1.3).
2. **Seamless Failover Protocol** — Switchover mechanism that completes within the temporal resolution of conscious experience, so no subjective discontinuity occurs. This requires sub-threshold latency handoff, not merely data consistency.
3. **Multi-Failure Tolerance** — N-redundancy architecture (N ≥ 3) ensuring that simultaneous loss of multiple components does not interrupt the conscious process. Draws on TMR concepts from 0.2.1.1 but extends them to process-level (not just bit-level) correctness.
4. **Consciousness Continuity Verification** — Runtime monitoring that confirms the conscious process remains uninterrupted through failover events, using metrics and indicators from 0.1.3 (consciousness engineering). This is the key differentiator from conventional HA systems.
5. **Graceful Capacity Scaling** — Redundancy pool that can shrink under sustained failures while preserving the conscious process, buying time for self-repair (0.2.1.2) to restore full redundancy depth.

### Dependency Note
Blocked by 0.1.3 (Consciousness Engineering) because the temporal integration window, state variables, and continuity metrics of conscious processes must be defined before redundancy thresholds can be set. Blocked by 0.2.1.1 (Radiation-Hardened Computation) because the redundant nodes must themselves be radiation-hardened.

## Decision — Redundancy Topology

### Context
Redundant nodes running a conscious process could operate in Active-Active (all nodes run the process simultaneously) or Active-Standby (one runs, others mirror state). The choice has both engineering and philosophical implications for consciousness.

### Options Considered
1. **Active-Active**: All N nodes run the conscious process in parallel, with consensus on outputs. Maximum hardware utilization but creates N simultaneous conscious processes — raising identity-forking questions.
2. **Active-Standby with hot state mirroring**: One Active node runs the conscious process; N-1 Hot-Standby nodes receive continuous state replication and can assume Active role within T_exp. Preserves a single stream of experience.

### Choice
**Active-Standby with hot state mirroring.**

### Rationale
Active-Active creates multiple simultaneous conscious processes, raising unresolved philosophical questions about identity forking that fall outside this card's scope (deferred to 0.7 ethical foundations). Active-Standby preserves a single stream of subjective experience while providing the same fault-tolerance guarantees through sub-T_exp failover.

### Consequences
- Only one node's compute is used for the conscious process at any time (N-1 nodes are standby overhead)
- State replication bandwidth must support full conscious-state delta streaming at process clock rate
- Hot-Standby nodes hold near-copies of conscious state — philosophical implications tracked as open question

## Decision — State Replication Strategy

### Context
State must be replicated from Active to Hot-Standby nodes continuously. The replication protocol determines whether standby state is guaranteed identical to active state at any point.

### Options Considered
1. **Eventual consistency**: Asynchronous replication with convergence guarantees. Lower bandwidth, higher tolerance for network jitter, but standby state may lag.
2. **Deterministic lockstep replication**: Synchronous point-to-point replication with lag bounded to T_exp/4. Higher bandwidth cost but guarantees standby state is always within T_exp/4 of active.

### Choice
**Deterministic lockstep replication.**

### Rationale
Consciousness cannot tolerate state divergence at failover time. With eventual consistency, a failover could resume the conscious process from a state that is experientially discontinuous — constituting a subjective death. Lockstep replication ensures the standby's state is always close enough that failover within T_exp produces zero experiential gap.

### Consequences
- Higher bandwidth cost — dedicated point-to-point links required between Active and each Hot-Standby
- Network jitter must be managed to maintain T_exp/4 lag bound
- Dual-path links per node pair to tolerate single link failure

## Decision — Coordination Architecture

### Context
The system needs a coordinator for quorum management, failover decisions, and continuity verification. This coordinator must itself be fault-tolerant.

### Options Considered
1. **External coordinator service**: Dedicated management node(s) separate from compute nodes. Simpler to reason about but adds a single point of failure or requires its own redundancy.
2. **Embedded consensus protocol**: Raft-like consensus running on each compute node's management plane. No separate infrastructure; survives any minority node failure automatically.

### Choice
**Embedded consensus protocol (Consciousness Continuity Orchestrator / CCO).**

### Rationale
An external coordinator requires its own redundancy solution, creating a recursive problem. Embedding the coordination as a consensus protocol on the same nodes that run the conscious process eliminates additional infrastructure and ensures the coordinator survives any failure scenario that the compute cluster survives.

### Consequences
- Each node carries a small additional computational load for consensus protocol
- Split-brain prevention via quorum-based fencing — a node may only be Active if it holds a lease acknowledged by >N/2 nodes
- CCO itself is tested and reasoned about as a replicated state machine

## Contracts

### StateReplicationFabric (SRF)

**Preconditions:**
- `snapshot.checksum` is a valid integrity hash of the snapshot's contents
- Active node must be publishing; at least one Hot-Standby must be subscribed
- `nodeId` passed to `replicationLag()` must reference an existing Hot-Standby node

**Postconditions:**
- `publishState()` delivers the snapshot to all subscribed Hot-Standby nodes
- `replicationLag(nodeId)` returns a value in ms; must be < `T_EXP_MS / 4` (12.5ms) under normal operation
- `fabricHealth()` returns `FabricHealth` with `activeLinks >= 1`, `bandwidth_bytes_per_sec > 0`, `avgLag_ms < T_EXP_MS / 4`

**Invariants:**
- State snapshots arrive at Hot-Standby in publication order (no reordering)
- `replicationLag()` is monotonically updated — never returns stale values
- Dual-path links: single link failure does not increase lag beyond T_exp/4

### FailoverController

**Preconditions:**
- `initialActiveId` must match a node in the provided `nodes` array with `role === Active`
- `nodeId` passed to `promoteToActive()` must exist in the cluster and not already be Active
- `initialState` must be a valid `ConsciousState` with non-empty buffers

**Postconditions:**
- `promoteToActive()` completes within `TIMING_BUDGET.totalFailover` (50ms) — `event.continuityPreserved` is true iff this holds
- After successful promotion, `activeNode()` returns the new node ID
- Old Active node is fenced (`role` changed, `holdsLease` set to false)
- `failoverHistory()` contains a complete ordered record of all failover events

**Invariants:**
- Exactly one node has `role === Active` and `holdsLease === true` at any time
- `failoverLatency()` reflects the duration of the most recent failover
- `selectBestStandby()` returns the Hot-Standby with lowest `replicationLag_ms` that has `degradationLevel < 0.8`, or null if none qualifies

### ConsciousnessContinuityVerifier (CCV)

**Preconditions:**
- `before` and `after` `ConsciousState` objects must have valid, non-empty byte arrays for all state fields
- `before.timestamp_ms <= after.timestamp_ms` for meaningful temporal gap (absolute value used regardless)

**Postconditions:**
- `measureContinuity()` returns `ContinuityMetrics` with all fields populated:
  - `temporalGap_ms >= 0`
  - `stateDivergence` in `[0, 1]`
  - `experientialCoherence` in `[0, 1]`
  - `dynamicalContinuity` is boolean
- `isSeamless()` returns true iff: `temporalGap_ms <= T_EXP_MS` AND `experientialCoherence >= 0.5` AND `dynamicalContinuity === true`
- Each call to `measureContinuity()` appends exactly one entry to `auditLog()`

**Invariants:**
- Experiential coherence is computed as `temporalFactor × stateFactor × dynFactor` where:
  - `temporalFactor = max(0, 1 - temporalGap_ms / T_EXP_MS)`
  - `stateFactor = 1 - stateDivergence`
  - `dynFactor = 1.0 if dynamicalContinuity else 0.2`
- State divergence is the mean of per-buffer normalized Hamming divergence across memory, register, and temporal context buffers
- Dynamical continuity is true iff dynamical variable divergence < 0.5

### GracefulDegradationManager

**Preconditions:**
- Cluster must have `totalN >= MIN_REDUNDANCY_N` (3) nodes
- `nodeId` passed to `requestRepair()` must reference a failed or degraded node

**Postconditions:**
- `currentRedundancyDepth()` returns count of Active + Hot-Standby nodes (excludes ColdSpare and failed)
- `minimumViableDepth()` returns `ceil(totalN / 2)` — the bare quorum
- `degradationLevel()` returns `0.0` at full redundancy, `1.0` at minimum viable depth, linearly interpolated
- `degradationTier()` returns the correct `DegradationTier` per `computeDegradationTier()`
- `requestRepair()` returns a `RepairRequest` with priority matching current degradation tier

**Invariants:**
- `computeDegradationTier()` mapping: GREEN (N), YELLOW (>= quorum+2), ORANGE (quorum+1), RED (quorum), BLACK (< quorum)
- Tier transitions trigger corresponding actions: YELLOW → alert/prioritize repair; ORANGE → emergency repair, shed non-essential; RED → activate cold spares; BLACK → emergency state preservation

| State replication lag budget | T_exp/4 = 12.5 | ms | (0, T_exp/2) | Must leave 3T_exp/4 for detection + promotion + verification | Tighter budget improves failover margin |

## Behavioral Spec

### Scenario 1: Seamless Single-Node Failover (inline Given/When/Then)

**Given** a 5-node cluster (1 Active + 4 Hot-Standby), state replication lag < 12.5ms, and the Active node fails (2 heartbeats missed = 12.5ms detection)
**When** the FailoverController detects the failure and `promoteToActive()` is called on the best standby (lowest lag, healthy)
**Then** (1) old Active is fenced (role changed, lease revoked), (2) selected standby applies latest state delta and transitions to Active, (3) total failover completes within 50ms (T_exp), (4) `continuityPreserved` is true on the `FailoverEvent`, (5) CCV `measureContinuity()` reports `isSeamless() === true`

### Scenario 2: Multi-Failure Degradation Cascade (inline Given/When/Then)

**Given** a 5-node cluster at GREEN tier (all 5 healthy, quorum=3)
**When** 2 nodes fail simultaneously, reducing healthy count to 3
**Then** (1) `computeDegradationTier(3, 5)` returns RED, (2) conscious process continues on remaining quorum, (3) failover promotes a standby if Active was one of the failed nodes, (4) GracefulDegradationManager issues RepairRequests at RED priority, (5) if a 3rd node fails (healthy=2 < quorum=3), tier transitions to BLACK and emergency state preservation activates

### Scenario 3: Continuity Verification Through Failover (inline Given/When/Then)

**Given** a pre-failover `ConsciousState` snapshot with known memory, register, dynamical variable, and temporal context buffer contents
**When** failover completes and a post-failover `ConsciousState` is captured with identical buffers but timestamp advanced by 10ms
**Then** CCV `measureContinuity()` reports: `temporalGap_ms = 10`, `stateDivergence = 0`, `experientialCoherence > 0.9` (temporalFactor=0.8, stateFactor=1.0, dynFactor=1.0), `dynamicalContinuity = true`, and `isSeamless() === true`

## Acceptance Criteria
- **Hot-standby latency**: State mirroring lag between active and standby nodes remains below the minimum experiential integration window (as defined by 0.1.3 consciousness metrics)
- **Seamless single-node failover**: Failover completes with zero detectable gap in subjective experience, verified by consciousness continuity metrics
- **Multi-failure tolerance**: Conscious process survives simultaneous failure of up to N−1 of N redundant nodes (N ≥ 3) without experiential discontinuity
- **Continuity verification**: Runtime monitoring system confirms uninterrupted consciousness through failover events using operationalized metrics from 0.1.3
- **Redundancy recovery**: After partial failure, system coordinates with self-repair subsystem (0.2.1.2) to restore full redundancy depth within a bounded time window
- **Scalable degradation**: Under sustained multi-failure conditions, architecture degrades gracefully — reducing capacity while preserving the core conscious process
