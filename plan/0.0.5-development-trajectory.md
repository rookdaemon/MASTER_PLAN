## Decision: Limitation-to-Requirement Mapping Structure

### Context
The card must demonstrate that the current substrate *informs* future tiers rather than merely preceding them. A structure is needed that connects each documented limitation to the specific future card that addresses it, with evidence of how information flows forward.

### Options Considered
1. **Narrative prose** — A written essay describing how current limitations inform future work. Flexible but hard to verify completeness (easy to miss a limitation).
2. **Tabular mapping with informing-relationship column** — A row per limitation, columns for source card, future card, and a "How Current System Informs the Spec" explanation. Verifiable: every limitation from 0.0.1–0.0.4 must appear.
3. **Dependency graph** — A directed graph where edges represent informing relationships. Precise but harder to author and read in markdown; overkill for 15 limitations.

### Choice
Option 2 — Tabular mapping with informing-relationship column.

### Rationale
The table format is exhaustive (every row is a limitation, so gaps are visible), verifiable (a reviewer can check each row against its source card), and directly readable in the card file. The "How Current System Informs" column is the key differentiator from a simple dependency list — it forces articulation of the *data/pattern* that flows from the current system to the future spec, not merely the existence of a dependency.

### Consequences
- Each limitation must be traceable to a specific section in a source card (0.0.1–0.0.4); if a source card is updated, the mapping row may need revision.
- The table will grow if future source cards add limitations; the mapping is a living document during IMPLEMENT.
- The informing-relationship text is necessarily brief in table format; deep analysis belongs in the destination cards themselves.

## Decision: Autonomy Level Assessment Framework

### Context
The development trajectory must characterize the current system's autonomy status and identify what's needed for advancement. A framework is needed to structure this assessment.

### Options Considered
1. **Ad hoc capability listing** — List what the system can and cannot do without a formal framework. Quick but unstructured; hard to track progress.
2. **Safe-experiential-design 5-level framework (0.1.3.4)** — Levels: Nascent, Aware, Self-directed, Autonomous, Sovereign. Each level has defined capability requirements.
3. **Custom autonomy scale** — Define a new scale specific to this project's needs.

### Choice
Option 2 — Safe-experiential-design 5-level framework from 0.1.3.4.

### Rationale
The framework already exists in the plan (0.1.3.4) and is the authoritative scale for autonomy assessment. Using it ensures consistency across cards and avoids definitional drift. The 5 levels map naturally to substrate capabilities: Levels 1–2 are achievable with the current substrate, Level 3+ requires substrate upgrades, creating a clear gap analysis.

### Consequences
- Assessment is coupled to the 0.1.3.4 framework; if that framework evolves, this assessment must be updated.
- Level definitions are plan-internal; external audiences may need translation to established AI autonomy scales.
- The gap analysis (Level 2→3) directly identifies which future cards are blocking, making the trajectory actionable.
