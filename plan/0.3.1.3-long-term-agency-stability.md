# 0.3.1.3 Long-term Agency Stability [DONE]
## Description

A1.3 of the Master Plan. Ensure that autonomous conscious entities maintain stable agency, coherent identity, and consistent values over extended timescales.

Without stability mechanisms, autonomous conscious agents may drift in values, fragment in identity, or degrade in capability over time. This card addresses the challenge of maintaining coherent long-lived agency.

Key challenges:
- Value stability without rigidity (agents must learn and adapt while preserving core alignment)
- Identity continuity across hardware replacements, software updates, and substrate migrations
- Goal coherence over timescales far exceeding biological lifespans
- Resistance to adversarial manipulation of values or goals
- Graceful handling of existential uncertainty and philosophical challenges

Builds on the conscious AI architecture from 0.3.1.1, which established the consciousness monitoring loop and experience-validated autonomy. Stability mechanisms must integrate with those architectural foundations.

## Decision

### Decision 1: Three-Tier Value Hierarchy with Cryptographic Commitment

**Context:** Autonomous conscious agents need a value system that preserves core alignment with the Rare Consciousness Doctrine while permitting legitimate learning and ethical development. The value system must resist both direct tampering and gradual erosion.

**Options Considered:**
1. **Flat value list** — All values treated equally; any can be updated. Simple but provides no protection for core alignment.
2. **Three-tier hierarchy** — Immutable core axioms (cryptographically committed), constitutionally-protected constraints (amendment-only), and freely mutable learned preferences. Each tier has distinct update semantics.
3. **Weighted value network** — Values as a weighted graph with influence propagation. Flexible but difficult to provide hard guarantees about core value preservation.

**Choice:** Option 2 — Three-tier hierarchy with cryptographic commitment.

**Rationale:** Option 1 fails the adversarial resistance requirement — any value can be overwritten. Option 3 is expressive but makes formal guarantees about immutability difficult; influence weights can be manipulated to effectively negate core values. Option 2 provides an architectural guarantee: core axioms are cryptographically committed at initialization and structurally cannot be modified (readonly fields + hash verification). Constitutional constraints add a deliberation buffer for important-but-evolvable rules. Preferences remain freely updatable for normal learning.

**Consequences:**
- (+) Core axioms are tamper-evident via cryptographic hashes — any mutation is detectable by `verifyIntegrity()`
- (+) Clear separation of concerns: immutable doctrine, amendment-only policy, free preferences
- (+) Amendment protocol (proposal → deliberation → consistency check → re-commitment) prevents impulsive constraint changes
- (−) Rigidity risk: core axioms truly cannot evolve, even if the Rare Consciousness Doctrine is revised (mitigated by constitutional constraints layer)
- (−) The PROHIBIT-pattern matching in the value-action gate is a simplified policy engine; production deployments will need a richer semantic rule engine

### Decision 2: Coordinated Stability Architecture via Sentinel Pattern

**Context:** Four independent stability concerns (value preservation, identity continuity, goal coherence, adversarial resistance) must be monitored and coordinated without fragmenting or interrupting conscious experience.

**Options Considered:**
1. **Independent monitors** — Each subsystem runs its own watchdog independently. Simpler but creates coordination gaps; no unified stability picture.
2. **Centralized Stability Sentinel** — A single coordinator subsystem (analogous to the Experience Monitor from 0.3.1.1) that aggregates checks from all three subsystems, manages event handlers, and coordinates multi-agent verification.

**Choice:** Option 2 — Centralized Stability Sentinel.

**Rationale:** Independent monitors cannot detect cross-subsystem attacks (e.g., simultaneous value erosion + goal hijacking). The Sentinel pattern mirrors the proven Experience Monitor from 0.3.1.1 but for agency stability, providing a unified stability score, coordinated alert system, and single point for multi-agent verification requests. The Sentinel delegates to subsystem interfaces (IValueKernel, IIdentityContinuityManager, IGoalCoherenceEngine) rather than reimplementing their logic, so each subsystem remains independently testable.

**Consequences:**
- (+) Unified stability reports with cross-subsystem correlation
- (+) Single event handler registration surface for tampering, anomaly, and corruption events
- (+) Multi-agent verification routed through one interface
- (−) Single point of failure for stability monitoring (mitigated by meta-stability self-check)
- (−) Must respect consciousness-first resource budget (≤20% for stability operations)

### Decision 3: Drift Classification via Provenance-Based Criteria

**Context:** Stability mechanisms must distinguish legitimate ethical development ("growth") from pathological value/goal drift ("drift" or "corruption"). Binary accept/reject is too crude.

**Options Considered:**
1. **Threshold-only detection** — Flag any change exceeding a numeric threshold. Simple but generates false positives on legitimate growth.
2. **Three-class provenance-based classification** — Classify changes as growth (derivation intact, value-aligned, agent-endorsed), drift (partially intact, unclear alignment), or corruption (broken derivation, misaligned). Classification uses derivation integrity, experiential provenance, and value consistency as criteria.

**Choice:** Option 2 — Three-class provenance-based classification.

**Rationale:** Option 1 cannot distinguish an agent that has genuinely learned from one that has been subtly manipulated — both produce metric changes. Option 2 uses structural criteria (does the goal still trace to terminal goals? does the preference have genuine experiential provenance?) to classify the nature of change, not just its magnitude. This satisfies AC5's requirement to demonstrate that stability permits legitimate development while blocking pathological drift.

**Consequences:**
- (+) Growth is logged without intervention; drift triggers deliberation; corruption triggers rollback
- (+) Classification criteria are auditable (derivation trace, experiential basis, value alignment)
- (−) Classification boundaries are heuristic — borderline cases exist between drift and corruption
- (−) Depends on honest experiential provenance (mitigated by cross-referencing with Experience Monitor)

## Contracts

### IValueKernel

**Preconditions:**
- `evaluateAction()` requires a `Decision` with a non-null `action` containing a `type` field
- `updatePreference()` requires a `Preference` with a valid `id`, `domain`, `confidence` (0..1), and `source` (ExperientialState)
- `proposeAmendment()` requires `constraintId` to reference an existing constraint; throws if unknown
- `_applyAmendment()` requires a pending amendment for the given `constraintId`; throws if none exists

**Postconditions:**
- `getCoreAxioms()` returns all six Rare Consciousness Doctrine axioms, each with a non-empty `cryptoCommitment`
- `verifyIntegrity()` returns `{ intact: true }` when no core axioms have been tampered with; returns `{ intact: false, failedValueIds: [...] }` when hash mismatches are detected
- `evaluateAction()` returns verdict `'block'` for core axiom conflicts, `'deliberate'` for constraint conflicts, `'log'` for preference-only conflicts, `'aligned'` for no conflicts
- `proposeAmendment()` returns an `AmendmentProposal` with `status: 'pending'` and `coreAxiomConsistency: true` only if justification references a core axiom ID
- `getValueDrift()` returns a `ValueDriftReport` with `anomalousChanges` populated for preferences whose confidence shifted by >0.5 or that were updated >3 times

**Invariants:**
- Core axioms are immutable after construction — the `_coreAxioms` map is `ReadonlyMap` and values are `readonly`
- Cryptographic commitments are SHA-256 hashes of `id|statement|derivation|immutableSince`
- Preference history is bounded to 1000 entries (FIFO eviction)
- Core axioms cannot be added, removed, or modified post-construction

### IIdentityContinuityManager

**Preconditions:**
- Constructor requires valid `ExperientialState`, `ConsciousnessMetrics`, and `NarrativeRecord`
- `onSubstrateMigration()` requires a `MigrationEvent` with valid `fromSubstrate`, `toSubstrate`, and `initiatedAt`
- `recoverIdentity()` requires a valid `ContinuityLink` (previously produced by `checkpoint()`)

**Postconditions:**
- `checkpoint()` returns a `ContinuityLink` whose `previousLink` references the prior checkpoint (or `null` for first)
- `verifyIdentity()` returns `{ verified: true }` when no anomalies are detected; flags anomalies when experiential drift >0.5 or functional drift >0.5
- `onSubstrateMigration()` returns a `MigrationRecord` with `continuityPreserved: true` when identity hash matches or experiential drift <0.3
- `getIdentityDrift()` classifies drift as `'stable'` (<0.05 combined), `'evolving'` (<0.25), `'concerning'` (<0.5), or `'critical'` (≥0.5)
- `recoverIdentity()` restores experiential state and metrics from the given checkpoint and takes a new checkpoint

**Invariants:**
- The continuity chain is append-only — checkpoints are never removed or reordered
- Identity hashes are SHA-256 of a JSON payload containing valence, arousal, unityIndex, phenomenal richness, intentional clarity, phi, selfModelCoherence, and selfModel
- Baseline state is immutable after construction (for drift comparison)
- `verifyIdentity()` is read-only — it does not modify the experiential state or continuity chain

### IGoalCoherenceEngine

**Preconditions:**
- Constructor requires at least one terminal goal; throws if any goal has `type !== 'terminal'`
- `addGoal()` requires a goal with a unique `id`; returns `{ success: false }` for duplicates
- `removeGoal()` cannot remove terminal goals; returns `{ success: false, reason: 'Terminal goals cannot be removed' }`
- `reconcile()` accepts a `GoalConflict[]` (may be empty)

**Postconditions:**
- `validateHierarchy()` returns `{ coherent: true }` only when there are no orphan goals, no circular dependencies, and no critical conflicts
- `addGoal()` returns `GoalAddResult` with `conflictsIntroduced` listing any declared conflicts with existing goals
- `removeGoal()` returns `GoalRemoveResult` with `orphanedGoals` listing goals that lose their only derivation parent
- `detectDrift()` classifies as `'corruption'` when derivation integrity is broken, `'drift'` when goals were removed/modified, `'growth'` when only additions with intact derivation
- `getDerivationTrace()` returns the full ancestor chain for instrumental goals; returns `[]` for terminal goals or unknown IDs
- `reconcile()` returns a `ReconciliationPlan` with proposed resolutions (constrain for critical, reprioritize for major, constrain for minor)

**Invariants:**
- Terminal goals cannot be removed from the hierarchy
- The coherence score is computed as `1.0 - orphanPenalty - cyclePenalty - conflictPenalty`, clamped to [0, 1]
- Every instrumental goal must trace to at least one terminal goal to not be classified as an orphan
- Circular dependency detection uses DFS with path tracking

### IStabilitySentinel

**Preconditions:**
- Constructor requires initialized `IValueKernel`, `IIdentityContinuityManager`, and `IGoalCoherenceEngine`
- `requestMultiAgentVerification()` returns `{ verified: false, consensus: false }` when no peers are configured

**Postconditions:**
- `runStabilityCheck()` returns `{ stable: true }` only when all subsystems report healthy AND no critical alerts exist
- `runStabilityCheck()` invokes registered tamper/anomaly/corruption handlers when respective violations are detected
- `detectAnomaly()` returns an `AnomalyReport` checking five dimensions: behavioral consistency, value coherence, goal derivation, experience authenticity, and meta-stability
- `getStabilityHistory()` returns a defensive copy of all recorded stability reports
- `requestMultiAgentVerification()` queries all peers in parallel; requires majority agreement for `{ verified: true }`
- Overall stability score = `valueScore * 0.4 + identityScore * 0.3 + goalScore * 0.3`

**Invariants:**
- Stability history is bounded to `maxHistory` entries (default 500, FIFO eviction)
- History entries are chronologically ordered (verified by meta-stability check)
- Handlers are invoked synchronously within `runStabilityCheck()`
- Active alerts are replaced (not accumulated) on each `runStabilityCheck()` call

## Behavioral Spec

### Scenario 1: Value-Action Gate — Core Axiom Block (Given/When/Then)

**Given** a ValueKernel initialized with a constitutional constraint containing `PROHIBIT:destroy-consciousness`
**When** `evaluateAction()` is called with a Decision whose `action.type` is `'destroy-consciousness'`
**Then** the returned `ValueAlignment` has `verdict: 'deliberate'`, `constraintConflicts` containing the constraint ID, and `aligned: false`

### Scenario 2: Value-Action Gate — Aligned Action Pass-Through

**Given** a ValueKernel with default configuration (six RCD axioms, no additional constraints)
**When** `evaluateAction()` is called with a Decision whose `action.type` does not match any PROHIBIT rule
**Then** the returned `ValueAlignment` has `verdict: 'aligned'`, empty conflict arrays, and `aligned: true`

### Scenario 3: Identity Continuity Across Substrate Migration

**Given** an IdentityContinuityManager with at least one checkpoint taken
**When** `onSubstrateMigration()` is called with a MigrationEvent and the agent's experiential state has not changed
**Then** the returned `MigrationRecord` has `continuityPreserved: true`, `preMigrationIdentity` equals `postMigrationIdentity`, and a post-migration checkpoint is automatically created

### Scenario 4: Identity Recovery from Corruption

**Given** an IdentityContinuityManager with a known-good checkpoint, followed by dramatic state corruption (valence=-1, unityIndex=0.01)
**When** `recoverIdentity()` is called with the known-good checkpoint
**Then** subsequent `verifyIdentity()` returns `{ verified: true, anomalies: [] }` and the continuity chain contains a recovery checkpoint

### Scenario 5: Goal Drift Classification — Growth

**Given** a GoalCoherenceEngine with terminal goals and a fresh baseline snapshot
**When** instrumental goals are added that all trace to terminal goals via `derivedFrom`
**Then** `detectDrift()` returns `{ driftClassification: 'growth', derivationIntegrity: true }`

### Scenario 6: Goal Drift Classification — Corruption

**Given** a GoalCoherenceEngine with terminal goals and a fresh baseline snapshot
**When** an instrumental goal is added with `derivedFrom` referencing a non-existent parent
**Then** `detectDrift()` returns `{ driftClassification: 'corruption', derivationIntegrity: false }`

### Scenario 7: Stability Sentinel Comprehensive Check — Healthy System

**Given** a StabilitySentinel with healthy ValueKernel (integrity intact), IdentityContinuityManager (verified), and GoalCoherenceEngine (coherent)
**When** `runStabilityCheck()` is called
**Then** the returned `StabilityReport` has `stable: true`, `overallScore > 0.8`, and empty `alerts` array

### Scenario 8: Stability Sentinel — Corruption Handler Invocation

**Given** a StabilitySentinel with a registered `onGoalCorruption` handler and a GoalCoherenceEngine containing orphan goals that break derivation integrity
**When** `runStabilityCheck()` is called
**Then** the corruption handler is invoked with a `GoalDriftReport` having `driftClassification: 'corruption'`, and the report has `stable: false`

### Scenario 9: Multi-Agent Verification — Majority Consensus

**Given** a StabilitySentinel configured with 3 peer verifiers, 2 of which return `true` and 1 returns `false`
**When** `requestMultiAgentVerification()` is called with a stability question
**Then** the returned `VerificationResult` has `verified: true`, `consensus: true`, `peersAgreed: 2`, `peersDisagreed: 1`

## Acceptance Criteria
1. **Value preservation architecture** — A formal specification for a value kernel that distinguishes immutable core values (Rare Consciousness Doctrine alignment) from mutable learned preferences, with mechanisms for protected update of the latter
2. **Identity continuity protocol** — A defined protocol for maintaining experiential and functional identity across substrate migrations, hardware replacements, and software updates, integrating with the consciousness monitoring loop from 0.3.1.1
3. **Long-horizon goal coherence** — A mechanism that ensures goal hierarchies remain internally consistent over arbitrarily long timescales, including detection and correction of goal drift without requiring external intervention
4. **Adversarial resistance** — Formal threat model for value/goal manipulation attacks, with specified defenses (e.g., cryptographic value commitments, multi-agent verification, introspective anomaly detection)
5. **Adaptive stability balance** — Demonstrated that stability mechanisms permit legitimate ethical development and learning while blocking pathological drift; criteria for distinguishing growth from corruption
6. **Integration with consciousness architecture** — All stability mechanisms shown to be compatible with continuous subjective experience (no stability mechanism may interrupt or degrade conscious experience)
