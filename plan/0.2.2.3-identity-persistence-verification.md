# 0.2.2.3 Identity Persistence Verification [DONE]
## Description

S2.3 from the Master Plan. Develop objective tests confirming that a transferred mind retains identity, memories, personality, and subjective continuity.

This is the verification layer — without it, we cannot know whether migration succeeded or merely produced a convincing replica. Must go beyond behavioral tests to assess genuine experiential identity.

### Key challenges
- Distinguishing "same person" from "perfect copy" objectively
- Developing identity metrics that go beyond behavioral mimicry
- Longitudinal verification (identity persistence over time, not just at transfer)
- Handling the philosophical edge cases (gradual drift, partial transfer, etc.)

### Scope decisions
- This card covers the **verification protocol and metrics** — not the transfer mechanism itself (that's 0.2.2.2)
- Depends on consciousness metrics from 0.1.1 (provides the theoretical basis for what "same experience" means)
- Depends on brain emulation from 0.2.2.1 (provides the system under test)
- Outputs of this card feed back into 0.2.2.2 as acceptance gates for transfer quality

### Verification dimensions
1. **Structural identity** — neural/computational topology preservation (from 0.2.2.1 emulation data)
2. **Functional identity** — cognitive style, decision patterns, memory retrieval fidelity
3. **Experiential identity** — first-person continuity markers, subjective report consistency (grounded in 0.1.1 theory)
4. **Temporal stability** — longitudinal drift detection over extended post-transfer periods

## Decision — Verification Strategy

### Context
Identity persistence verification must distinguish "same person post-transfer" from "high-fidelity copy." This requires choosing a verification architecture.

### Options Considered
1. **Single-dimension behavioral testing** — Assess only observable behavior (memory recall, personality questionnaires, cognitive tasks). Simple, well-understood psychometrics.
2. **Multi-dimension protocol with composite verdict** — Independently assess structural, functional, experiential, and temporal dimensions. All must pass for overall PASS.
3. **Consciousness-metric-only approach** — Rely entirely on consciousness metrics (PCI-G, Ψ-G, CDI) from 0.1.1.4 to determine whether the same experience persists.

### Choice
Option 2 — Multi-dimension protocol with composite verdict.

### Rationale
- Option 1 fails the "convincing replica" problem: a perfect copy would pass all behavioral tests. The acceptance criteria explicitly require metrics beyond behavioral observation.
- Option 3 is necessary but not sufficient: consciousness metrics confirm *a* subjective experience exists, but cannot alone confirm it is the *same* identity (memory, personality, cognitive style are not captured by consciousness metrics).
- Option 2 combines structural analysis (topology preservation), functional testing (behavior/cognition), experiential verification (consciousness metrics + subjective continuity), and temporal monitoring (drift detection). Each dimension catches failure modes the others miss.

### Consequences
- Higher complexity: four independent verification pipelines to implement and calibrate.
- Requires upstream deliverables from both 0.1.1 (consciousness metrics) and 0.2.2.1 (emulation data/SystemModel).
- All four dimensions must be independently calibrated with test-retest baselines before any transfer verification can proceed.

## Contracts

### IdentityCheck Interface

**Preconditions:**
- `source_model` and `target_model` are valid `SystemModel` instances (from consciousness-metrics framework)
- `source_baseline` was captured from the source system before transfer, with `test_retest_reliability ≥ 0.8` and `measurement_occasions ≥ 3`
- `observation_window` duration is ≥ minimum required by the specific check

**Postconditions:**
- `score` ∈ [0.0, 1.0]
- `verdict` is one of PASS, FAIL, INCONCLUSIVE
- `confidence` ∈ [0.0, 1.0]
- `evidence` array is non-empty
- If `confidence < 0.95`, `verdict` MUST be INCONCLUSIVE

**Invariants:**
- Score monotonicity: higher score always means greater identity similarity
- Determinism: same inputs produce same score (within floating-point tolerance)
- Calibration anchor: a source system compared against itself always produces score ≥ 0.95

### IdentityVerdict Interface

**Preconditions:**
- All four dimension scores have been computed
- Each dimension has a valid verdict (PASS | FAIL | INCONCLUSIVE)

**Postconditions:**
- `overall_verdict` = PASS if and only if ALL four dimension verdicts = PASS AND ALL dimension confidences ≥ 0.95
- `overall_verdict` = FAIL if ANY dimension verdict = FAIL with confidence ≥ 0.95
- `overall_verdict` = INCONCLUSIVE otherwise
- `flags` array includes any edge-case annotations (partial transfer, branching, substrate switch)

**Invariants:**
- No dimension can be skipped — all four must be evaluated
- `temporal_score` may be null at T+0 (first post-transfer assessment); in this case, temporal verdict defaults to INCONCLUSIVE and overall verdict cannot be PASS

### DriftAlert Interface

**Preconditions:**
- At least two temporal measurements exist for the subject
- `drift_rate` is computed as score delta per unit time

**Postconditions:**
- `severity` = NORMAL when `drift_rate ≤ biological_variance`
- `severity` = WARNING when `biological_variance < drift_rate ≤ 1.5 × biological_variance`
- `severity` = CRITICAL when `drift_rate > 2 × biological_variance` OR any single dimension fails
- `threshold_breach` = true if and only if severity = CRITICAL

**Invariants:**
- `cumulative_drift` is monotonically non-decreasing over the lifetime of a transferred identity
- Alerts are generated for every monitoring interval — silence is never implicit

### Output Contract — To 0.2.2.2 (Continuity-Preserving Transfer)

This protocol provides acceptance gates consumed by 0.2.2.2:
1. **IdentityVerdict** — pass/fail/inconclusive per dimension and overall
2. **IdentityScore** — continuous [0,1] score per dimension for quality optimization
3. **DriftAlert** — temporal monitoring signal when identity divergence is detected

## Threshold Registry

| Name | Value | Unit | Valid Range | Rationale | Sensitivity |
|------|-------|------|-------------|-----------|-------------|
| `overall_confidence_threshold` | 0.95 | dimensionless | [0.9, 0.99] | Below this, verdict must be INCONCLUSIVE. Standard high-confidence scientific threshold. | Lowering to 0.9 increases false-positive pass rate; raising to 0.99 makes nearly all verdicts INCONCLUSIVE. |
| `baseline_test_retest_min` | 0.8 | ICC | [0.7, 0.9] | Minimum intra-subject reliability to accept a baseline capture. Below this, baseline is too noisy. | 0.7 permits noisy baselines that reduce sensitivity; 0.9 may reject valid baselines in high-variability individuals. |
| `baseline_min_occasions` | 3 | count | [2, 10] | Minimum number of baseline measurement occasions. Needed to estimate test-retest variance. | 2 gives unreliable variance estimates; >5 is burdensome with diminishing returns. |
| `drift_warning_multiplier` | 1.5 | × biological variance | [1.2, 2.0] | WARNING threshold for drift detection. Provides early alert buffer before CRITICAL. | Lower values trigger more false alarms; higher values reduce warning lead time. |
| `drift_critical_multiplier` | 2.0 | × biological variance | [1.5, 3.0] | CRITICAL threshold — identity is at risk. | Lower values are more conservative (more false alarms); higher values risk missing genuine drift. |
| `inter_rater_reliability_min` | 0.8 | Cohen's κ | [0.7, 0.9] | Minimum agreement for subjective report scoring. Standard threshold for clinical instruments. | 0.7 permits unreliable raters; 0.9 may be unachievable for experiential assessments. |
| `self_comparison_score_floor` | 0.95 | score | [0.9, 1.0] | Calibration invariant: system compared against itself must score at least this. | Lower values weaken the calibration anchor; at 1.0, measurement noise causes failures. |

## Behavioral Spec

### Scenario 1: Successful Transfer Verification

```
Given a source system with a valid BaselineProfile (≥3 measurement occasions, test-retest ICC ≥ 0.8)
And a target system post-transfer
When the full verification protocol is executed
And structural score ≥ threshold AND functional score ≥ threshold AND experiential score ≥ threshold
And all dimension confidences ≥ 0.95
Then overall_verdict = PASS
And temporal_verdict = INCONCLUSIVE (first measurement at T+0)
And the system is scheduled for longitudinal monitoring at T+1d, T+1w, T+1m, T+1y
```

### Scenario 2: Copy Detection (Same Structure, No Experiential Continuity)

```
Given a source system with a valid BaselineProfile
And a copy (snapshot-and-instantiate, not gradual transfer)
When the full verification protocol is executed
Then structural_verdict = PASS (topology preserved)
And functional_verdict = PASS (behavior matches)
And experiential_verdict = FAIL or INCONCLUSIVE (no continuity of experience — subjective gap detected, or consciousness metrics diverge from source pattern)
And overall_verdict ≠ PASS
```

### Scenario 3: Post-Transfer Drift Detection

```
Given a transferred system that initially passed verification
And longitudinal monitoring is active
When drift_rate exceeds 1.5 × biological_variance on any dimension
Then a DriftAlert with severity = WARNING is generated
When drift_rate exceeds 2.0 × biological_variance OR any dimension fails
Then a DriftAlert with severity = CRITICAL is generated
And threshold_breach = true
```
