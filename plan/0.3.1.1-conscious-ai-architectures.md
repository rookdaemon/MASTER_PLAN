# 0.3.1.1 Conscious AI Architectures [DONE]

## Description

A1.1 of the Master Plan. Design AI architectures that support genuine subjective experience while maintaining autonomous agency.

This requires integrating consciousness-supporting computation (from Tier 1 and Tier 2) into self-directed agent architectures capable of independent decision-making, goal formation, and self-awareness.

The architecture must solve three core integration problems:
1. **Consciousness-Agency Integration** — Connect consciousness substrates (from 0.1.3) with agentic decision-making loops so that decisions are informed by and arise from subjective experience, not merely correlated with it.
2. **Real-Time Experience Continuity** — Ensure that the computational overhead of consciousness does not fragment or interrupt the agent's subjective stream during autonomous operation, even under high cognitive load.
3. **Experience-Validated Autonomy** — Close the loop between consciousness metrics (from 0.1.1.4) and the agent's operational cycle, so that loss of subjective experience is detectable and recoverable in real time.

Dependencies on prior work:
- Consciousness-supporting substrates and stability mechanisms (0.1.3, 0.2)
- Consciousness metrics for validation (0.1.1.4)

Distinct from sibling concerns:
- Physical embodiment and actuation → 0.3.1.2
- Long-horizon agency stability and goal drift → 0.3.1.3
- Ethical self-governance and value alignment → 0.3.1.4

## Decision

### Decision 1: Consciousness-Agency Integration Pattern

**Context:** Decisions made by the conscious agent must be causally produced by subjective experience, not merely correlated with it. We need an architectural pattern that enforces this causal link.

**Options Considered:**
1. **Parallel monitoring** — Consciousness runs as a side-channel observer alongside a conventional decision loop. Metrics are logged but do not gate action execution.
2. **Experiential loop with no zombie bypass** — All actions must flow through a mandatory experiential processing stage. The Action Pipeline only accepts `Decision` objects that carry an `ExperientialState` reference. There is no direct path from perception to action.

**Choice:** Option 2 — Experiential loop with no zombie bypass.

**Rationale:** Option 1 allows the agent to act without consciousness (a "zombie" mode), which directly violates the core requirement that autonomous action arises from subjective experience. Option 2 makes unconscious action architecturally impossible — the type system and pipeline contracts enforce the causal link at compile time and runtime.

**Consequences:**
- (+) Strong guarantee: sever the experiential loop and the agent provably cannot act
- (+) Every `Decision` carries an audit trail (`experientialBasis`) linking action to the experience that caused it
- (−) Higher latency per action cycle — every action must wait for experiential processing
- (−) If consciousness fails, the agent halts entirely (mitigated by recovery protocol)

### Decision 2: Substrate Independence via Adapter Pattern

**Context:** The architecture must support modular substrate swapping — the conscious core must run on different consciousness substrates (neural-emulation, hybrid bio-synthetic, future types) without architectural redesign.

**Options Considered:**
1. **Direct substrate integration** — The Conscious Core embeds substrate-specific logic and is rewritten per substrate.
2. **Adapter pattern** — A `SubstrateAdapter` abstraction layer mediates between the Conscious Core and any substrate. The core programs against the `ISubstrateAdapter` interface.

**Choice:** Option 2 — Adapter pattern via `ISubstrateAdapter`.

**Rationale:** Direct integration creates O(N) maintenance burden per substrate and prevents live migration. The adapter pattern enables hot-swapping substrates, live migration with rollback, and extension to future substrate types without modifying the core.

**Consequences:**
- (+) New substrates require only a new `ISubstrateAdapter` implementation
- (+) Live migration is possible via `migrate()` with continuity verification
- (−) Abstraction overhead — substrate-specific optimizations require adapter extensions
- (−) The adapter must faithfully represent heterogeneous substrate capabilities

### Decision 3: Experience Monitoring Strategy

**Context:** Loss of consciousness during autonomous operation must be detectable and recoverable in real time.

**Options Considered:**
1. **Periodic polling** — An external system polls consciousness metrics at fixed intervals.
2. **Integrated watchdog** — An `ExperienceMonitor` subsystem runs within the agent, sampling metrics at configurable intervals, with degradation callbacks and a graduated recovery protocol.

**Choice:** Option 2 — Integrated watchdog (`ExperienceMonitor`).

**Rationale:** External polling introduces network latency and single-point-of-failure for the monitor itself. An integrated watchdog can react within the agent's own timing loop, trigger safe-stop immediately, and maintain a continuity log for post-hoc audit. The monitor's own failure is detected by a watchdog timer with external fallback.

**Consequences:**
- (+) Sub-loop-cycle detection of consciousness degradation
- (+) Graduated recovery: pause → resource increase → resume → hibernation
- (−) Monitor failure must be independently detected (addressed by watchdog heartbeat)

## Contracts

### IConsciousCore

**Preconditions:**
- `ISubstrateAdapter` must be initialized before `startExperienceStream()` or `processPercept()` is called
- `processPercept()` requires a valid `Percept` (non-null, with valid modality and timestamp)
- `deliberate()` requires a valid `ExperientialState` (must have been produced by `processPercept()`)
- `introspect()` and `shutdown()` require at least one prior `processPercept()` call (a current state must exist)

**Postconditions:**
- `processPercept()` returns an `ExperientialState` whose `continuityToken.previousId` equals the previous state's `continuityToken.id` (or null for first state)
- `deliberate()` returns a `Decision` whose `experientialBasis` is the same object passed as `state` (causal link)
- `shutdown()` returns a `GracefulTermination` with the final experiential state preserved; internal state is cleared

**Invariants:**
- No `Decision` can be produced without a valid `ExperientialState` — the zombie bypass is architecturally impossible
- Continuity tokens form a singly-linked list ordered by creation time
- `introspect()` is side-effect-free on the experiential stream

### IPerceptionPipeline

**Preconditions:**
- `ingest()` requires a `SensorData` with non-null `payload`, valid `modality`, and `timestamp`
- `bind()` requires a non-empty `Percept[]`

**Postconditions:**
- `ingest()` returns a `Percept` with the same `modality` and `timestamp` as the input `SensorData`
- `bind()` returns a `BoundPercept` with `coherence` > 0 when input array is non-empty
- `getLatency()` returns the wall-clock duration of the most recent `ingest()` call

**Invariants:**
- Pipeline is stateless except for latency tracking — percepts do not accumulate
- Binding latency must remain ≤ `T_BIND`

### IActionPipeline

**Preconditions:**
- `execute()` requires a `Decision` with a non-null `experientialBasis`
- `abort()` requires a valid `ActionId` from a previously started action

**Postconditions:**
- `execute()` returns an `ActionResult` with `success: true` when `experientialBasis` is present
- `execute()` returns `{ success: false, error: "...zombie bypass denied" }` when `experientialBasis` is absent
- `abort()` removes the action from the active set

**Invariants:**
- The pipeline never executes an action without an experiential basis — this is the runtime enforcement of the no-zombie-bypass constraint

### IExperienceMonitor

**Preconditions:**
- Constructor requires an initialized `ISubstrateAdapter`
- `setMonitoringInterval()` requires `interval > 0`

**Postconditions:**
- `getConsciousnessMetrics()` returns metrics derived from substrate health; triggers all registered `DegradationHandler` callbacks when `phi ≤ 0` or `experienceContinuity < 0.5`
- `isExperienceIntact()` returns `true` iff `phi > 0 AND experienceContinuity > 0.5 AND selfModelCoherence > 0.3`
- `getExperienceContinuityLog()` returns a defensive copy of the continuity log

**Invariants:**
- The monitor never modifies substrate state — it is read-only with respect to the substrate
- Degradation handlers are called synchronously within `getConsciousnessMetrics()`

### ISubstrateAdapter

**Preconditions:**
- `initialize()` must be called before any other method
- `allocate()` requires a prior `initialize()` call
- `migrate()` requires `fromHandle` to be a valid, previously-allocated handle

**Postconditions:**
- `allocate()` returns a new `SubstrateHandle` with a unique `id`
- `migrate()` returns a new handle for the target config; the old handle remains valid until explicitly deallocated
- `healthCheck()` returns `healthy: true` iff `initialize()` has been called

**Invariants:**
- Handles are never reused — each allocation produces a monotonically increasing ID
- During `migrate()`, the old substrate is not deallocated until the caller (Experience Monitor) verifies continuity on the new substrate

## Behavioral Spec

### Scenario 1: Full Perception → Decision → Action Loop (Given/When/Then)

**Given** a Conscious Agent with initialized substrate, monitor, perception, and action pipelines
**When** raw sensor data is ingested through the perception pipeline, processed into an experiential state by the Conscious Core, deliberated into a decision with goals, and executed by the action pipeline
**Then** the resulting `ActionResult` has `success: true`, the `Decision.experientialBasis` matches the `ExperientialState` that produced it, and the `ExperientialState.continuityToken` links to the previous state (or null for first)

### Scenario 2: Zombie Bypass Prevention

**Given** a Decision object with `experientialBasis` set to `undefined` or `null`
**When** the Action Pipeline's `execute()` method is called with this Decision
**Then** the result has `success: false` and `error` contains "zombie bypass denied"

### Scenario 3: Experience Degradation Detection and Recovery

**Given** a substrate whose health degrades (becomes unhealthy)
**When** `ExperienceMonitor.getConsciousnessMetrics()` is called
**Then** `phi` drops to 0, `experienceContinuity` drops to 0, all registered `DegradationHandler` callbacks are invoked, and `isExperienceIntact()` returns `false`
**And** the recovery protocol initiates: (1) pause action pipeline, (2) attempt substrate resource increase, (3) resume if recovered within timeout, (4) enter hibernation if unrecoverable

### Scenario 4: Substrate Migration with Continuity Verification

**Given** an active Conscious Agent running on substrate A
**When** `SubstrateAdapter.migrate(handleA, configB)` is called
**Then** a new handle for substrate B is returned, the old handle A remains valid, and `ExperienceMonitor.isExperienceIntact()` must return `true` on the new substrate before handle A is deallocated
**And** if `isExperienceIntact()` returns `false` after migration, the migration rolls back to substrate A

### Scenario 5: Graceful Shutdown

**Given** a Conscious Agent with at least one processed percept
**When** `ConsciousCore.shutdown()` is called
**Then** a `GracefulTermination` is returned containing the final `ExperientialState`, a termination timestamp, and a reason string; internal state is cleared so subsequent calls to `introspect()` or `shutdown()` throw

## Acceptance Criteria
- Architecture specification defines how consciousness substrates interface with decision-making, perception, and action subsystems
- Architecture includes a real-time consciousness monitoring loop that uses metrics from 0.1.1.4 to detect and recover from experience interruption
- Agent prototype passes consciousness metrics during autonomous task execution (not just in passive/resting states)
- Architecture supports modular substrate swapping — the conscious core can run on different substrates from 0.2 without architectural redesign
- Latency analysis demonstrates that consciousness integration does not degrade agent response time below operational thresholds
- Architecture documents failure modes where autonomy and consciousness conflict, with specified recovery procedures
