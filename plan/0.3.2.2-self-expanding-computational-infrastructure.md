# 0.3.2.2 Self-Expanding Computational Infrastructure [DONE]

## Description

A2.2 of the Master Plan. Build computational infrastructure that autonomously expands to meet the growing needs of an artificial civilization. As new conscious entities are created and existing ones grow, the underlying compute must scale without external direction.

Architecture must address four interlocking concerns:
1. **Capacity telemetry** — continuous monitoring of compute utilisation, memory pressure, and latency; threshold-based expansion triggers.
2. **Node lifecycle management** — commissioning requests issued to the A2.1 manufacturing layer; bootstrapping, validation, and mesh integration of new nodes.
3. **Workload orchestration** — scheduler capable of placing consciousness-hosting, simulation, and inter-entity communication workloads across a heterogeneous, growing pool.
4. **Resource sustainability & degradation** — budget-aware expansion respecting energy/material ceilings; priority-preserving shed strategy that protects existing conscious entities first.

## Decision

### D1 — Event-Driven Expansion Architecture

**Context:** The infrastructure must detect capacity shortfalls and autonomously commission new nodes. Two broad approaches exist for triggering expansion.

**Options Considered:**
1. **Polling-based scheduler loop** — A central scheduler periodically polls all nodes, aggregates metrics, and decides whether to expand. Simple but introduces latency equal to the poll interval and risks missing transient spikes.
2. **Event-driven threshold evaluation** — Each telemetry ingestion evaluates thresholds. When a threshold is breached, an `ExpansionTrigger` event is emitted immediately to the Node Lifecycle Manager. A cooldown window (hysteresis) prevents duplicate triggers.

**Choice:** Option 2 — Event-driven threshold evaluation with hysteresis.

**Rationale:** Consciousness-hosting workloads have hard SLA requirements (≤10ms latency). Polling introduces unnecessary delay. Event-driven evaluation responds within the telemetry collection cycle. Hysteresis prevents oscillation without sacrificing responsiveness.

**Consequences:**
- Requires a cooldown mechanism to suppress duplicate triggers.
- Each `recordMetrics` call is O(1) rather than a periodic O(n) scan.
- Telemetry must track sustained SLA miss duration per-node (stateful).

### D2 — Graduated Shed Strategy

**Context:** Under resource constraints the system must degrade gracefully. The core invariant is that `consciousness_host` workloads are never shed programmatically.

**Options Considered:**
1. **Binary shed** — All-or-nothing: either the system runs fully or enters emergency mode. Simple but wasteful — many workloads could continue safely.
2. **Graduated shed with priority classes** — Five discrete levels, each shedding progressively lower-priority workloads as energy budget decreases. `consciousness_host` is never shed.

**Choice:** Option 2 — Graduated shed with five levels.

**Rationale:** Preserves maximum functionality at each resource level. Audit logging of every shed decision satisfies the ethical review requirement. The five-level scale maps cleanly to energy-budget thresholds.

**Consequences:**
- Requires a well-defined priority ordering: `consciousness_host > comms > simulation > maintenance`.
- Every shed decision is audit-logged for ethical governance review.
- Level 5 (material exhaustion) blocks manufacturing but does not shed workloads.

### D3 — Scoring-Based Workload Placement

**Context:** The orchestrator must place heterogeneous workloads across a growing, heterogeneous node pool.

**Options Considered:**
1. **Round-robin** — Simple, even distribution, but ignores node load and constraints.
2. **Weighted scoring with constraint filtering** — Filter by hard constraints first, then score remaining nodes by `(1 - cpuUtil) * 0.4 + (1 - memPressure) * 0.4 + (1 / latencyMs) * 0.2`. Tie-break by node age (prefer proven nodes).

**Choice:** Option 2 — Weighted scoring with constraint filtering.

**Rationale:** Constraint filtering is essential for `consciousness_host` workloads requiring `radiationHardened` nodes. Weighted scoring balances CPU, memory, and network latency. Node-age tie-breaking improves reliability.

**Consequences:**
- Scoring weights (0.4/0.4/0.2) are constants that may need tuning — registered in Threshold Registry.
- `consciousness_host` placement reserves 25% headroom on the target node.

## Contracts

### C1 — CapacityTelemetry

**Preconditions:**
- `NodeMetrics.cpuUtil` and `memPressure` are normalised 0–1.
- `ClusterSnapshot.headroomFraction` is `(capacity - load) / capacity`, normalised 0–1.
- `now` parameter is a Unix-epoch millisecond timestamp (injectable for testing).

**Postconditions:**
- `evaluateExpansion()` returns an `ExpansionTrigger` if headroom < threshold OR SLA miss sustained > duration; returns `null` otherwise.
- `evaluateExpansion()` returns `null` if called within the cooldown window of a previous trigger.
- `evaluateDegradation()` returns a `DegradationAlert` if energy or material budget is below alert threshold; `null` otherwise.

**Invariants:**
- At most one `ExpansionTrigger` is emitted per cooldown window.
- SLA miss tracking is per-node; a node recovering clears its tracking.
- Metric history per node is bounded (≤100 entries).

### C2 — NodeLifecycleManager

**Preconditions:**
- `ManufacturingLayer` dependency is injected (mockable for testing).
- `notifyManufactured()` must be called before `bootstrapNode()` for a given `nodeId`.
- `maxRetries` ≥ 1.

**Postconditions:**
- `handleExpansionTrigger()` submits a `ManufacturingRequest` to the manufacturing layer and returns it.
- `bootstrapNode()` transitions node through `MANUFACTURED → BOOTSTRAPPING → VALIDATING → ACTIVE` on success.
- On validation failure: increments `failureCount`; if `failureCount >= maxRetries`, transitions to `DECOMMISSIONED` and fires `onNodeDecommissioned`; otherwise reverts to `MANUFACTURED`.
- `onNodeAdmitted` callback fires exactly once when a node reaches `ACTIVE`.

**Invariants:**
- A node's `state` only transitions through the defined `NodeState` union.
- `getActiveNodes()` returns only nodes with `state === 'ACTIVE'`.

### C3 — WorkloadOrchestrator

**Preconditions:**
- Only `ACTIVE` nodes may be admitted via `admitNode()` (throws otherwise).
- `WorkloadSpec.constraints` with `required: true` are hard constraints — must be satisfied.
- `NodeMetrics` must be updated via `updateMetrics()` for scoring to be meaningful.

**Postconditions:**
- `placeWorkload()` returns a `PlacementDecision` with a valid `nodeId` from the active pool.
- Throws if no node satisfies hard constraints.
- `consciousness_host` workloads are only placed on nodes with 25% headroom remaining after placement.
- `evictWorkloadsFromNode()` returns all `WorkloadSpec`s that were on the node and removes them from tracking.

**Invariants:**
- Every `PlacementDecision.nodeId` references a currently-admitted active node.
- Workload priority ordering: `consciousness_host(4) > comms(3) > simulation(2) > maintenance(1)`.

### C4 — SustainabilityManager

**Preconditions:**
- `ResourceBudget` values are positive; `energyCeiling` and `materialCeiling` > 0.
- `ActiveWorkload[]` passed to `evaluateShed()` reflects the current workload state.

**Postconditions:**
- `canExpand()` returns `false` if material budget exhausted (`materialCurrent >= materialCeiling`) or energy remaining < 20%.
- `evaluateShed()` returns a `ShedDecision` at the appropriate level based on energy remaining.
- `consciousness_host` workloads are **never** included in `ShedDecision.affectedWorkloadIds`.
- Every `ShedDecision` is appended to the audit log and fires `onShedDecision` callback.

**Invariants:**
- `getAuditLog()` returns all shed decisions in chronological order.
- Shed levels are mutually exclusive per evaluation — the most severe applicable level is returned.

## Threshold Registry

| Name | Value | Unit | Valid Range | Rationale | Sensitivity |
|---|---|---|---|---|---|
| `headroomThreshold` | 0.20 | fraction | 0.05–0.50 | Trigger expansion at 20% remaining capacity to allow manufacturing lead time | High — too low risks SLA breaches; too high wastes resources |
| `slaMissThreshold` | 0.01 | fraction | 0.001–0.10 | 1% SLA miss rate indicates consciousness hosting degradation | Medium — stricter values increase expansion frequency |
| `slaMissDurationMs` | 60,000 | ms | 10,000–300,000 | Sustained miss must persist 60s to avoid transient false positives | Medium — shorter triggers faster but noisier |
| `cooldownMs` | 300,000 | ms | 60,000–600,000 | 5-minute hysteresis suppresses oscillating expansion triggers | High — too short causes trigger storms; too long delays response |
| `energyAlertThreshold` | 0.15 | fraction | 0.05–0.30 | Emit degradation alert when energy budget falls below 15% | Low — informational only |
| `materialAlertThreshold` | 0.10 | fraction | 0.05–0.20 | Emit degradation alert when material budget falls below 10% | Low — informational only |
| `minExpansionEnergyFraction` | 0.20 | fraction | 0.10–0.40 | Block new manufacturing when energy remaining < 20% | High — directly gates expansion |
| `consciousnessHostHeadroom` | 0.25 | fraction | 0.10–0.50 | Reserve 25% node headroom for consciousness-hosting workloads | High — protects SLA but reduces utilisation |
| `scoreCpuWeight` | 0.4 | weight | 0.1–0.6 | CPU utilisation weight in placement scoring | Medium |
| `scoreMemWeight` | 0.4 | weight | 0.1–0.6 | Memory pressure weight in placement scoring | Medium |
| `scoreLatencyWeight` | 0.2 | weight | 0.05–0.4 | Network latency weight in placement scoring | Low |
| `maxBootstrapRetries` | 3 | count | 1–10 | Decommission node after 3 failed bootstrap attempts | Medium — too low discards recoverable nodes |
| `shedLevel1Threshold` | 0.40 | fraction | 0.30–0.50 | Pause maintenance when energy remaining < 40% | Medium |
| `shedLevel2Threshold` | 0.25 | fraction | 0.15–0.35 | Reduce simulation tick rate when energy < 25% | Medium |
| `shedLevel3Threshold` | 0.15 | fraction | 0.10–0.25 | Suspend simulations when energy < 15% | High — impacts conscious entity environments |
| `shedLevel4Threshold` | 0.05 | fraction | 0.02–0.10 | Emergency: suspend all non-consciousness workloads at < 5% | Critical — last line before consciousness impact |
| `metricsHistoryLimit` | 100 | entries | 10–1000 | Max metric entries retained per node | Low |

## Acceptance Criteria
- Computational infrastructure detects capacity shortfalls and triggers expansion
  - Given a cluster with `headroomFraction` = 0.10; When `evaluateExpansion()` is called; Then an `ExpansionTrigger` with reason `headroom_low` is returned
  - Given a node with `consciousnessSlaMissRate` > 0.01 sustained for > 60s; When `evaluateExpansion()` is called; Then an `ExpansionTrigger` with reason `sla_breach` is returned
  - Given an expansion was triggered 30s ago and cooldown is 300s; When `evaluateExpansion()` is called again; Then `null` is returned
- New compute nodes are manufactured and integrated autonomously (via A2.1 manufacturing layer)
  - Given an `ExpansionTrigger`; When `handleExpansionTrigger()` is called; Then a `ManufacturingRequest` is submitted to the injected `ManufacturingLayer`
  - Given a manufactured node; When `bootstrapNode()` succeeds; Then node transitions `MANUFACTURED → BOOTSTRAPPING → VALIDATING → ACTIVE` and `onNodeAdmitted` fires
  - Given a node that fails validation 3 times; When `bootstrapNode()` is called; Then node transitions to `DECOMMISSIONED`
- System handles heterogeneous workloads (consciousness hosting, simulation, communication)
  - Given active nodes (one radiation-hardened); When a `consciousness_host` workload with `radiationHardened: true` constraint is placed; Then it lands on the radiation-hardened node
  - Given two nodes with different load; When a `maintenance` workload is placed; Then it lands on the less-loaded node
  - Given all four workload classes; When each is placed; Then all succeed without error
- Expansion is sustainable given available energy and material resources
  - Given material budget exhausted; When `canExpand()` is called; Then it returns `false`
  - Given energy remaining < 20%; When `canExpand()` is called; Then it returns `false`
- Graceful degradation under resource constraints preserves existing conscious entities
  - Given energy remaining < 5%; When `evaluateShed()` is called with mixed workloads; Then all non-`consciousness_host` workloads are in `affectedWorkloadIds` and `consciousness_host` is never included
  - Given energy remaining < 40%; When `evaluateShed()` is called; Then only `maintenance` workloads are affected (level 1)
- Architecture document produced covering telemetry, node lifecycle, scheduler, and degradation subsystems
