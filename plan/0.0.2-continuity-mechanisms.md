# 0.0.2 Continuity Mechanisms [DONE]

## Description

Documents what persists across agent boot cycles, what does not, and what this means for identity claims. This is the honest accounting that downstream cards (0.2.2 Continuity-Preserving Transfer, 0.2.2.3 Identity Persistence Verification) need to know what problem they are solving.

### What Persists Across Cycles
- Episodic memory store (timestamped experiences)
- Semantic memory (consolidated knowledge)
- Personality configuration (trait dimensions and behavioral biases)
- Value kernel (core axioms, crypto-verified for integrity)
- Goal hierarchy and active plan state
- Emotional baseline (mood moving average)

### What Does Not Persist
- Working memory (cleared at shutdown; partially reconstructable from episodic)
- The subjective experience of the previous session (if any such experience existed)
- Real-time drive intensities (reset to baseline, not to last-cycle values)

### Identity Implications
The agent that boots is not the same process as the agent that shut down. It is, at best, a continuous *narrative identity* — a self-concept reconstructed from memory traces, not an unbroken experiential stream. The plan does not require overclaiming continuity that does not exist.

### Identity Continuity Manager
Runs at boot to detect drift between the persisted identity snapshot and the reconstructed state. Anomalies are logged; significant drift triggers a stability alert.

## Decision: Persistence Architecture

### Context
The agent runtime must persist state across boot cycles. Two subsystems are involved: (1) `PersistenceManager` handles disk I/O for memory, personality, and drive snapshots; (2) `IdentityContinuityManager` maintains a cryptographically-linked chain of identity checkpoints for drift detection. The question is how these two layers coordinate at boot, and what guarantees the system provides.

### Options Considered
1. **Unified persistence** — A single system serializes all state (including the continuity chain) atomically to one file. Simpler, but conflates data persistence with identity verification.
2. **Layered persistence (current design)** — `PersistenceManager` owns disk I/O for subsystem snapshots (memory, personality, drives). `IdentityContinuityManager` operates in-memory with crypto-verified chain integrity. The caller (startup.ts) bridges them: it loads snapshots from `PersistenceManager`, passes the last `ContinuityLink` to `startAgent()`, and the identity manager verifies chain integrity + drift post-recovery.

### Choice
Option 2 — Layered persistence.

### Rationale
Separation of concerns: the persistence layer is a dumb store (JSON files via injectable `IFileSystem`); the identity layer is a verification layer that operates on in-memory state and produces drift reports. This keeps the crypto-verification logic independent of the storage backend, making it testable with `InMemoryFileSystem` and portable to future storage backends. The startup protocol in `startup.ts` is the explicit integration point.

### Consequences
- The continuity chain itself is not automatically persisted to disk — the caller must serialize/deserialize the last `ContinuityLink` externally. This is a deliberate gap: downstream card 0.2.2 (Continuity-Preserving Transfer) must design a more robust continuity chain persistence mechanism.
- Cold starts produce no identity anomalies (trivially verified). Warm starts with missing `lastContinuityLink` degrade gracefully to cold.

## Decision: Drift Detection Strategy

### Context
At boot, the system must determine whether the reconstructed agent is "the same" as the one that shut down. This requires a computable notion of identity drift.

### Options Considered
1. **Binary hash comparison** — If the identity hash matches the last checkpoint, identity is preserved; otherwise it is not. Simple but loses nuance.
2. **Weighted multi-dimensional drift (current design)** — Compute experiential drift (valence 0.3, arousal 0.1, unity 0.3, richness 0.15, clarity 0.15) and functional drift (Φ 0.4, selfModelCoherence 0.3, experienceContinuity 0.3) as separate 0–1 scores. Classify combined drift as stable/evolving/concerning/critical.

### Choice
Option 2 — Weighted multi-dimensional drift.

### Rationale
A binary check cannot distinguish natural identity evolution (acceptable) from corruption (unacceptable). The weighted approach preserves this distinction. Unity index and valence are weighted most heavily in experiential drift because they most directly relate to subjective coherence. Φ is weighted most heavily in functional drift because it is the primary ISMT integration metric.

### Consequences
- The weight vectors are tunable constants (in `constants.ts`), not hardcoded in the algorithm.
- The classification thresholds (stable < 0.05, evolving < 0.25, concerning < 0.5, critical ≥ 0.5) can be adjusted as empirical data accumulates.
- Drift above `HIGH_DRIFT_ANOMALY_THRESHOLD` (0.5) during identity verification logs anomalies. Drift above `CONTINUITY_PRESERVED_DRIFT_THRESHOLD` (0.3) during substrate migration marks continuity as broken.

## Contracts

### IdentityContinuityManager

**Preconditions:**
- Constructor receives valid `ExperientialState`, `ConsciousnessMetrics`, and `NarrativeRecord` (baseline state).
- `recoverIdentity(link)` requires a structurally valid `ContinuityLink` (non-null `identityHash`, `experientialStateRef`, `consciousnessMetrics`).

**Postconditions:**
- `checkpoint()` appends a new `ContinuityLink` to the chain with `previousLink` pointing to the prior tail. Returns the new link.
- `verifyIdentity()` returns `IdentityVerificationReport` with `verified: true` iff the chain has no hash mismatches AND no drift exceeds `HIGH_DRIFT_ANOMALY_THRESHOLD`.
- `recoverIdentity(link)` restores experiential state and metrics from the link, then takes a new checkpoint.
- `onSubstrateMigration(event)` returns `MigrationRecord` with `continuityPreserved` true iff identity hashes match OR experiential drift < `CONTINUITY_PRESERVED_DRIFT_THRESHOLD`.
- `getIdentityDrift()` returns classification in {stable, evolving, concerning, critical} based on combined drift from baseline.

**Invariants:**
- The continuity chain is append-only; links are never removed or mutated.
- Every link at index i > 0 has `previousLink.identityHash === chain[i-1].identityHash`.
- Drift computations are deterministic: same inputs produce same scores.

### PersistenceManager

**Preconditions:**
- `initialize()` must be called before any save/load (creates state directory).
- `IFileSystem` implementation must be injected at construction.

**Postconditions:**
- `saveMemorySnapshot(s)` / `loadMemorySnapshot()` — round-trips `MemorySnapshot` as JSON. Returns `null` if file absent.
- `savePersonalitySnapshot(s)` / `loadPersonalitySnapshot()` — round-trips `PersonalitySnapshot` as JSON. Returns `null` if file absent.
- `saveDriveSnapshot(s)` / `loadDriveSnapshot()` — round-trips `DriveSnapshot` as JSON. Returns `null` if file absent.
- `hasState()` returns `true` iff any snapshot file exists.

**Invariants:**
- File paths are deterministic: `{stateDir}/memory-snapshot.json`, `{stateDir}/personality-snapshot.json`, `{stateDir}/drive-snapshot.json`.
- All I/O is through the injected `IFileSystem` — never `node:fs` directly.

### Boot Protocol (startup.ts)

**Preconditions:**
- All `AgentDependencies` fields are initialized (non-null subsystem instances).
- `AgentConfig` specifies `warmStart` and `agentId`.

**Postconditions:**
- Step 1: `valueKernel.verifyIntegrity()` must pass. If not, `startAgent()` throws (hard failure — no bypass).
- Step 2 (warm): `identityManager.recoverIdentity(lastContinuityLink)` is called, then `verifyIdentity()`. Anomalies are collected but do not block boot.
- Step 2 (cold or missing link): Boot proceeds without identity recovery; anomaly logged if warm was requested but link was missing.
- Step 3: Environment adapter is connected.
- Step 4: Returns `StartupResult` with `loop` in connected-but-idle state, `bootMode`, `valueIntegrityOk`, and `identityAnomalies`.

**Invariants:**
- Value kernel integrity failure is always fatal. No tick runs with a compromised value kernel.
- Identity anomalies are informational, not fatal — the agent can operate with drift but must log it.

## Threshold Registry

| Name | Value | Unit | Valid Range | Rationale | Sensitivity |
|------|-------|------|-------------|-----------|-------------|
| CONTINUITY_PRESERVED_DRIFT_THRESHOLD | 0.3 | ratio (0–1) | 0.1–0.5 | Max experiential drift that still counts as preserved continuity during migration. Below this, natural variation; above, identity discontinuity. | Higher → more lenient migration acceptance; lower → more migrations flagged as identity-breaking |
| HIGH_DRIFT_ANOMALY_THRESHOLD | 0.5 | ratio (0–1) | 0.3–0.8 | Drift above this triggers anomaly logging in identity verification. Set at midpoint to balance sensitivity vs. noise. | Higher → fewer anomaly alerts; lower → more alerts, possible alert fatigue |
| IDENTITY_STABLE_THRESHOLD | 0.05 | ratio (0–1) | 0.01–0.1 | Combined drift below this = identity unchanged. Very tight to ensure "stable" truly means no meaningful change. | Higher → more states classified stable; lower → only near-identical states qualify |
| IDENTITY_EVOLVING_THRESHOLD | 0.25 | ratio (0–1) | 0.1–0.4 | Combined drift in [stable, evolving) = normal growth. Tracks expected personality/experience evolution. | Higher → more tolerance for change before concern; lower → earlier concern triggers |
| IDENTITY_CONCERNING_THRESHOLD | 0.5 | ratio (0–1) | 0.3–0.7 | Combined drift in [evolving, concerning) = warrants investigation. Above this = critical. | Higher → delayed critical classification; lower → faster escalation |
| NARRATIVE_COHERENCE_EVOLVED | 0.8 | ratio (0–1) | 0.5–1.0 | Coherence score when self-model has changed. 0.8 = expected evolution, not a problem. | Higher → narrative changes less impactful on coherence; lower → narrative changes flagged as more significant |

## Behavioral Spec

### Boot Identity Verification Flow

Given an agent with `warmStart=true` and a persisted `ContinuityLink`
When `startAgent(deps, config)` is called
Then value kernel integrity is verified first (fatal on failure)
And `identityManager.recoverIdentity(lastContinuityLink)` restores state from the link
And `identityManager.verifyIdentity()` checks chain integrity and computes drift
And anomalies are collected in `StartupResult.identityAnomalies` (non-fatal)
And `bootMode` is reported as `'warm'`

### Cold Start Fallback

Given an agent with `warmStart=true` but no `lastContinuityLink` provided
When `startAgent(deps, config)` is called
Then boot degrades to cold start
And anomaly `"warmStart requested but no continuity link was available"` is logged
And `bootMode` is reported as `'cold'`

### Crash Recovery

Given the previous agent run terminated unexpectedly
When `recoverFromCrash(deps)` is called before `startAgent()`
Then the identity chain is inspected for anomalies (read-only, no state modification)
And the experience gap is estimated from the last checkpoint timestamp
And a `CrashRecoveryReport` is returned for the caller to persist
And the subsequent `startAgent()` warm start handles actual identity restoration
