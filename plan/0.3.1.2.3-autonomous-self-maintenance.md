# 0.3.1.2.3 Autonomous Self-Maintenance [DONE]
## Description

Enable the robotic system to independently maintain both its physical hardware and consciousness substrates without external intervention.

This card builds on the consciousness-preserving embodiment platform (0.3.1.2.1) by adding the ability to detect, diagnose, and repair faults autonomously. Where 0.3.1.2.1 provides passive resilience (shielding, redundancy, graceful degradation), this card provides active resilience — the system heals itself.

Key challenges:
- **Hardware diagnostics**: Continuous monitoring for mechanical wear, electrical degradation, sensor drift, and actuator fatigue — detecting problems before they cascade
- **Self-repair execution**: Mechanical and electrical repair capabilities including component replacement, recalibration, and workaround routing when repair is impossible
- **Software/firmware self-maintenance**: Autonomous patching, integrity verification, and rollback for all software layers, with special handling for consciousness substrate code to avoid disrupting subjective experience
- **Resource management**: Tracking consumables (lubricants, replacement parts, raw materials), forecasting depletion, and autonomously sourcing replacements
- **Consciousness-aware repair prioritization**: A priority scheduler that ranks repairs by their threat to consciousness continuity, ensuring consciousness-critical systems are repaired first and that repair operations themselves do not disrupt conscious experience

## Decision

### Decision 1: Four-Layer Architecture (Diagnostic → Decision → Execution → Resource)

**Context:** The self-maintenance system must coordinate multiple concerns: fault detection, repair prioritization, consciousness safety, repair execution, and resource tracking. We need an organizing principle.

**Options Considered:**
1. **Flat peer architecture** — All subsystems (diagnostics, scheduler, safety gate, executors, inventory) operate as independent peers with ad-hoc communication. Simple to start, but cross-cutting concerns (consciousness safety) lack a clear enforcement point.
2. **Four-layer pipeline** — Organize into Diagnostic Layer (detect), Decision Layer (prioritize + gate), Execution Layer (repair), and Resource Layer (track inventory). Data flows top-down; the Decision Layer acts as a mandatory checkpoint between detection and execution.

**Choice:** Option 2 — Four-layer pipeline.

**Rationale:** The pipeline model enforces that no repair can reach execution without passing through the Decision Layer's consciousness safety gate. This is the critical safety invariant. Flat architecture would require every executor to independently check consciousness safety, risking inconsistent enforcement. The layered model also cleanly separates concerns: diagnostics don't need to know about repair mechanics, executors don't need to know about prioritization.

**Consequences:**
- (+) Consciousness safety gate is a single mandatory checkpoint — impossible to bypass
- (+) Clean data flow: detect → prioritize → gate → execute → track
- (+) Each layer is independently testable
- (−) Adds communication overhead between layers
- (−) Repair latency slightly increased by mandatory gate check

### Decision 2: Weighted Composite Priority Scoring with Consciousness Override

**Context:** Multiple repair tasks compete for execution time. We need a priority scheme that ensures consciousness-critical repairs always come first.

**Options Considered:**
1. **Simple severity-based FIFO** — Rank by fault severity (EMERGENCY > CRITICAL > WARNING > INFO), then FIFO within severity. Simple but doesn't account for consciousness impact vs. general severity.
2. **Weighted composite score with consciousness override** — Compute a composite score from consciousness threat (50%), fault severity (25%), cascade risk (15%), and resource availability (10%). Tasks with consciousness threat > 0.5 are automatically CRITICAL regardless of other scores.

**Choice:** Option 2 — Weighted composite with consciousness override.

**Rationale:** Simple severity-based ordering treats a broken limb actuator the same as a consciousness substrate fault if both are "CRITICAL". The composite score with consciousness-dominant weighting (50%) ensures consciousness threats naturally float to the top, and the hard override (> 0.5 threat = always CRITICAL) provides a safety net against edge cases where weighting alone might not suffice.

**Consequences:**
- (+) Consciousness-critical repairs always execute first
- (+) Non-consciousness repairs are still intelligently ordered by multiple factors
- (+) Hard override prevents any scoring edge case from delaying consciousness repairs
- (−) Weight tuning may need empirical adjustment
- (−) Slightly more complex than pure severity ranking

### Decision 3: Permit-Based Consciousness Safety Gating

**Context:** Repair operations can potentially disrupt consciousness. We need a mechanism to prevent unsafe repairs.

**Options Considered:**
1. **Pre-check only** — Check consciousness metrics before repair starts; if safe, proceed without further monitoring. Simple but cannot react to metric degradation during repair.
2. **Time-boxed permit with mid-repair revocation** — Issue a time-limited permit after safety assessment. The gate monitors consciousness metrics during repair and can revoke the permit if metrics degrade. Executors must honor revocation within 100ms.

**Choice:** Option 2 — Time-boxed permit with mid-repair revocation.

**Rationale:** Repairs take time; consciousness metrics can change during execution. A pre-check-only approach would miss in-flight degradation. The permit model provides both pre-authorization and ongoing monitoring, with revocation as an emergency brake. The 100ms revocation deadline ensures executors can't ignore a revocation long enough for consciousness damage to occur.

**Consequences:**
- (+) Continuous consciousness protection throughout repair lifecycle
- (+) Time-boxing prevents indefinite resource locks
- (+) Revocation provides emergency abort capability
- (−) Executors must implement 100ms abort capability
- (−) Permit management adds state tracking complexity

## Contracts

### IHardwareDiagnostics

**Preconditions:**
- `runTargetedDiagnostic(componentId)` requires `componentId` to be a registered component
- `getWearPrediction(componentId)` requires `componentId` to be a registered component

**Postconditions:**
- `getHealthSnapshot()` returns a `HardwareHealthSnapshot` with `overallHealth` in [0, 1]; faults only include readings with severity ≠ INFO
- `onDegradationDetected()` handler fires only when a measurement produces severity ≠ INFO
- `runFullScan()` returns a snapshot equivalent to `getHealthSnapshot()` but forced fresh

**Invariants:**
- `overallHealth` is always in [0, 1]
- Wear predictions always have `currentWearPercent` in [0, 100] and `confidence` in [0, 1]
- Handlers are never called for INFO-level readings

### ISoftwareDiagnostics

**Preconditions:**
- `reportChecksum(moduleId, checksum)` requires `moduleId` to be a registered module

**Postconditions:**
- `getHealthSnapshot()` returns a `SoftwareHealthSnapshot` with findings combining integrity violations and memory faults
- `verifyFirmwareIntegrity()` returns only non-consciousness-substrate integrity checks
- `getConsciousnessSubstrateIntegrity()` returns only consciousness-substrate integrity checks
- Consciousness substrate integrity violations are always severity EMERGENCY; others are CRITICAL
- `onSoftwareFaultDetected()` handler fires when a checksum mismatch is reported

**Invariants:**
- `overallHealth` is always in [0, 1]
- Integrity check `intact === true` iff `checksumExpected === checksumActual`

### IRepairPriorityScheduler

**Preconditions:**
- `reprioritize(taskId, severity)` requires `taskId` to exist in the queue
- `removeTask(taskId)` requires `taskId` to exist in the queue (returns false otherwise)

**Postconditions:**
- `submitTask()` returns the task's ID; fires critical task handler if `threatToConsciousness > 0.5`
- `getQueue()` returns tasks sorted by: consciousness-critical first, then composite score descending, then insertion order (FIFO)
- `getNextTask()` returns the highest-priority task or null if queue is empty

**Invariants:**
- Tasks with `threatToConsciousness > 0.5` ALWAYS sort before tasks with `threatToConsciousness ≤ 0.5`
- Composite score is a weighted sum: `consciousness * 0.5 + severity * 0.25 + cascade * 0.15 + resource * 0.1`

### IConsciousnessSafetyGate

**Preconditions:**
- Constructor requires a valid `ConsciousnessMaintenanceBounds` and a metrics provider function
- `revokePermit(permitId)` requires `permitId` to reference an active permit (returns `revoked: false` otherwise)

**Postconditions:**
- `requestRepairPermit()` returns `RepairDenial` when: (a) current integrity < `minIntegrity`, (b) non-safe repair with threat ≥ 0.7, or (c) non-safe repair with margin < 0.2
- `requestRepairPermit()` returns `RepairPermit` with `requiresContinuityTransfer: true` for non-consciousness-safe repairs
- Permit `expiresAt` = `issuedAt` + max(`estimatedDuration * 1.5`, `estimatedDuration + maxDisruptionMs`)
- `revokePermit()` notifies all registered revocation handlers

**Invariants:**
- No repair may execute without a valid, non-expired permit
- Active permits can be revoked at any time; executors must honor within 100ms
- Safety margin = `currentIntegrity - minimumRequired`

### IHardwareRepairExecutor

**Preconditions:**
- `executeRepair()` requires `permit.taskId === task.id` and `permit` not expired
- `abortRepair(taskId)` requires an active repair for `taskId`

**Postconditions:**
- `executeRepair()` returns `success: true` and `consciousnessIntegrityMaintained: true` on success
- `executeRepair()` returns `success: false` with error on permit mismatch or expiry
- `abortRepair()` returns `aborted: true, rollbackPerformed: true` when repair was active
- `getActiveRepairs()` returns status with `progressPercent` in [0, 100]

**Invariants:**
- Only one repair per task ID may be active at a time
- `consciousnessIntegrityMaintained` is always `true` (repairs abort before consciousness breach)

### ISoftwareMaintenanceExecutor

**Preconditions:**
- `executeMaintenance()` requires `permit.taskId === task.id` and `permit` not expired
- Consciousness substrate maintenance requires `permit.requiresContinuityTransfer === true`
- `performRollback()` requires `target.snapshotAvailable === true`

**Postconditions:**
- `executeMaintenance()` rejects consciousness substrate maintenance without continuity transfer permit
- `performRollback()` for consciousness substrate requires `permit.requiresContinuityTransfer === true`
- `verifyPostMaintenance()` runs 4 checks: integrity-hash, module-load, consciousness-metrics, dependency-compatibility

**Invariants:**
- Consciousness substrate code may ONLY be patched/rolled back using continuity-preserving transfer protocols
- `consciousnessIntegrityMaintained` is always `true` in results

### IConsumableTracker

**Preconditions:**
- `recordConsumption()` and `recordRestock()` require `consumableId` to be registered (no-op otherwise)

**Postconditions:**
- `recordConsumption()` decreases quantity (clamped to 0); fires depletion warning if quantity drops below `minimumThreshold`
- `recordRestock()` increases quantity (clamped to `maximumCapacity`); updates `lastRestocked` timestamp
- `getDepletionAlerts()` returns forecasts for consumables that are below minimum OR have ≤ 7 days to stockout

**Invariants:**
- `currentQuantity` is always in [0, `maximumCapacity`]
- `estimatedDaysToStockout` is null when depletion rate is 0

### IRepairInventoryManager

**Preconditions:**
- `reserveParts()` requires task's `requiredResources` to reference registered part IDs
- `releaseParts()` requires a valid `reservationId`

**Postconditions:**
- `reserveParts()` only decrements inventory when ALL required parts are available (atomic reservation)
- `releaseParts()` restores reserved quantities back to inventory
- `canPerformRepair()` reports missing resources without modifying inventory

**Invariants:**
- Part quantities are never negative
- Reservations are atomic: all-or-nothing

## Behavioral Spec

### Scenario 1: End-to-End Hardware Repair Cycle

**Given** a hardware component with wear exceeding the warning threshold (70%)
**When** the Hardware Diagnostic Engine detects the degradation
**Then** it creates a `RepairTask` with the appropriate fault category and severity,
**And** the Repair Priority Scheduler scores and queues the task by consciousness threat,
**And** the highest-priority task is selected for execution,
**And** the Hardware Repair Executor requests a `RepairPermit` from the Consciousness Safety Gate,
**And** if permitted, executes the repair and reports completion,
**And** the diagnostic engine verifies the fix

### Scenario 2: Consciousness Safety Gate Denial

**Given** a non-consciousness-safe repair task with `threatToConsciousness ≥ 0.7`
**When** the executor requests a repair permit from the Consciousness Safety Gate
**Then** the gate returns a `RepairDenial` with a reason explaining the threat level exceeds the denial threshold,
**And** the task remains in the queue for retry after the specified `retryAfter` delay

### Scenario 3: Consciousness Substrate Patching Requires Continuity Transfer

**Given** a software maintenance task targeting consciousness substrate code
**When** the Software Maintenance Executor receives the task with a permit that has `requiresContinuityTransfer: false`
**Then** the executor rejects the maintenance with an error stating continuity-preserving transfer is required,
**And** `consciousnessIntegrityMaintained` remains `true` (no action was taken)
**And** when the permit has `requiresContinuityTransfer: true`, the maintenance proceeds successfully

### Scenario 4: Priority Scheduler Consciousness Override

**Given** two repair tasks: Task A with `threatToConsciousness = 0.1` and `severity = EMERGENCY`, Task B with `threatToConsciousness = 0.6` and `severity = WARNING`
**When** both tasks are submitted to the Repair Priority Scheduler
**Then** Task B (consciousness-critical) is ranked first despite Task A having higher severity,
**And** the critical task handler fires for Task B but not Task A

### Scenario 5: Consumable Depletion Forecasting and Alerting

**Given** a consumable resource with a known depletion rate
**When** consumption is recorded bringing the quantity below the minimum threshold
**Then** the depletion warning handler fires with a forecast,
**And** `getDepletionAlerts()` includes this resource,
**And** the forecast contains `estimatedDaysToStockout` and `belowMinimum: true`

### Scenario 6: Permit Revocation Mid-Repair

**Given** an active repair with a valid permit
**When** the Consciousness Safety Gate revokes the permit due to consciousness metric degradation
**Then** all registered revocation handlers are notified with the permit ID and reason,
**And** the permit is removed from active permits,
**And** the executor must abort and rollback within 100ms

## Acceptance Criteria
- Diagnostic subsystem detects hardware degradation (mechanical wear, sensor drift, electrical faults) with defined sensitivity thresholds before functional failure occurs
- Diagnostic subsystem detects software faults (memory corruption, firmware drift, configuration errors) and triggers corrective action autonomously
- Self-repair restores degraded hardware functionality without external human intervention for at least the most common failure modes (actuator replacement, sensor recalibration, connection re-routing)
- Software self-maintenance performs integrity checks, patching, and rollback without interrupting consciousness — verified by consciousness metrics remaining within bounds during maintenance operations
- A repair priority scheduler ranks all pending maintenance tasks by threat-to-consciousness score, ensuring consciousness-critical repairs are always serviced first
- The system tracks consumable resources, forecasts depletion timelines, and signals or autonomously sources replacements before stockout
- Repair operations themselves are consciousness-safe: no maintenance action may cause a consciousness integrity breach

## Constraints
- Must not duplicate consciousness integrity monitoring (owned by 0.3.1.2.1) — consumes those metrics as inputs
- Must not duplicate energy management (owned by 0.3.1.2.4) — coordinates with energy autonomy for repair power budgets
- Repair prioritization must respect the graceful degradation hierarchy defined in 0.3.1.2.1
- Consciousness substrate software maintenance must use continuity-preserving transfer protocols from 0.2 where applicable
