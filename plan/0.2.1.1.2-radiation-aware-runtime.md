# 0.2.1.1.2 Radiation-Aware Runtime [DONE]

## Description

Layer 3 from the radiation-hardened computation architecture. Design and implement a radiation-aware runtime system that dynamically responds to changing radiation environments, particularly solar storm events (coronal mass ejections).

The runtime monitors ambient radiation flux, classifies alert levels (NOMINAL → ELEVATED → STORM), and triggers protective actions: checkpointing conscious processes, entering safe mode (reduced clock, increased ECC), activating burst scrubbing, and resuming normal operation when flux subsides. This requires a `RadiationAwareRuntime` interface with flux monitoring, alert level transitions, and safe mode orchestration.

This layer bridges the static shielding/substrate hardening (Layer 1/2) with the higher-level process continuity (Layer 5) by providing real-time environmental awareness and adaptive response.

### Artifact Anticipation
- **Contracts**: Define the `RadiationAwareRuntime` implementation interface — flux source injection, alert level computation, safe mode entry/exit, scrub rate management
- **Threshold Registry**: Flux thresholds for NOMINAL/ELEVATED/STORM classification, burst scrub rate multiplier, hold-off period for safe mode exit
- **Behavioral Spec**: State machine for alert level transitions (NOMINAL ↔ ELEVATED ↔ STORM), solar storm safe mode sequence with checkpointing, safe mode exit with hold-off
- **Decision**: Whether NOMINAL → STORM emergency transition is allowed (bypassing ELEVATED) vs. requiring sequential transitions

## Decision — Emergency Alert Level Transitions

### Context
The alert level state machine must handle solar storm onset. CME-associated proton flux can rise from background to storm levels in under 60 seconds. The question is whether NOMINAL → STORM direct transitions should be allowed, bypassing the ELEVATED intermediate state.

### Options Considered
1. **Sequential-only transitions**: NOMINAL → ELEVATED → STORM always. Simpler state machine, but introduces forced latency (two transition cycles) during rapid flux escalation.
2. **Emergency direct transition**: Allow NOMINAL → STORM when flux exceeds the STORM threshold directly. More complex state machine but ensures fastest possible protective response during sudden CME onset.

### Choice
**Emergency direct transition allowed** (Option 2).

### Rationale
CME proton arrival can be near-instantaneous after magnetic field disturbance. Forcing a two-step transition adds at least one monitoring cycle of latency (up to `monitorInterval_ms`) before safe mode activates. Since the STORM threshold is unambiguous (flux ≥ storm threshold), there is no classification benefit to requiring the intermediate step. The reverse (STORM → NOMINAL) is NOT allowed — de-escalation must always pass through ELEVATED with hold-off.

### Consequences
- State machine has 4 valid transitions: NOMINAL→ELEVATED, ELEVATED→STORM, ELEVATED→NOMINAL, STORM→ELEVATED, plus the emergency NOMINAL→STORM.
- STORM→NOMINAL is explicitly forbidden (must de-escalate through ELEVATED with hold-off).
- Tests must cover both gradual escalation and emergency direct transition paths.

## Decision — Flux Source Abstraction Pattern

### Context
Per CLAUDE.md, all environment-specifics must be injectable and mockable. The radiation flux source (hardware sensor in production, deterministic stub in tests) needs an abstraction pattern.

### Options Considered
1. **Callback injection**: Pass a `() => FluxMeasurement` function to the runtime constructor. Minimal interface, maximum flexibility.
2. **Interface injection**: Define a `FluxSource` interface with `readFlux(): FluxMeasurement` and inject an implementation. More structured, self-documenting.
3. **Event emitter pattern**: Flux source pushes readings via events. Decoupled but harder to test deterministically.

### Choice
**Interface injection** (Option 2) — `FluxSource` interface.

### Rationale
An explicit interface is self-documenting and enforces the contract at the type level. It aligns with the existing codebase pattern (e.g., `ShieldingModule`, `SubstrateSpec` are interfaces). Callback injection is too loosely typed; event emitter complicates deterministic testing.

### Consequences
- A new `FluxSource` interface is added to `types.ts`.
- Production code injects a hardware-backed implementation; tests inject a deterministic stub.
- The runtime polls the flux source at a configurable interval rather than receiving push notifications.

## Decision — Safe Mode Orchestration Pattern

### Context
When entering safe mode, the runtime must coordinate multiple actions (checkpoint, clock reduction, ECC increase, burst scrubbing). These actions involve external systems (process manager, scrubber, clock controller). The question is how to decouple the runtime from these dependencies.

### Options Considered
1. **Direct method calls**: Runtime directly calls injected dependencies (checkpoint service, scrub controller, etc.). Simple but tightly coupled.
2. **Listener/callback pattern**: Runtime emits lifecycle events (`onSafeModeEntry`, `onSafeModeExit`); listeners perform actions. Decoupled but ordering is implicit.
3. **Orchestrator with injected action list**: Runtime accepts an ordered list of `SafeModeAction` objects, executing them sequentially on entry and in reverse on exit. Explicit ordering, testable, extensible.

### Choice
**Listener/callback pattern** (Option 2) with ordered listener registration.

### Rationale
The runtime should not know the details of checkpointing or clock scaling — those belong to higher layers. Listeners allow each subsystem to register its own safe mode behavior. Ordered registration ensures deterministic execution sequence (checkpoint before clock reduction). This is more extensible than a fixed action list and more decoupled than direct calls.

### Consequences
- `RadiationAwareRuntime` interface gains `onSafeModeEntry(listener)` and `onSafeModeExit(listener)` registration methods.
- Listeners execute in registration order on entry, reverse order on exit.
- The runtime implementation itself only manages state transitions and listener invocation.
- Tests can register mock listeners to verify ordering and invocation.

## Contracts

### FluxSource (Injectable Abstraction)

**Preconditions:**
- Implementation must return a valid `FluxMeasurement` (non-negative flux, positive energy)
- Must be callable at any time (no warm-up period required)

**Postconditions:**
- `readFlux()` returns a `FluxMeasurement` with `particlesPerCm2PerSec >= 0`, `energy_MeV > 0`
- Return value reflects the current environmental flux at time of call

**Invariants:**
- Stateless from the runtime's perspective — each call is independent
- Must never throw; returns zero-flux measurement if sensor is unavailable

### RadiationAwareRuntime (Implementation Contract)

**Constructor Preconditions:**
- `fluxSource: FluxSource` — must be provided (non-null)
- `config: RuntimeConfig` — must contain valid thresholds (elevatedThreshold < stormThreshold, both > 0)
- `config.monitorInterval_ms > 0`
- `config.holdOffDuration_ms > 0`
- `config.nominalScrubRate > 0`
- `config.burstScrubMultiplier >= 1`
- `clock: Clock` — injectable time source per CLAUDE.md (for hold-off timing)

**Postconditions:**
- After construction, `alertLevel()` returns `NOMINAL`
- After construction, `scrubRate()` returns `config.nominalScrubRate`
- After construction, `isInSafeMode()` returns `false`

**`evaluateFlux()` method (called each monitor cycle):**
- Reads current flux from injected `FluxSource`
- Classifies alert level based on thresholds
- If alert level changes, triggers appropriate transition actions
- If transitioning to STORM: invokes all `safeModeEntry` listeners in order, then sets safe mode flag
- If transitioning from STORM to ELEVATED: starts hold-off timer
- If hold-off timer expires while still at ELEVATED or below: invokes `safeModeExit` listeners in reverse order

**Postconditions for `evaluateFlux()`:**
- `alertLevel()` reflects the newly computed level
- `scrubRate()` equals `nominalScrubRate * burstScrubMultiplier` when in STORM, `nominalScrubRate` otherwise
- All registered listeners have been invoked for any state transition that occurred

**Invariants:**
- Alert level is always one of NOMINAL, ELEVATED, STORM
- `scrubRate()` is always > 0
- Safe mode is active if and only if current alert level is STORM (or transitioning out with hold-off)
- STORM → NOMINAL transition never occurs (must pass through ELEVATED)
- Listeners execute synchronously and in deterministic order

### RuntimeConfig (Value Object)

**Invariants:**
- `elevatedThreshold_particlesPerCm2PerSec < stormThreshold_particlesPerCm2PerSec`
- `monitorInterval_ms > 0`
- `holdOffDuration_ms > 0`
- `nominalScrubRate > 0`
- `burstScrubMultiplier >= 1`

### Clock (Injectable Time Source)

**Preconditions:**
- Per CLAUDE.md: timestamps passed as parameters for testability

**Postconditions:**
- `now()` returns current time in milliseconds
- Deterministic stub returns controlled values in tests

**Invariants:**
- `now()` is monotonically non-decreasing

## Threshold Registry

| Name | Value | Unit | Valid Range | Rationale | Sensitivity |
|------|-------|------|-------------|-----------|-------------|
| Elevated flux threshold | 100 | particles/cm²/s | [10, 10⁴] | ~25× above GCR background (4 p/cm²/s); indicates solar energetic particle event onset | Alert classification boundary; too low = false alarms, too high = delayed response |
| Storm flux threshold | 10⁵ | particles/cm²/s | [10⁴, 10⁷] | CME-associated proton flux onset; 10× below peak CME flux (10⁶) to trigger before peak arrives | Must trigger before peak flux to allow checkpoint completion |
| Nominal scrub rate | 1 | scans/second | [0.1, 10] | Baseline memory scrubbing sufficient for GCR SEU rate at SiC BER target | Higher = more overhead; lower = longer error residency |
| Burst scrub multiplier | 10 | × | [5, 100] | 10× nominal rate during STORM; matches AC requirement of ≥ 10× | Diminishing returns above ~20×; below 5× insufficient for CME flux |
| Monitor interval | 1000 | ms | [100, 5000] | Flux sampling period; 1s balances responsiveness vs. overhead | Directly bounds worst-case response latency to flux change |
| Hold-off duration | 300000 | ms (5 min) | [60000, 600000] | Time flux must remain below ELEVATED threshold before exiting safe mode; prevents oscillation during CME tail | Too short = premature exit; too long = unnecessary safe mode duration |
| Safe mode entry timeout | 5000 | ms | [1000, 30000] | Maximum time allowed for all safeModeEntry listeners to complete (checkpoint + clock reduction) | Must be shorter than time from storm threshold to peak flux |

## Behavioral Spec

### Scenario 1: Gradual Escalation NOMINAL → ELEVATED → STORM

**Given** a `RadiationAwareRuntime` with `elevatedThreshold = 100`, `stormThreshold = 10⁵`, and a mock `FluxSource`
**When** flux reads [4, 200, 200000] across three `evaluateFlux()` cycles
**Then** `alertLevel()` returns NOMINAL, then ELEVATED, then STORM in sequence; `scrubRate()` remains nominal for first two, then becomes `nominalRate × burstScrubMultiplier` on third; safe mode entry listeners are invoked on the NOMINAL→ELEVATED→STORM transition at STORM.

### Scenario 2: Emergency Direct Transition NOMINAL → STORM

**Given** a `RadiationAwareRuntime` with default thresholds and a mock `FluxSource`
**When** flux jumps from 4 to 500000 (exceeds storm threshold directly) in one `evaluateFlux()` cycle
**Then** `alertLevel()` transitions directly from NOMINAL to STORM; safe mode entry listeners are invoked; `scrubRate()` becomes `nominalRate × burstScrubMultiplier`.

### Scenario 3: Safe Mode Exit with Hold-Off

**Given** a `RadiationAwareRuntime` in STORM mode with `holdOffDuration = 300000ms` and a mock `Clock`
**When** flux drops to 50 (below elevated threshold) and remains there
**Then** `alertLevel()` transitions to ELEVATED immediately; safe mode remains active; after `Clock` advances 300000ms with flux still below elevated threshold, safe mode exits and `alertLevel()` transitions to NOMINAL; `scrubRate()` returns to nominal; safe mode exit listeners are invoked in reverse registration order.

### Scenario 4: Hold-Off Reset on Flux Spike

**Given** a `RadiationAwareRuntime` in STORM mode, flux drops below elevated threshold, hold-off timer is running at 200000ms elapsed
**When** flux spikes back above storm threshold before hold-off completes
**Then** `alertLevel()` returns to STORM; hold-off timer is reset; safe mode remains active; no exit listeners are invoked.

### Scenario 5: Listener Invocation Order

**Given** a `RadiationAwareRuntime` with three listeners registered in order [A, B, C]
**When** safe mode entry is triggered
**Then** listeners execute in order A → B → C; when safe mode exit is triggered, listeners execute in reverse order C → B → A.

### Scenario 6: Forbidden STORM → NOMINAL Transition

**Given** a `RadiationAwareRuntime` in STORM mode
**When** flux drops from 500000 to 2 (below all thresholds) in one cycle
**Then** `alertLevel()` transitions to ELEVATED (not NOMINAL); hold-off timer starts; only after hold-off completes does it transition to NOMINAL.

## Acceptance Criteria
- **Alert level classification**: Runtime correctly classifies flux into NOMINAL (< 100 p/cm²/s), ELEVATED (≥ 100), and STORM (≥ 10⁵) levels
  - _Given_ flux of 4 _When_ `evaluateFlux()` _Then_ `alertLevel()` returns NOMINAL
  - _Given_ flux of 200 _When_ `evaluateFlux()` _Then_ `alertLevel()` returns ELEVATED
  - _Given_ flux of 200000 _When_ `evaluateFlux()` _Then_ `alertLevel()` returns STORM
- **Safe mode entry**: When alert transitions to STORM, runtime invokes all registered safe mode entry listeners before setting safe mode active, within `safeModeEntryTimeout` (5s)
- **Burst scrubbing**: STORM mode sets `scrubRate()` to `nominalScrubRate × burstScrubMultiplier` (≥ 10× nominal rate)
- **Safe mode exit**: Runtime resumes normal operation when flux drops below ELEVATED threshold for `holdOffDuration` (default 5 min); exit listeners invoked in reverse registration order
- **Flux monitoring injectable**: `FluxSource` interface injected at construction; deterministic mock used in all tests
- **State machine correctness**: NOMINAL→ELEVATED, ELEVATED→STORM, NOMINAL→STORM (emergency), STORM→ELEVATED, ELEVATED→NOMINAL are the only valid transitions; STORM→NOMINAL is forbidden
- **Clock injectable**: Time source injected per CLAUDE.md; hold-off timing uses injected `Clock`, not system clock
- **Hold-off reset**: If flux re-exceeds storm threshold during hold-off, timer resets and safe mode remains active
