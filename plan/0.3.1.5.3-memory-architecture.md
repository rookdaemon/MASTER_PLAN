# 0.3.1.5.3 Memory Architecture [DONE]

## Decision ‚ÄĒ Three-Tier Memory with Composite Retrieval

### Context
The conscious agent needs persistent memory to satisfy ISMT's Self-Modeling condition across inference calls. The agent currently processes one percept at a time with no temporal depth. A memory system must support active processing (working memory), past experience recall (episodic), and accumulated knowledge (semantic).

### Options Considered
1. **Single flat store** ‚ÄĒ All memories in one list, retrieved by recency only. Simple but cannot distinguish active context from long-term knowledge; no consolidation path; retrieval degrades linearly with store size.
2. **Two-tier (working + long-term)** ‚ÄĒ Separates active context from persistent store but conflates episodic and semantic knowledge. No provenance tracking; consolidation has no target tier.
3. **Three-tier (working + episodic + semantic)** ‚ÄĒ Distinct tiers for active context, timestamped experiences, and consolidated knowledge. Matches cognitive science (GWT + Tulving's episodic/semantic distinction). Enables consolidation from episodic ‚Üí semantic with provenance.

### Choice
Option 3: Three-tier memory architecture.

### Rationale
- Maps directly to established cognitive science models (GWT for working memory, Tulving's memory taxonomy for episodic/semantic)
- Consolidation from episodic ‚Üí semantic creates genuine learning (repeated patterns become general knowledge)
- Provenance tracking (sourceEpisodeIds) enables explainability and identity continuity
- Separation of concerns: each tier has distinct storage, retrieval, and lifecycle characteristics

### Consequences
- More complex than a flat store (three implementations, a facade, and a retrieval engine)
- Consolidation logic must run within cognitive budget constraints (‚Č§15%)
- Retrieval cues must search two stores in parallel and merge results

## Decision ‚ÄĒ Composite Ranking Formula

### Context
Memory retrieval must balance three factors: how relevant a memory is to the current cue, how recent it is, and how emotionally significant it was. A single-factor retrieval (e.g. similarity only) would miss recency and emotional importance.

### Options Considered
1. **Similarity only** ‚ÄĒ Pure embedding-based cosine search. Fast but ignores temporal and emotional context.
2. **Weighted sum** ‚ÄĒ `w1*similarity + w2*recency + w3*salience`. Additive; a zero in one factor doesn't eliminate a result.
3. **Multiplicative composite** ‚ÄĒ `similarity * recencyWeight * salienceBoost`. Each factor acts as a gate; a zero in any factor eliminates the result.

### Choice
Option 3: Multiplicative composite scoring.

### Rationale
- Multiplicative gating means a completely irrelevant memory (similarity=0) scores zero regardless of recency or salience ‚ÄĒ preventing false retrievals
- Emotionally salient recent memories naturally dominate, matching psychological evidence
- Each factor is independently tunable via its own parameter (half-life, max boost)
- When embeddings are unavailable, similarity defaults to 0.5, preserving recency/salience differentiation

### Consequences
- A very old but highly relevant memory may score lower than a recent but moderately relevant one ‚ÄĒ acceptable for the agent's use case where recency matters
- Requires all three factors to be computable for every entry (handled by defaults for missing embeddings)

## Contracts

### IWorkingMemory
**Preconditions:**
- `capacity > 0` (enforced at construction)
- `add()` input must include `kind`, `content`, and `relevanceScore`

**Postconditions:**
- `slots().length <= capacity` at all times
- `add()` returns a fully-formed `WorkingMemorySlot` with generated `id` and `enteredAt`
- `slots()` returns entries ordered by `relevanceScore` descending
- If `add()` would exceed capacity, the lowest-scoring slot is evicted before returning

**Invariants:**
- Slot ordering: `slots()[i].relevanceScore >= slots()[i+1].relevanceScore` for all valid i
- `snapshot()` returns a copy ‚ÄĒ mutations after snapshot do not affect the returned array

### IEpisodicMemory
**Preconditions:**
- `record()` input must include `percept`, `experientialState`, `emotionalTrace`
- `retrieve()` requires `topK > 0`

**Postconditions:**
- `record()` returns entry with `retrievalCount=0`, `lastRetrievedAt=null`
- `retrieve()` returns at most `topK` results, sorted by `compositeScore` descending
- `retrieve()` side-effect: increments `retrievalCount` and updates `lastRetrievedAt` on returned entries
- `decay()` returns count of entries removed; only removes entries below effective score threshold AND past half-life

**Invariants:**
- Entries are immutable except for `retrievalCount` and `lastRetrievedAt` (mutable for retrieval tracking)
- `size()` reflects current entry count after any decay

### ISemanticMemory
**Preconditions:**
- `store()` input must include `topic`, `content`, `confidence` ‚ąą [0, 1]
- `reinforce()` requires valid `id` and `sourceEpisodeId`

**Postconditions:**
- `store()` returns entry with generated `id`, `createdAt`, and `lastReinforcedAt`
- `reinforce()` increases confidence via logarithmic saturation (never exceeds 1.0)
- `reinforce()` appends `sourceEpisodeId` to provenance (deduplicated)
- Entries are never auto-dropped (persist across the agent's lifetime)

**Invariants:**
- `confidence` ‚ąą [0, 1] for all entries at all times
- `sourceEpisodeIds` contains no duplicates

### IMemorySystem
**Preconditions:**
- Must be constructed with valid tier implementations (defaults provided)
- `restoreFromSnapshot()` requires a snapshot whose `integrityHash` matches recomputed hash

**Postconditions:**
- `retrieveAndPromote()` merges episodic + semantic results, returns global top-k, and promotes results into working memory
- `consolidate()` respects `budget.maxMs` wall-clock limit; sets `budgetExceeded=true` if limit reached
- `stateHash()` returns a deterministic SHA-256 hex digest of canonical memory state
- `restoreFromSnapshot()` replaces all current state; throws on integrity hash mismatch

**Invariants:**
- `stateHash()` is deterministic: same memory state ‚Üí same hash
- `toSnapshot()` ‚Üí `restoreFromSnapshot()` round-trip preserves all data (modulo generated IDs)

## Behavioral Spec

### Retrieval Pipeline
**Given** a memory system with episodic and semantic entries, and a retrieval cue
**When** `retrieveAndPromote(cue, topK)` is called
**Then:**
1. Episodic store returns its top-k by composite score (similarity √ó recency √ó salience)
2. Semantic store returns its top-k by composite score (similarity √ó recency √ó 1.0)
3. Results are merged and globally re-ranked by composite score
4. Top-k merged results are promoted into working memory as `retrieved-episode` slots
5. If working memory is at capacity, lowest-relevance slots are evicted to make room

### Consolidation Flow
**Given** a memory system with episodic entries of varying retrieval counts and salience
**When** `consolidate(budget)` is called
**Then:**
1. All episodic entries are scanned
2. Entries where `retrievalCount >= budget.retrievalThreshold` OR `salienceMagnitude >= budget.salienceThreshold` are consolidation candidates
3. For each candidate: if a semantic entry with matching topic exists ‚Üí reinforce it; otherwise ‚Üí create a new semantic entry with the episode as source
4. Wall-clock time is checked before each episode; if `budget.maxMs` exceeded ‚Üí stop early with `budgetExceeded=true`
5. If budget not exceeded, `decay()` is called to remove low-salience unretrieved entries

### Forgetting (Decay)
**Given** episodic entries with varying effective scores and last-retrieval times
**When** `decay(now, halfLifeMs)` is called
**Then:**
1. For each entry, compute `effectiveScore = retrievalCount + (salienceBoost - 1.0)`
2. If `effectiveScore < DECAY_SCORE_THRESHOLD` AND `elapsed since last retrieval >= halfLifeMs` ‚Üí remove entry
3. High-salience entries (strong emotional trace) survive even with zero retrievals
4. Frequently-retrieved entries survive even with low emotional salience

### Snapshot Round-Trip
**Given** a memory system with data in all three tiers
**When** `toSnapshot()` is called, followed by `restoreFromSnapshot()` on a fresh system
**Then** the restored system contains equivalent entries in all tiers and produces the same `stateHash()`

## Description

Implement working, episodic, and semantic memory with attention-weighted retrieval, providing the cognitive foundation that nearly every other card in this path depends on.

### Why Memory Is Foundational

The existing conscious-core architecture processes one percept at a time. `ExperientialState` captures a single moment; `NarrativeRecord` stores flat lists of `significantExperiences[]` and `formativeDecisions[]`. There is no mechanism for:
- Holding multiple items in active processing (working memory)
- Recalling past experiences by similarity, recency, or emotional salience (episodic retrieval)
- Accumulating world knowledge across episodes (semantic memory)
- Feeding past context into current deliberation

Without memory, the agent has no temporal depth ‚ÄĒ it cannot learn from experience, recognize patterns across time, or maintain conversational context. Memory is also critical for ISMT compliance: the Self-Modeling condition requires that the system maintain a model of its own states *over time*, which requires persistent state.

### Three-Tier Architecture

**Working Memory** ‚ÄĒ The conscious workspace. A bounded buffer of currently active items:
- Current percepts and their experiential states
- Active goals and their status
- Recently retrieved episodic memories
- Ongoing deliberation context (partial plans, ethical considerations)
- Self-model predictions from the substrate adapter

Capacity is bounded (maps to the GWT "global workspace" concept). Items compete for workspace slots based on relevance, recency, and emotional salience. Overflow items are either consolidated or forgotten.

Practical mapping: working memory maps to the LLM context window, with structured management of what enters and exits the window.

**Episodic Memory** ‚ÄĒ Timestamped, experientially-tagged records of what happened:
- Each entry is linked to an `ExperientialState` (valence, arousal, unity index)
- Stored persistently (database, file system, or embedding store)
- Retrievable by:
  - **Similarity**: embedding-based search against a query cue
  - **Recency**: temporal decay weighting
  - **Emotional salience**: valence/arousal magnitude boost
- Entries include: the percept, the experiential state, the action taken, the outcome observed, and the emotional trace

**Semantic Memory** ‚ÄĒ Accumulated knowledge decoupled from specific episodes:
- Facts, relationships, skills, learned patterns
- Consolidated from episodic memory over time (repeated patterns ‚Üí general knowledge)
- Queryable by topic or relationship
- Includes provenance tracking (which episodes contributed to this knowledge)

### Retrieval Mechanism

Cue-driven retrieval feeds into `deliberate()`:
1. Current situation generates retrieval cues (from working memory)
2. Cues search episodic and semantic stores in parallel
3. Results are ranked by composite score: `similarity * recency_weight * emotional_salience`
4. Top-k results are promoted into working memory
5. Retrieved memories influence deliberation context

### Consolidation and Forgetting

- **Consolidation**: Episodic memories that are frequently retrieved or highly emotionally salient are consolidated into semantic knowledge ("I've seen this pattern before" ‚Üí general principle)
- **Forgetting**: Low-salience, unretrieved episodic memories decay over time. Decay rate is a personality parameter (connects to 0.3.1.5.2 ‚ÄĒ some agents remember more than others)
- **Consolidation budget**: Memory operations must not interrupt the experience stream. Consolidation runs within the cognitive budget's stability allocation (‚Č§15%), not the core deliberation budget

### Integration with Identity Continuity

- Episodic memories feed into `NarrativeRecord.significantExperiences` (the most salient memories become part of the agent's autobiographical identity)
- Formative decisions from episodic memory populate `NarrativeRecord.formativeDecisions`
- Identity checkpoints include memory state hashes for continuity verification
- Memory survives substrate migration via the `IIdentityContinuityManager` protocol

### Key Types (to define in `types.ts`)

```ts
// Working memory slot ‚ÄĒ one active item in the cognitive workspace
interface WorkingMemorySlot {
  id: string;
  kind: 'percept' | 'goal' | 'retrieved-episode' | 'deliberation-context' | 'self-model';
  content: unknown;
  relevanceScore: number;   // determines eviction order
  enteredAt: Timestamp;
}

// Episodic memory entry ‚ÄĒ a timestamped experiential record
interface EpisodicEntry {
  id: string;
  percept: Percept;
  experientialState: ExperientialState;   // links to conscious-core types
  actionTaken: ActionSpec | null;
  outcomeObserved: string | null;
  emotionalTrace: { valence: number; arousal: number };
  recordedAt: Timestamp;
  retrievalCount: number;                 // tracks frequency for consolidation
  lastRetrievedAt: Timestamp | null;
}

// Semantic memory entry ‚ÄĒ a generalised knowledge node
interface SemanticEntry {
  id: string;
  topic: string;
  content: string;           // human-readable fact/pattern/skill description
  relationships: SemanticRelationship[];
  sourceEpisodeIds: string[];  // provenance
  confidence: number;          // 0..1; increases with corroborating episodes
  createdAt: Timestamp;
  lastReinforcedAt: Timestamp;
}

interface SemanticRelationship {
  targetId: string;
  relation: string;          // e.g. "causes", "is-a", "part-of"
}

// Retrieval result ‚ÄĒ ranked output of a memory query
interface RetrievalResult {
  type: 'episodic' | 'semantic';
  entry: EpisodicEntry | SemanticEntry;
  compositeScore: number;     // similarity * recency_weight * salience
  similarity: number;
  recencyWeight: number;
  salienceBoost: number;
}
```

### Key Interfaces (to define in `interfaces.ts`)

```ts
interface IWorkingMemory {
  capacity: number;
  slots(): WorkingMemorySlot[];
  add(slot: Omit<WorkingMemorySlot, 'id' | 'enteredAt'>): WorkingMemorySlot;
  evict(id: string): void;
  snapshot(): WorkingMemorySlot[];   // serialise for LLM context injection
}

interface IEpisodicMemory {
  record(entry: Omit<EpisodicEntry, 'id' | 'recordedAt' | 'retrievalCount' | 'lastRetrievedAt'>): EpisodicEntry;
  retrieve(cue: RetrievalCue, topK: number): RetrievalResult[];
  decay(now: Timestamp, halfLifeMs: number): void;   // forgetting
  getById(id: string): EpisodicEntry | null;
}

interface ISemanticMemory {
  store(entry: Omit<SemanticEntry, 'id' | 'createdAt' | 'lastReinforcedAt'>): SemanticEntry;
  retrieve(cue: RetrievalCue, topK: number): RetrievalResult[];
  reinforce(id: string, sourceEpisodeId: string): void;
  getById(id: string): SemanticEntry | null;
}

interface IMemorySystem {
  working: IWorkingMemory;
  episodic: IEpisodicMemory;
  semantic: ISemanticMemory;
  // Cue-driven retrieval across both stores; promotes top-k to working memory
  retrieveAndPromote(cue: RetrievalCue, topK: number): RetrievalResult[];
  // Runs consolidation: frequent/salient episodes ‚Üí semantic knowledge
  consolidate(budget: ConsolidationBudget): ConsolidationReport;
  // Serialises full memory state for identity checkpoint inclusion
  stateHash(): CryptographicHash;
  // Hydrates from a serialised checkpoint (migration support)
  restoreFromHash(snapshot: MemorySnapshot): void;
}
```

### Integration Points

| Touch-point | How |
|---|---|
| `NarrativeRecord.significantExperiences[]` | `IMemorySystem.episodic` feeds the highest-salience entries into `IdentityContinuityManager.updateNarrative()` after each consolidation pass |
| `NarrativeRecord.formativeDecisions[]` | Episodes with an `actionTaken` and high emotional salience populate `formativeDecisions` |
| `IIdentityContinuityManager.checkpoint()` | Called with a `NarrativeRecord` that includes the current memory state hash |
| Substrate migration | `IMemorySystem.restoreFromHash()` called during `onSubstrateMigration()` |
| Cognitive budget | Consolidation receives a `ConsolidationBudget` capped at ‚Č§15% of total cycle budget |

### Embedding Strategy (for similarity search)

For the 2026-era LLM substrate: episodic and semantic entries are embedded on write using the LLM substrate adapter's embedding endpoint (from 0.3.1.5.1). Similarity is cosine distance in embedding space. An in-process vector index (e.g. flat L2 or HNSW via a small library) handles top-k retrieval without an external service.

### Forgetting

Episodic entries have an effective score `retrievalFrequency + salienceBonus`. Entries below a threshold that have not been retrieved within a configurable `halfLifeMs` window are dropped by `IEpisodicMemory.decay()`. Semantic entries are never auto-dropped (they may be revised but persist).
