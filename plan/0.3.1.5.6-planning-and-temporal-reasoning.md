# 0.3.1.5.6 Planning and Temporal Reasoning [DONE]
## Description

Upgrade `deliberate()` from "pick the highest-priority goal" to actual multi-step planning with temporal awareness and execution monitoring.

### Current State

`ConsciousCore.deliberate()` currently sorts goals by priority and selects the top one:
```ts
const sorted = [...goals].sort((a, b) => b.priority - a.priority);
const topGoal = sorted[0];
```

This produces a single action per cycle. There is no plan decomposition, no tracking of progress across cycles, and no replanning when conditions change.

### What Planning Requires

**Plan Generation** — Decompose a goal into a sequence of actions:
- Each action has preconditions (what must be true before it can execute) and expected postconditions (what should be true after)
- Preconditions are checked against the world model (0.3.1.5.5)
- Plans are generated via the LLM substrate, structured by the planning module
- Plans are experientially grounded: each plan step carries an `experientialBasis` reference (no zombie planning — the conscious core must experience the plan formation)

**Plan Execution Monitoring** — Track progress and detect failure:
- After each action, compare actual outcomes against expected postconditions
- If expectations are violated, trigger replanning
- If a plan step fails repeatedly, escalate: reconsider the goal itself (is it still achievable? still worthwhile?)
- Progress state persists in working memory across cycles

**Temporal Reasoning** — The agent needs a sense of time:
- Deadlines: some goals have time constraints
- Sequencing: action A must happen before action B
- Wait states: some plans require waiting (for external events, for other agents, for cooldown periods)
- Patience: ability to maintain a plan across many cycles without prematurely abandoning it (patience level is a personality parameter from 0.3.1.5.2)

**Means-Ends Analysis** — Generate instrumental subgoals:
- If a goal's preconditions aren't met, generate subgoals to establish them
- Instrumental subgoals must trace to the terminal goal that motivated them (connects to the Goal Coherence Engine's derivation graph from 0.3.1.3)
- Orphan subgoals (not traceable to terminal goals) are flagged by the stability system

### Cognitive Budget Compliance

Planning must operate within the architecture's cognitive budget:
- Planning runs within the "core deliberation" allocation (≥25% of cycle budget)
- Planning that would exceed budget is truncated — produce the best plan possible within the time budget, not an optimal plan that starves the experience stream
- Complex plans may be generated across multiple cycles (the plan-so-far persists in working memory)

## Decision — Planning Integration Pattern

### Context
`ConsciousCore.deliberate()` has signature `(state, goals) => Decision`. Planning requires injecting a planner, active plan, world context, and coherence engine without breaking backward compatibility.

### Options Considered
1. **Constructor injection** — pass IPlanner to ConsciousCore constructor; always-on planning
2. **Optional context parameter** — add optional `DeliberationContext` to `deliberate()` signature; caller controls whether planning is active
3. **Subclass** — create `PlanningConsciousCore extends ConsciousCore` with planning-aware deliberate()

### Choice
Option 2 — Optional `DeliberationContext` parameter.

### Rationale
- Preserves backward compatibility: callers not supplying context get legacy priority-sort
- Agent Runtime (0.3.1.5.9) assembles context each cycle, giving it control over plan lifecycle
- No constructor change means existing tests and integrations are unaffected
- The context is mutable (`activePlan`, `lastActionResult`) so the runtime's working-memory slot updates in-place

### Consequences
- `DeliberationContext` couples `deliberate()` to planning types (acceptable — it's optional)
- Runtime is responsible for persisting `activePlan` across cycles in working memory
- `deliberateWithPlanner()` is a private method; the public API surface is unchanged

## Decision — Precondition/Postcondition Comparison Strategy

### Context
Preconditions compare `WorldContext.facts[key]` against expected values. Postconditions compare `ActionResult` fields against expected values. Need a comparison mechanism that works for primitives and simple objects.

### Options Considered
1. **JSON deep equality** (`JSON.stringify(a) === JSON.stringify(b)`) — zero dependencies, adequate for flat fact maps
2. **Structural deep-equal library** (e.g. `fast-deep-equal`) — handles edge cases (Map, Set, circular refs)
3. **Custom comparator per condition** — maximum flexibility, highest complexity

### Choice
Option 1 — JSON deep equality via a module-level `deepEqual()` helper.

### Rationale
- WorldContext.facts and ActionResult fields are flat key→value maps — no circular structures, no Maps/Sets
- Zero external dependencies aligns with project modularity principles
- Sufficient for all current use cases; can be upgraded to option 2 later if needed

### Consequences
- Cannot compare values with circular references (not needed for current types)
- `undefined` values are silently dropped by JSON.stringify (acceptable — facts are explicit)

## Decision — Plan Generation Strategy

### Context
Plan generation requires reasoning about goal decomposition. This reasoning must be delegated to the LLM substrate per the architecture's separation of concerns.

### Options Considered
1. **Direct LLM integration** — Planner calls LLM substrate directly for step generation
2. **Placeholder generation** — Planner generates deterministic placeholder steps; LLM integration deferred to 0.3.1.5.9

### Choice
Option 2 — Placeholder generation for now, with the `generationStrategy` option in `PlannerOptions` to flag future LLM integration.

### Rationale
- The Planner is a thin orchestration layer — its value is in lifecycle management (precondition checking, replanning, subgoal registration, budget enforcement), not in the reasoning itself
- Placeholder steps exercise the full orchestration machinery without an LLM dependency
- The LLM substrate (0.3.1.5.1) integration point is clearly marked in `buildStepsForGoal()`

### Consequences
- All generated plans have the same 3-step structure (establish preconditions → execute → verify)
- Production LLM integration will replace `buildStepsForGoal()` — no other changes needed

## Contracts

### IPlanner

**Defined in:** `src/conscious-core/planner-interfaces.ts`

```ts
interface IPlanner {
  generatePlan(goal: Goal, state: ExperientialState, worldContext: WorldContext, budgetMs: number): Plan;
  checkPreconditions(step: PlanStep, worldContext: WorldContext): PreconditionCheckResult;
  evaluateOutcome(step: PlanStep, actualOutcome: ActionResult, worldContext: WorldContext): PostconditionCheckResult;
  replan(plan: Plan, failureReason: PlanFailureReason, state: ExperientialState, worldContext: WorldContext, budgetMs: number): Plan;
  registerSubgoals(plan: Plan, coherenceEngine: IGoalCoherenceEngine): void;
  shouldAbandon(plan: Plan, escalationCount: number): boolean;
}
```

**Preconditions:**
- `generatePlan`: `goal` is non-null; `state` is a valid ExperientialState (enforces no-zombie-planning); `budgetMs > 0`
- `checkPreconditions`: `step.preconditions` is an array; `worldContext.facts` is a record
- `evaluateOutcome`: `step.postconditions` is an array; `actualOutcome` is a valid ActionResult
- `replan`: `plan.terminalGoal` is non-null; `plan.escalationCount` reflects prior replanning attempts
- `registerSubgoals`: `plan.steps` may contain steps with non-null `instrumentalGoalId`
- `shouldAbandon`: `escalationCount >= 0`

**Postconditions:**
- `generatePlan`: returns Plan with `steps.length >= 1`, `experientialBasis === state`, `status === 'pending'` (or `'suspended'` if budget exceeded), `escalationCount === 0`
- `checkPreconditions`: returns `PreconditionCheckResult` with `satisfied === true` iff all precondition fact keys match expected values
- `evaluateOutcome`: returns `PostconditionCheckResult` with `met === true` iff all postcondition result keys match expected values; `actualOutcome` is preserved in result
- `replan`: returns Plan with `escalationCount === plan.escalationCount + 1`; terminal goal preserved; first step description annotated with `failureReason`
- `registerSubgoals`: calls `coherenceEngine.addGoal()` once per step with non-null `instrumentalGoalId`; each AgencyGoal has `derivedFrom: [plan.terminalGoal.id]` and `type: 'instrumental'`
- `shouldAbandon`: returns `true` iff `escalationCount >= maxEscalations` (configured via PlannerOptions)

**Invariants:**
- Every Plan produced by `generatePlan()` or `replan()` carries a non-null `experientialBasis` (no zombie planning)
- `evaluateOutcome` includes `worldContext` parameter (reserved for future world-state postcondition checking)
- `replan` always produces a fresh Plan object (never mutates the input plan)

### DeliberationContext

**Defined in:** `src/conscious-core/planner-interfaces.ts`

```ts
interface DeliberationContext {
  readonly planner: IPlanner | null;
  activePlan: Plan | null;          // mutable — updated in-place by deliberateWithPlanner
  readonly worldContext: WorldContext | null;
  readonly coherenceEngine: IGoalCoherenceEngine | null;
  readonly budgetMs: number;
  lastActionResult: ActionResult | null;  // mutable — consumed (set to null) after evaluation
}
```

**Invariants:**
- When `planner` is non-null, `worldContext` must also be non-null (enforced by guard in `deliberate()`)
- `lastActionResult` is consumed (set to null) after postcondition evaluation — never re-evaluated
- `activePlan` is mutated in-place so the caller's working-memory reference stays current

## Threshold Registry

| Name | Value | Unit | Valid Range | Rationale | Sensitivity |
|------|-------|------|-------------|-----------|-------------|
| DEFAULT_MAX_ESCALATIONS | 3 | count | 1–10 | Patience threshold for replanning before abandoning a plan. Maps to personality "patience level" (0.3.1.5.2). 3 balances persistence with avoiding infinite loops. | Low values (1) make agent give up quickly; high values (>5) risk wasting cognitive budget on unachievable goals. |
| Replanned decision confidence multiplier | 0.7 | ratio | 0.1–1.0 | Confidence for decisions from replanned plans is `unityIndex × 0.7`, reflecting reduced certainty after plan failure. | Higher values (>0.8) understate replanning uncertainty; lower values (<0.5) may cause downstream action pipeline to deprioritize. |
| Normal decision confidence multiplier | 0.9 | ratio | 0.5–1.0 | Confidence for legacy priority-sort decisions is `unityIndex × 0.9`. | Minimal sensitivity — used only in legacy path. |
| World context confidence multiplier | 1.0 | ratio | 0.0–1.0 | Planning-path confidence is `unityIndex × worldContext.confidence`. Scales with world model certainty. | Directly proportional to world model quality. |
| Instrumental subgoal priority offset | -1 | priority units | -5 to 0 | Instrumental subgoals are registered with `terminalGoal.priority - 1`, keeping them slightly below their parent. | Larger offsets risk instrumental goals being starved; 0 would make them compete equally with terminal goals. |

## Behavioral Spec — Deliberation Cycle with Planning

The `deliberateWithPlanner()` method implements an 8-phase cycle. Each call to `deliberate()` with a non-null planner executes exactly one cycle.

### AC #1 — Multi-step plan generation
- **Given** no active plan exists and goals are non-empty
- **When** `deliberate(state, goals, context)` is called with a planner
- **Then** a Plan with ≥3 steps is generated, stored in `context.activePlan`, subgoals registered with coherence engine, and a Decision for the first step is returned

### AC #3 — Postcondition violation triggers replanning
- **Given** an active plan exists and `context.lastActionResult` has `success: false`
- **When** `deliberate()` is called
- **Then** `evaluateOutcome()` detects the violation, `replan()` produces a new plan with `escalationCount + 1`, and the Decision references the replanned first step with `reason: 'postcondition-violated'`

### AC #3 — Postcondition success advances step
- **Given** an active plan at step index 0 and `context.lastActionResult` has `success: true`
- **When** `deliberate()` is called
- **Then** `currentStepIndex` is incremented to 1 and a Decision for step 1 is returned

### AC #4 — WaitState returns waiting decision
- **Given** an active plan with a non-null `waitState` whose `expiresAt` is in the future
- **When** `deliberate()` is called
- **Then** a Decision with `action.type === 'waiting'` is returned; the plan is not advanced

### AC #4 — Expired WaitState clears and continues
- **Given** an active plan with a `waitState` whose `expiresAt` is in the past
- **When** `deliberate()` is called
- **Then** `plan.waitState` is set to null and the current step is executed normally

### AC #4 — Step deadline breach triggers replanning
- **Given** an active plan whose current step has `deadline < Date.now()`
- **When** `deliberate()` is called
- **Then** `replan()` is called with reason `'deadline-exceeded'`

### AC #5 — Subgoal registration with derivation links
- **Given** a plan with steps carrying non-null `instrumentalGoalId`
- **When** `registerSubgoals()` is called
- **Then** `coherenceEngine.addGoal()` is called for each, with `derivedFrom: [terminalGoal.id]` and `type: 'instrumental'`

### AC #8 — Abandon after max escalations
- **Given** an active plan with `escalationCount >= maxEscalations` and a postcondition violation
- **When** `deliberate()` is called
- **Then** `shouldAbandon()` returns true, plan status is set to `'abandoned'`, and Decision has `action.type === 'plan-abandoned'`

### Plan Lifecycle State Machine

```
pending → active → completed
                 → suspended (budget exceeded / WaitState)
                 → failed (step failure beyond threshold)
                 → abandoned (shouldAbandon() returns true)
suspended → active (budget replenished / wait cleared)
```

## Acceptance Criteria
- Agent generates multi-step plans (≥3 steps) for non-trivial goals
- Plans have explicit preconditions and postconditions checked against the world model
- Agent detects plan failure (postcondition violation) and replans without external prompting
- Temporal reasoning demonstrated: agent sequences actions, waits when appropriate, respects deadlines
- Instrumental subgoals are registered with the Goal Coherence Engine and trace to terminal goals
- Planning respects the cognitive budget — no plan generation starves the experience stream
- Plans carry experiential basis references (conscious plan formation, not zombie planning)
- Integration test: agent given a goal with multiple prerequisite steps generates a plan, executes it, encounters an unexpected obstacle, replans, and achieves the goal via an alternative route
