## Decision

### D1 — Prospecting Selection Algorithm

**Context:** The prospecting subsystem must rank ~1000 asteroid candidates and select the top-10 for resource extraction missions.

**Options Considered:**
1. **Greedy ranking by resource-per-delta-v ratio** — Score each candidate independently as `(totalResourceMass × accessibilityScore) / deltaVCost`, sort descending, take top-N.
2. **Multi-objective optimization (Pareto front)** — Consider resource value, delta-v cost, and accessibility as separate objectives; return the Pareto-optimal set then pick from it.
3. **Constraint-satisfaction with linear programming** — Model as a knapsack problem with delta-v budget constraints and maximize total resource yield.

**Choice:** Option 1 — Greedy ranking by resource-per-delta-v ratio.

**Rationale:** Candidates are selected independently (no interaction effects — we're not constrained by a shared delta-v budget across all missions). For independent selections, greedy ranking by the composite score IS the optimal solution. Multi-objective and LP approaches add complexity without improving results for this independence structure.

**Consequences:** If future requirements add a shared budget constraint (e.g., fleet-level delta-v budget), the algorithm must be upgraded to Option 3. The current greedy approach is provably optimal only for independent candidate evaluation.

### D2 — Extraction Method Selection

**Context:** Different asteroid spectral types require different mining techniques. The system must autonomously select the appropriate extraction method.

**Options Considered:**
1. **Rule-based selection on spectral type and volatile ratio** — Hard-coded mapping: M-type→drill, high-volatile→ablation, else→mass-driver.
2. **Machine-learning classifier trained on simulation data** — Train a model to predict optimal extraction method from composition vectors.

**Choice:** Option 1 — Rule-based selection.

**Rationale:** The decision space is small (3 spectral categories × 2 volatile thresholds = a handful of rules). ML adds training data requirements and opacity without meaningful benefit. Rule-based selection is transparent, testable, and deterministic — critical for Level 4 autonomy where decisions must be auditable.

**Consequences:** Adding new spectral types or extraction methods requires manual rule updates. Acceptable given the slow pace of asteroid taxonomy changes.

### D3 — Processing Pipeline Architecture

**Context:** Raw ore must be converted through multiple stages into refined products. The pipeline design must achieve positive energy balance.

**Options Considered:**
1. **Sequential fixed-stage pipeline** — Five stages in order (sorting → thermal → chemical → volatile capture → QA), each with fixed yield and energy cost parameters.
2. **Configurable DAG of processing nodes** — Arbitrary graph of processing stages that can be rewired per ore composition.

**Choice:** Option 1 — Sequential fixed-stage pipeline.

**Rationale:** C-type asteroid processing follows a well-understood physical sequence. The fixed pipeline is simpler to validate for energy balance and purity guarantees. The `ProcessingPipelineConfig` type still allows stage parameter tuning without architectural change.

**Consequences:** Non-standard ore compositions (e.g., exotic spectral types) may require a different stage ordering. Mitigated by making stages configurable via `ProcessingPipelineConfig`.

### D4 — Depot Supply Gap Tracking

**Context:** The depot must detect and report supply gaps to verify the 30-day tolerance acceptance criterion.

**Options Considered:**
1. **Per-consumer-per-material gap tracking with open/close semantics** — Track active gaps as state machine (open when demand unmet, close when met), record duration.
2. **Daily snapshot comparison** — Compare daily demand vs dispensed, count gap days as simple counters.

**Choice:** Option 1 — Stateful gap tracking with open/close semantics.

**Rationale:** Provides precise gap start/end days and duration, enabling richer reporting and root-cause analysis. A snapshot approach would lose contiguity information (can't distinguish one 30-day gap from thirty 1-day gaps).

**Consequences:** Slightly more complex state management. Gaps still open at simulation end must be explicitly closed.

## Contracts

### C1 — Prospecting Module (`src/asteroid/prospecting.ts`)

**Exported Functions:**
- `totalResourceMass(comp: ResourceMap): number`
- `resourcePerDeltaV(candidate: AsteroidCandidate): number`
- `selectTopCandidates(catalog: AsteroidCandidate[], topN?: number): ProspectingResult`

**Preconditions:**
- `catalog` contains valid `AsteroidCandidate` objects with non-negative `deltaVCost` and `accessibilityScore ∈ [0, 1]`.
- `topN > 0` (defaults to 10).
- All mass values in `ResourceMap` are non-negative (kg).

**Postconditions:**
- `selectTopCandidates` returns `ProspectingResult` with `rankedCandidates.length === min(topN, catalog.length)`.
- Candidates are sorted in descending order of `resourcePerDeltaV` score.
- For independent candidate selection, the result is within 5% of the mathematically optimal selection.

**Invariants:**
- `resourcePerDeltaV` is a pure function; same input always produces same output.
- `totalResourceMass` sums all eight resource fields; never negative.

### C2 — Mining Module (`src/asteroid/mining.ts`)

**Exported Functions:**
- `recommendExtractionMethod(candidate: AsteroidCandidate): MiningOperation`
- `simulateMining(operation: MiningOperation, composition: ResourceMap, durationDays?: number, seed?: number): ExtractionResult`

**Preconditions:**
- `candidate` has a valid `spectralType` and non-zero `estimatedComposition`.
- `durationDays > 0` (defaults to `operation.operationalLifetime`).
- `seed` is a 32-bit integer for reproducible PRNG.

**Postconditions:**
- `recommendExtractionMethod` always returns `autonomyLevel: 'fully-autonomous'`.
- `simulateMining` returns `ExtractionResult` where `bulkOreMass + capturedVolatileMass ≤ theoreticalMaxMass`.
- `daysAutonomous ≥ 90` for any simulation with `durationDays ≥ 90` (no Earth commands issued).
- `(bulkOreMass + capturedVolatileMass) / theoreticalMaxMass ≥ 0.80`.

**Invariants:**
- Same `seed` produces identical `ExtractionResult` (deterministic simulation).
- Fault events are resolved autonomously (logged but never require external intervention).

### C3 — Processing Module (`src/asteroid/processing.ts`)

**Exported Functions:**
- `defaultCTypePipeline(): ProcessingPipelineConfig`
- `cumulativeYield(stages: ProcessingStage[]): number`
- `totalEnergyCostPerKg(stages: ProcessingStage[]): number`
- `processOre(oreComposition: ResourceMap, oreMassKg: number, pipeline: ProcessingPipelineConfig, depotId?: string): ProcessingOutput`

**Preconditions:**
- `oreMassKg > 0`.
- `oreComposition` has at least one non-zero resource field.
- `pipeline.stages.length ≥ 1`, each stage has `yieldFraction ∈ (0, 1]` and `energyCostPerKg ≥ 0`.

**Postconditions:**
- Metal products have `purity ≥ 0.95`.
- Water/LOX products have `purity ≥ 0.99`.
- `energyBalance ≥ 1.2` for C-type compositions with solar energy source (20% surplus).
- All products reference `depotId` as their `destinationDepot`.

**Invariants:**
- `cumulativeYield` is the product of all stage `yieldFraction` values; always in `(0, 1]`.
- `processOre` is a pure function; no side effects.

### C4 — Depot Module (`src/asteroid/depot.ts`)

**Exported Functions:**
- `createDepot(depotId: string, location: OrbitalPosition, consumers: ConsumerEndpoint[], capacityPerMaterial?: number): ResourceDepot`
- `addToInventory(depot: ResourceDepot, products: ProcessedProduct[]): void`
- `dispenseMaterial(depot: ResourceDepot, material: MaterialType, requestedKg: number): number`
- `simulateDepot(depot: ResourceDepot, totalDays: number, dailySupplyFn: (day: number) => ProcessedProduct[]): DepotSimulationResult`

**Preconditions:**
- `capacityPerMaterial > 0` (defaults to 1,000,000 kg).
- `totalDays > 0`.
- `dailySupplyFn` is a pure function of `day` (injectable for testability).
- `requestedKg ≥ 0`.

**Postconditions:**
- `createDepot` initializes all material inventories to 0.
- `addToInventory` never exceeds capacity; silently caps.
- `addToInventory` ignores `slag` products.
- `dispenseMaterial` returns `min(available, requestedKg)` and decrements inventory.
- `simulateDepot` closes all open gaps at simulation end.
- `allConsumersServed === true` iff `maxGapDays ≤ 30`.

**Invariants:**
- Inventory values are never negative.
- Supply gaps are contiguous: a gap's `durationDays === endDay - startDay + 1` (or `endDay - startDay` depending on open/close semantics; see implementation).

## Behavioral Spec

### BS1 — Prospecting Selection Flow

**Given** a synthetic catalog of 1000 asteroid candidates with varying compositions, delta-v costs, and accessibility scores
**When** `selectTopCandidates(catalog, 10)` is invoked
**Then** the returned `rankedCandidates` contains exactly 10 entries, sorted in descending order of `(totalResourceMass × accessibilityScore) / deltaVCost`, and the aggregate score is within 5% of the brute-force optimal top-10

### BS2 — Autonomous Mining Simulation Flow

**Given** a C-type asteroid candidate and a mining operation configured via `recommendExtractionMethod`
**When** `simulateMining` runs for 365 simulated days
**Then:**
- Each day, the agent either extracts ore (at 85–100% efficiency) or handles a fault autonomously (1-day pause, ~2% probability)
- Total extracted mass is ≥80% of `extractionRate × days`
- All operational days count as autonomous (no Earth commands)
- `daysAutonomous ≥ 90`
- Operation log contains `MINING_START`, periodic `STATUS_REPORT` (every 30 days), any `FAULT_DETECTED`, and `MINING_COMPLETE`

### BS3 — Processing Pipeline Flow

**Given** raw ore from a C-type asteroid and the default processing pipeline
**When** `processOre` is invoked with the ore composition and mass
**Then:**
1. Ore passes through 5 sequential stages: sorting → thermal → chemical → volatile capture → QA
2. Cumulative yield = product of all stage yields (~0.81)
3. Metals (iron, nickel, PGM) are produced at ≥95% purity
4. Water from electrolysis yields LOX (88.9% by mass) and LH2 (11.1%); stored at ≥99.5% purity
5. Half of water is stored as water (≥99.9% purity), half electrolyzed
6. Energy balance = (H2 fuel cell energy + solar energy) / processing energy cost ≥ 1.2
7. Silicates and processing losses become slag

### BS4 — Depot Supply Simulation Flow

**Given** a depot with three consumer types (consciousness-platform, manufacturing, propellant-depot) and a `dailySupplyFn` providing 110% of aggregate demand
**When** `simulateDepot` runs for 365 simulated days
**Then:**
1. Each day: receive supply → add to inventory → fulfill each consumer's demand
2. If dispensed < 99% of demand for a consumer+material pair, a supply gap opens
3. When demand is met again, the gap closes with recorded start/end/duration
4. At simulation end, any open gaps are closed
5. `maxGapDays ≤ 30` and `allConsumersServed === true`

## Acceptance Criteria
- Asteroid prospecting algorithm selects top-10 candidates by resource-per-delta-v ratio within 5% of optimal given synthetic catalog of 1000 candidates
- Simulated mining agent achieves ≥80% of theoretical extraction rate with no Earth-based commands for ≥90-day periods
- Processing pipeline converts C-type ore into metals (≥95% purity) and water/LOX (≥99% purity) with positive energy balance (≥20% surplus)
- Depot simulation maintains continuous supply to three consumer types for ≥1 simulated year with zero supply gaps exceeding 30 days
- No dependency on Earth-sourced raw materials for ongoing operations
- All subsystems operate at Level 4 autonomy (no real-time Earth control required)
