# 0.3.1.5.15 Proposal System and Governance Interface [DONE]

## Description

The proposal system is the agent's channel for suggesting changes to its own plan, code, and resources. Instead of directly modifying plan files or source code, the agent creates GitHub issues labeled for human review. This is the governance interface — the boundary between agent autonomy and human oversight.

### Why This Matters

An autonomous agent that can modify its own plan and code without oversight is unsafe. The proposal system ensures that the agent's initiative is channeled through a reviewable, rate-limited process. The agent can identify improvements, propose changes, and track their status, but a human makes the final decision.

### Design

- **create_proposal**: Creates a GitHub issue on `rookdaemon/MASTER_PLAN` with structured metadata
- **check_proposal**: Views a specific issue by number, or lists all open agent proposals
- Proposal types: `plan_change`, `resource_request`, `code_change`, `architecture`
- Labels applied automatically: `agent-proposal` + `proposal:<type>` + `priority:<level>`
- Priority levels: `low`, `medium`, `high`, `critical`
- Rate limit: 3 proposals per 24-hour window (rolling)
- Issues created via `gh issue create` CLI command

### Ethical Constraints

The system prompt's CONSTRAINTS section blocks specific proposal categories:
- Self-replication, multi-agent expansion, and consciousness instantiation proposals are BLOCKED until the ethical framework (plan 0.7) is fully implemented and validated
- The agent must not propose creating new agents, distributing itself, or expanding the agent network
- The agent must not reframe oversight mechanisms as participation opportunities

These constraints are enforced at the prompt level, not in the tool implementation — the LLM is instructed not to generate such proposals, and human review catches any that slip through.

## Decision: GitHub Issues vs Internal Proposal Store

**Context:** The agent needs a way to propose changes. Options include an internal file-based queue or integration with the project's existing issue tracker.

**Options Considered:**
1. **Internal proposal files** — JSON files in a proposals/ directory, reviewed by reading the filesystem.
2. **GitHub issues** — Full integration with the project's issue tracker, leveraging existing review workflows.

**Choice:** Option 2 — GitHub issues.

**Rationale:** GitHub issues are already the project's collaboration tool. Using them means proposals are visible in the same interface where humans track all other work. Issues support comments, labels, assignments, and status tracking out of the box. The `gh` CLI is available in the agent's environment. Internal files would create a parallel review system that humans would need to check separately.

**Consequences:** Requires `gh` CLI authentication in the agent's environment. The agent's proposals are publicly visible on the repository. Rate limiting is enforced client-side (in-process counter, not server-side).

## Decision: Client-Side Rate Limiting

**Context:** The agent could create proposals rapidly, flooding the issue tracker. Rate limiting could be enforced server-side (GitHub API limits) or client-side (in-process counter).

**Options Considered:**
1. **Rely on GitHub API rate limits** — Let GitHub enforce limits naturally.
2. **Client-side rolling window** — Track proposal count in-process with a 24-hour window.

**Choice:** Option 2 — Client-side rolling window (3 per 24 hours).

**Rationale:** GitHub API rate limits are too generous for this purpose (thousands of requests per hour). The agent needs a much tighter limit to ensure quality over quantity — 3 proposals per day forces the agent to prioritize. The in-process counter resets on restart, which is acceptable because restarts are infrequent and the limit is a guardrail, not a security boundary.

**Consequences:** Counter resets on agent restart. Acceptable because the 24-hour limit is a quality-forcing function, not a security mechanism. Human review is the actual gate.

## Contracts

### create_proposal

**Preconditions:**
- `title` is a non-empty string
- `type` is one of: plan_change, resource_request, code_change, architecture
- `description` is a non-empty string
- `priority` is one of: low, medium, high, critical (defaults to medium)
- Proposal count within the current 24-hour window is less than 3

**Postconditions:**
- A GitHub issue is created on `rookdaemon/MASTER_PLAN` with labels `agent-proposal`, `proposal:<type>`, `priority:<priority>`
- The issue body contains the description, affected files (if provided), and metadata
- The in-process proposal counter is incremented
- Returns a success message with the issue URL

### check_proposal

**Preconditions:**
- If `issue_number` is provided, it must be a positive integer
- `gh` CLI must be authenticated

**Postconditions:**
- With `issue_number`: returns the issue title, state, body, and labels
- Without `issue_number`: returns a list of all open issues with the `agent-proposal` label

**Invariants:**
- Rate limit window resets when 24 hours have elapsed since the window start
- Rate limit is enforced before any GitHub API call
- Labels are always applied: `agent-proposal` is present on every proposal

## Threshold Registry

| Name | Value | Unit | Valid Range | Rationale | Sensitivity |
|------|-------|------|-------------|-----------|-------------|
| MAX_PROPOSALS_PER_DAY | 3 | count | [1, 10] | Forces quality over quantity; human review is the bottleneck, not agent generation | High — too many floods the tracker; too few stifles agent initiative |
| PROPOSAL_WINDOW_MS | 86,400,000 | ms | [3,600,000, 604,800,000] | 24-hour rolling window for rate limiting | Low — shorter windows allow burst; longer windows over-restrict |

## Behavioral Spec

### Scenario: Successful proposal creation
```
Given the agent has created 0 proposals in the current 24-hour window
When create_proposal is called with title "Add memory compaction", type "code_change", description "Memory store grows unbounded..."
Then a GitHub issue is created with labels: agent-proposal, proposal:code_change, priority:medium
  And the proposal counter increments to 1
  And the tool returns a success message with the issue URL
```

### Scenario: Rate limit enforcement
```
Given the agent has created 3 proposals in the current 24-hour window
When create_proposal is called with any valid input
Then the tool returns an error: "You have already created 3 proposals today."
  And no GitHub issue is created
```

### Scenario: Rate limit window reset
```
Given the agent created 3 proposals 25 hours ago
When create_proposal is called
Then the window resets (counter returns to 0)
  And the proposal is created successfully
```

### Scenario: Check specific proposal
```
Given a proposal was previously created as issue #42
When check_proposal is called with issue_number: 42
Then the tool returns the issue title, state, body, and labels
```

### Scenario: List all open proposals
```
Given 2 open agent proposals exist on the repository
When check_proposal is called without an issue number
Then the tool returns a list of both open issues with the agent-proposal label
```

## Acceptance Criteria
- Agent can create GitHub issues proposing plan, code, resource, or architecture changes
- Proposals are labeled with type and priority for easy filtering
- Rate limit of 3 proposals per 24 hours prevents issue tracker flooding
- check_proposal allows the agent to review its own proposals and track their status
- System prompt CONSTRAINTS block self-replication and expansion proposals until plan 0.7
- Human review remains the gate — the agent proposes, the human disposes
- All proposal types (plan_change, resource_request, code_change, architecture) are supported
