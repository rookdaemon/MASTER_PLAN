# 0.2.2.1.3 Neural Simulation [DONE]
## Description

Build computational infrastructure capable of simulating a complete biological brain at the required fidelity in real-time or faster.

### Key challenges
- Computational scale: a molecular-level simulation of a human brain may require exascale or beyond
- Real-time constraint: subjective experience likely requires temporal dynamics to proceed at biological speed or faster
- Numerical stability over extended simulation runs (years, decades)
- Energy efficiency: the biological brain operates on ~20W; the simulation must be practically sustainable
- Integration with enduring substrates (S1) for long-term operation

### Deliverable

An architecture document (`docs/neural-simulation/ARCHITECTURE.md`) specifying the simulation engine design, compute requirements, and integration points with scan data (0.2.2.1.2) and enduring substrates (0.2.1).

### Artifact Anticipation
- **Decision section:** Simulation algorithm selection among alternatives (compartmental, spiking, molecular, hybrid) — justified against L2+ fidelity from 0.2.2.1.1
- **Contracts section:** Scan data ingestion interface consuming `BrainScanDataset` from 0.2.2.1.2; HAL interface for enduring substrate migration from 0.2.1; output contract for validation by 0.2.2.1.4
- **Threshold Registry:** FLOPS/memory/bandwidth estimates per fidelity level, time-step constants, error accumulation bounds, energy budget thresholds, validation tolerances
- **Behavioral Spec:** Scan data ingestion pipeline (multi-step process), biological validation benchmark execution, scaling milestone progression

## Decision: Simulation Algorithm

### Context
The upstream fidelity spec (0.2.2.1.1) sets L_min = L2+ (cellular-level, augmented with stochastic noise, neuromodulation, and astrocyte calcium signaling). The simulation must model ~86 billion neurons with ~20–50 state dimensions each, at ≤0.1 ms time steps, running at ≥1× real-time. We must select a simulation algorithm that satisfies L2+ fidelity while remaining computationally tractable.

### Options Considered

**O1: Point-neuron spiking models (LIF / Izhikevich)**
- Pros: Highly parallelizable (~10^18 FLOPS for human brain real-time); well-understood; mature GPU implementations (NEST, Brian2, GeNN). Each neuron is a single ODE or map.
- Cons: Cannot capture dendritic computation, back-propagating action potentials, or NMDA spikes — all required by L2+. L2+ mandates "multi-compartment (Hodgkin-Huxley-class), minimum 5 compartments per neuron." No sub-neuronal integration.
- **Verdict: Insufficient.** Fails the L2+ multi-compartment requirement.

**O2: Multi-compartment Hodgkin-Huxley (HH) models**
- Pros: Captures dendritic computation, active conductances, cable equation dynamics. Each neuron decomposed into ~5–500 compartments with per-compartment ion channel distributions. Reproduces action potential waveforms, f-I curves, dendritic nonlinearities. Well-validated in NEURON, CoreNeuron, Arbor simulators. Directly satisfies L2+ specification (multi-compartment, conductance-based synapses, stochastic noise injection).
- Cons: ~10^21 FLOPS for human brain real-time (1 ZettaFLOPS) — 500× beyond current supercomputers. Memory ~1 PB. Requires massive parallelization or neuromorphic hardware.
- **Verdict: Sufficient.** Directly satisfies all L2+ requirements. Computational cost is the primary challenge.

**O3: Molecular dynamics at synapses (Gillespie/tau-leaping)**
- Pros: Captures stochastic molecular signaling (vesicle release, receptor binding, second messengers) with high fidelity. Would satisfy L3.
- Cons: ~10^34 FLOPS for human brain — exceeds any foreseeable hardware by >10^12×. L2+ only requires stochastic *noise approximation* of molecular effects, not full molecular simulation.
- **Verdict: Unnecessarily fine.** L2+ explicitly states molecular effects are well-approximated by noise models at the cellular level.

**O4: Hybrid multi-compartment HH + event-driven optimization**
- Pros: Uses full HH multi-compartment for active neurons but skips quiescent compartments via event-driven scheduling. At ~5 Hz average firing rate, most compartments are inactive >99% of the time. Reduces effective compute by 10–100× vs. time-driven HH. Still captures all L2+ dynamics when neurons are active. Compatible with neuromorphic hardware (spike-driven computation is native).
- Cons: More complex scheduling logic; worst-case (all neurons firing maximally) reverts to O2 cost. Event-driven correctness requires careful spike-ordering guarantees.
- **Verdict: Optimal.** Full L2+ fidelity with significant compute reduction through activity-dependent optimization.

### Choice
**O4: Multi-compartment Hodgkin-Huxley with event-driven optimization**, augmented per L2+ with:
1. Stochastic noise injection (Langevin channel-noise diffusion approximation)
2. Neuromodulatory gain modulation (DA, 5-HT, ACh, NE at region-level tonic + phasic)
3. Astrocyte calcium signaling model (tripartite synapse, calcium waves, gliotransmitter release)
4. Multi-rate time-stepping (fast: 0.025 ms for V_m/gating; medium: 0.1 ms for synapses/calcium; slow: 1–100 ms for plasticity/glia)
5. Semi-implicit Euler integration for stiff HH equations with periodic conservation correction

### Rationale
- **Fidelity-complete:** Multi-compartment HH directly satisfies every L2+ requirement from 0.2.2.1.1 (≥5 compartments/neuron, conductance-based synapses, stochastic noise, glial model, neuromodulation).
- **Compute-feasible:** Event-driven optimization reduces effective FLOPS from ~10^21 to ~10^19–10^20, bringing the target within reach of neuromorphic-conventional hybrid hardware.
- **Biologically validated:** HH compartmental models are the gold standard for reproducing neural dynamics — oscillatory rhythms, bursting patterns, dendritic plateau potentials, STDP.
- **Hardware-compatible:** Event-driven spike computation maps naturally to neuromorphic architectures (Loihi 2, SpiNNaker 2, BrainScaleS-2).

### Consequences
- Scanning (0.2.2.1.2) output must include neuron morphology for compartmentalization, ion channel distributions, and synapse locations — already specified in `BrainScanDataset`.
- Hardware must support both time-driven (conservation corrections) and event-driven (spike-triggered) computation modes.
- Validation (0.2.2.1.4) must verify that event-driven optimization does not degrade causal transition fidelity below ε_transition.
- Energy budget is dominated by spike routing interconnect at scale; neuromorphic hardware is essential.

## Decision: Hardware Target

### Context
The simulation requires ~10^19–10^21 sustained FLOPS depending on optimization effectiveness. Current best supercomputers achieve ~2×10^18 FLOPS. We must select a hardware architecture that can reach the required compute within practical energy and cost constraints, and that has a migration path to enduring substrates (0.2.1).

### Options Considered

**O1: Conventional GPU cluster**
- Pros: Programmable, available today, well-understood. GPU-accelerated HH simulation demonstrated (CoreNeuron, Arbor).
- Cons: ~500 MW–1 GW power for 1 ZettaFLOPS. 500× beyond current capability. No path to <10 MW target.
- **Verdict: Insufficient alone.** Energy consumption is unsustainable.

**O2: Pure neuromorphic hardware**
- Pros: 100–1000× energy efficiency for spiking neuron models. Native spike routing. Analog computation of cable equations approaches thermodynamic limits.
- Cons: Limited programmability for plasticity rules, glial models, and molecular-level augmentations. Current chips (Loihi 2: ~1M neurons) are 5 orders of magnitude short of human scale. No existing chip implements multi-compartment HH natively.
- **Verdict: Insufficient alone.** Cannot handle non-neural computation (glia, plasticity, validation).

**O3: Neuromorphic-conventional hybrid**
- Pros: Neuromorphic chips handle neuron/synapse dynamics (~10^6 neurons per chip, ~10^6 chips needed); conventional GPU/CPU handles glia, plasticity, molecular augmentations, orchestration. Combines energy efficiency with programmability. Natural migration path: Phase 1 (datacenter), Phase 2 (radiation-hardened neuromorphic from S1.1), Phase 3 (fully integrated enduring substrate).
- Cons: Complex heterogeneous programming model. Requires custom neuromorphic chip design for multi-compartment HH.
- **Verdict: Optimal.** Balances compute, energy, programmability, and migration path.

### Choice
**O3: Neuromorphic-conventional hybrid** with Hardware Abstraction Layer (HAL) for substrate migration.

### Rationale
- Neuromorphic component provides 100–1000× energy reduction for the dominant workload (neuron/synapse dynamics).
- Conventional component provides programmability for non-stereotyped computation.
- HAL isolates simulation logic from hardware, enabling migration to enduring substrates without rewriting the simulation engine.

### Consequences
- Requires custom neuromorphic chip supporting multi-compartment HH (not just LIF/Izhikevich).
- HAL interface must abstract neuron allocation, spike routing, state checkpointing, and health monitoring.
- Energy budget target: <10 MW Phase 1, <1 MW Phase 2 (with mature neuromorphic hardware).

## Decision: Time-Stepping Scheme

### Context
The HH system is stiff: fast Na⁺ activation (~0.1 ms) vs. slow K⁺ inactivation (~10 ms) vs. very slow plasticity (~seconds–hours). A single time step is either wastefully small for slow variables or unstably large for fast variables. We must select an integration scheme that is stable, efficient, and provides bounded error accumulation over ≥1 simulated year.

### Options Considered

**O1: Fixed-step explicit Euler**
- Pros: Simple. Trivially parallelizable.
- Cons: Requires dt ≤ 0.01 ms for HH stability (fast Na⁺ gating). Wasteful for slow variables. No error control.
- **Verdict: Insufficient.** Unstable or wasteful; no error bounds for long runs.

**O2: Adaptive Runge-Kutta (RK45)**
- Pros: Automatic error control. Well-understood. Adapts step size to local dynamics.
- Cons: Explicit method — still limited by stiffness. Adaptive steps create synchronization difficulties across parallel neurons. Variable step sizes break the global clock needed for spike ordering.
- **Verdict: Impractical.** Synchronization issues in massively parallel simulation.

**O3: Multi-rate semi-implicit Euler with periodic conservation correction**
- Pros: Implicit treatment of fast linear terms (Na⁺/K⁺ gating) allows larger dt for fast variables (0.025 ms vs. 0.01 ms). Multiple rate tiers match different timescales. Fixed step sizes maintain global clock synchronization. Periodic conservation correction (every simulated second) prevents floating-point drift. Error accumulation formally bounded.
- Cons: Semi-implicit requires solving a small linear system per compartment per fast step (tridiagonal for cable equation — O(n) via Hines algorithm). Slightly more complex than explicit.
- **Verdict: Optimal.** Balances stability, efficiency, parallelism, and formal error bounds.

### Choice
**O3: Multi-rate semi-implicit Euler** with 4 rate tiers and periodic conservation correction.

### Rationale
- Semi-implicit treatment of the stiff HH system allows 2.5× larger fast time steps vs. explicit Euler.
- Multi-rate stepping avoids wasting compute on slow variables at fast rates.
- Fixed step sizes maintain deterministic spike ordering across parallel neurons.
- Hines algorithm solves the cable equation in O(n) per neuron — no additional asymptotic overhead.
- Periodic conservation correction bounds error accumulation to <1 mV over 1 simulated year.

### Consequences
- Four rate tiers: dt_fast = 0.025 ms, dt_medium = 0.1 ms, dt_slow = 1.0 ms, dt_structural = 100 ms.
- Global synchronization barrier every dt_slow (1 ms) for spike delivery consistency.
- Conservation correction every 1000 ms: total ion concentrations, ATP budget, charge balance.
- Error bound: local truncation error O(dt²) = 6.25×10⁻⁷ mV per fast step; accumulated <1 mV/year with correction.

## Contracts: Scan Data Ingestion Interface

### Preconditions
- `BrainScanDataset` from 0.2.2.1.2 is available and passes `validateBrainScanDataset()` with `valid: true`.
- `BrainScanDataset.fidelity_level` is `"cellular"` or finer (compatible with L2+ simulation).
- `BrainScanDataset.schema_version` is compatible with the simulation engine's supported schema versions.
- HDF5 file is accessible and passes checksum verification.

### Postconditions
The ingestion pipeline produces an `IngestionResult` such that:
1. `neuron_count` matches `BrainScanDataset.subject_metadata.neuron_count_estimate` within ±5%.
2. `synapse_count` matches `BrainScanDataset.subject_metadata.synapse_count_estimate` within ±10%.
3. Every neuron has ≥ `compartments_per_neuron_min` (5) compartments with valid morphology trees.
4. Every synapse has a valid pre/post neuron pair, receptor type, and initial weight from scan data.
5. Dynamic state is initialized from `DynamicStateSnapshot` where available; resting-state defaults applied elsewhere.
6. Spatial decomposition assigns all neurons to exactly one compute partition.
7. Validation checkpoint passes: 10 s warm-up produces biologically plausible spontaneous activity (firing rates 0.1–50 Hz per neuron type, oscillatory power in expected bands).
8. `validation_passed === true` in the returned `IngestionResult`.

### Invariants
- No neuron exists without at least one compartment (soma).
- No synapse references a nonexistent neuron or compartment.
- Total neuron count across all spatial partitions equals `neuron_count`.
- Ion channel parameters are within physiologically plausible ranges (e.g., Na⁺ conductance 0–200 mS/cm²).
- Membrane potential initialization is within [-90, +40] mV for all compartments.

## Contracts: Hardware Abstraction Layer (HAL) Interface

### Preconditions
- Hardware platform is operational and passes `health_check()`.
- Sufficient capacity exists for the requested neuron allocation.

### Postconditions
1. `allocate_neurons()` returns a `NodeAllocation` mapping each neuron to a hardware node, with all neurons allocated.
2. `send_spikes()` delivers spike events to target neurons within the axonal delay budget (≤1 simulation time step after delay expiry).
3. `receive_spikes()` returns all spikes targeted at the given node since last call, in temporal order.
4. `checkpoint_state()` produces a `StateSnapshot` that, when passed to `restore_state()`, produces a simulation state indistinguishable from the original (bitwise identical for deterministic components).
5. `health_check()` reports hardware status including node failures, temperature, power consumption, and error rates.

### Invariants
- No spike is lost in transit (guaranteed delivery).
- Spike ordering is preserved within each source neuron (spikes from neuron N arrive in temporal order).
- `checkpoint_state()` followed by `restore_state()` is idempotent — the simulation produces identical future output.
- The HAL interface is identical across all hardware targets (datacenter, radiation-hardened, enduring substrate).

## Contracts: Validation Output Interface

### Preconditions
- Simulation has completed ≥10 s of simulated time after ingestion warm-up.
- Biological reference data is available for all specified benchmarks.

### Postconditions
1. `ValidationSuite` contains results for all registered benchmarks (single-neuron, circuit, whole-brain).
2. Each `ValidationResult` includes `simulated_value`, `deviation_from_reference`, and `passed` boolean.
3. `ValidationSuite.overall_pass` is `true` iff all individual benchmarks pass within their specified tolerances.
4. Results are serializable and timestamped for reproducibility.

### Invariants
- Benchmark definitions are immutable once registered (reference values and tolerances do not change between runs).
- A passing `ValidationSuite` at organism scale N is a prerequisite for advancing to scale N+1.

## Behavioral Spec

### Scenario 1: Scan data ingestion pipeline (inline Given/When/Then)

**Given** a valid `BrainScanDataset` from 0.2.2.1.2 with `fidelity_level = "cellular"`, containing structural data (neuron morphology, synapse map, receptor distributions, glial topology) and dynamic state (activity recording, neurotransmitter map, synaptic weights)

**When** the ingestion pipeline executes:
1. Validate scan: check schema version compatibility, verify checksums, confirm fidelity ≥ L2+
2. Extract morphology: convert segmented neurons into multi-compartment trees (≥5 compartments per neuron via Rall equivalent cylinder)
3. Assign ion channels: map channel distributions from scan receptor data + published atlases to per-compartment `ActiveConductance` parameters
4. Initialize synapses: create `Synapse` objects from connectivity graph with weights from `SynapticWeightMap`, receptor types from scan
5. Initialize glia: assign astrocyte territories from scan data, initialize calcium and metabolic state
6. Set dynamic state: initialize V_m and gating variables from `DynamicStateSnapshot` where available; resting-state defaults (-65 mV, steady-state gating) elsewhere
7. Spatial decomposition: partition neurons into ~1000 brain regions → ~10^6 cortical columns, each assigned to a compute node
8. Validation checkpoint: run 10 s simulated time, verify spontaneous activity is biologically plausible

**Then**
- `IngestionResult.validation_passed === true`
- `IngestionResult.neuron_count` within ±5% of `BrainScanDataset.subject_metadata.neuron_count_estimate`
- `IngestionResult.synapse_count` within ±10% of scan estimate
- All compartments have V_m ∈ [-90, +40] mV
- All partitions are non-empty
- Spontaneous firing rates ∈ [0.1, 50] Hz per neuron type

### Scenario 2: Biological validation benchmark execution (inline Given/When/Then)

**Given** a simulation initialized from scan data that has passed the ingestion validation checkpoint, and a `ValidationSuite` containing benchmarks at single-neuron (AP waveform, f-I curve), circuit (oscillatory rhythms, stimulus-evoked responses), and whole-brain (resting-state FC, sleep stages) scales

**When** each benchmark is executed by:
1. Configuring simulation input (current injection for single-neuron, stimulus protocol for circuit, free-running for resting-state)
2. Running simulation for the benchmark-specified duration
3. Extracting the benchmark metric from simulation output
4. Comparing against biological reference data within the specified tolerance

**Then**
- Single-neuron: AP peak amplitude and timing within 5% of experimental reference
- Circuit: oscillatory band power (alpha, beta, gamma) within 20% of EEG/MEG reference
- Whole-brain: resting-state FC correlation ≥ 0.8 with empirical fMRI data
- `ValidationSuite.overall_pass === true` only if all benchmarks pass

### Scenario 3: Scaling milestone progression (inline Given/When/Then)

**Given** the simulation engine is deployed at organism scale N (C. elegans → Drosophila → Mouse → Human) and `ValidationSuite` has passed at scale N

**When** scaling to N+1:
1. Load `BrainScanDataset` for organism N+1
2. Execute ingestion pipeline
3. Measure: compute time vs. neuron count, communication overhead, memory usage, numerical stability
4. Run `ValidationSuite` at scale N+1
5. Verify real-time execution (simulation_speed_ratio ≥ 1.0)

**Then**
- Compute time scales approximately linearly with neuron count
- Communication overhead remains bounded by axonal delay buffering
- Memory scales linearly with compartment count
- Error bounds (accumulated error <1 mV/year) hold at scale N+1
- `ScalingMilestone.validation_suite_passed === true`
- `ScalingMilestone.simulation_speed_ratio ≥ 1.0`
- Scale N+1 may only be attempted after scale N passes all criteria
