# 0.3.2.3 Knowledge Preservation Systems [DONE]

## Description

A2.3 of the Master Plan. Ensure continuity of knowledge across generations of artificial minds. As conscious entities are created, evolve, and potentially end, the accumulated knowledge, experience, and cultural memory of the civilization must persist. This system is the long-term memory of a synthetic civilization — analogous to libraries, oral traditions, and cultural institutions for biological civilizations, but engineered for indefinite durability and machine-accessible retrieval.

## Decision — Storage & Addressing Strategy

### Context
The system must store knowledge items durably, immutably, and with deduplication across a distributed network of nodes. Items must be retrievable by identity and by semantic/metadata queries. The addressing scheme determines how items are identified, deduplicated, and verified.

### Options Considered
1. **UUID-based addressing** — Each item gets a random UUID at creation. Simple but offers no deduplication, no tamper detection, and no content-verification guarantee.
2. **Content-addressed hashing (SHA-256)** — The item's identity is the cryptographic hash of its content. Identical content always yields the same address; any mutation produces a new address. Enables deduplication, tamper detection, and immutability by construction.
3. **Hybrid (UUID + content hash)** — UUID for human-readable routing, content hash for verification. Added complexity with marginal benefit over pure content-addressing.

### Choice
**Option 2 — Content-addressed hashing (SHA-256)**

### Rationale
- Immutability is enforced structurally: changing content changes the hash, so overwrites are impossible without detection.
- Deduplication is automatic: identical knowledge produces identical IDs.
- Tamper evidence is built into the addressing scheme itself.
- Aligns with proven systems (IPFS, Git) for distributed content-addressed storage.
- SHA-256 provides sufficient collision resistance for the foreseeable corpus scale (up to 10^12 items).

### Consequences
- Mutations produce new items linked via `versionChain` rather than in-place updates — requires chain traversal for history.
- Content-addressing means metadata changes (e.g., adding a signature) would change the hash. Mitigation: the `id` is computed from the `content` field only, not from metadata/provenance.
- Requires a canonical serialization to ensure deterministic hashing across nodes.

## Decision — Ontology Evolution Model

### Context
The civilization's conceptual vocabulary must evolve over time. Deprecated concepts must remain accessible. The ontology itself is part of the knowledge corpus.

### Options Considered
1. **Static ontology with versioned snapshots** — Ontology is frozen periodically; changes require a full new snapshot. Simple but rigid and space-inefficient.
2. **Append-only diff-based ontology** — Changes are expressed as versioned diffs (add nodes, deprecate nodes, add edges). Diffs are applied in sequence; deprecated nodes retain `deprecatedAt` and `mappedTo` pointers. The ontology is self-referential (stored as knowledge items).

### Choice
**Option 2 — Append-only diff-based ontology**

### Rationale
- Preserves full history of conceptual evolution — no information is lost.
- Deprecation-with-mapping ensures old references remain resolvable.
- Diff-based changes are lightweight and composable.
- Self-storage as knowledge items means the ontology benefits from the same replication, provenance, and audit trail as all other knowledge.

### Consequences
- Ontology traversal must handle deprecated nodes (follow `mappedTo` pointers when encountering deprecated concepts).
- Diff application order matters; diffs must carry timestamps and be applied sequentially.
- Quorum consensus is required for diff application in production (not enforced in the in-memory implementation, which captures local application logic only).

## Contracts

### ContentAddressedStorageLayer (CASL)

**Preconditions:**
- `store(item)`: `item.id` must equal `hashContent(item.content)`. Item must be a well-formed `KnowledgeItem`.
- `get(id)`: `id` is a string (ContentAddressedHash).
- `registerRemoteCopy(itemId, remoteNodeId)`: `itemId` must already exist in the local store.

**Postconditions:**
- `store(item)`: If item hash is new, item is persisted and retrievable via `get(item.id)`. If identical item exists, no-op. If hash matches but content differs, a `TamperAnomalyRecord` is logged and an error is thrown.
- `get(id)`: Returns the stored `KnowledgeItem` or `null`. Never mutates state.
- `history(id)`: Returns an ordered array (newest-first) of items linked by `versionChain`, terminating at genesis (versionChain === null).
- `merkleManifest()`: Returns a `MerkleManifest` with sorted item IDs and a deterministic Merkle root.

**Invariants:**
- Items are immutable once stored — no field of a stored item may change.
- `itemCount()` is monotonically non-decreasing (append-only).
- For any stored item, `get(item.id)` returns a value structurally equal to the originally stored item.
- `tamperAnomalies()` is append-only — anomaly records are never removed.

### QueryRetrievalInterface (QRI)

**Preconditions:**
- `search(filter)`: `filter` is a valid `MetadataFilter` (all fields optional).
- `auditTrail(id)`: `id` is a string (ContentAddressedHash).

**Postconditions:**
- `get(id)`: Delegates to CASL; returns `KnowledgeItem | null`.
- `search(filter)`: Returns all items matching ALL specified filter criteria. Superseded items excluded by default unless `includeSuperseded: true`.
- `history(id)`: Delegates to CASL; returns version chain newest-first.
- `auditTrail(id)`: Returns a `ProvenanceTree` with recursively resolved `sourceIds`. Handles diamond-shaped graphs (same source reachable via multiple paths). Returns `null` for unknown items. Cycle-safe via ancestor tracking.

**Invariants:**
- QRI is read-only — it never mutates the underlying CASL.
- `search({})` returns all non-superseded items.
- `search({ includeSuperseded: true })` returns all items regardless of supersession status.

### LivingOntologyEngine (LOE)

**Preconditions:**
- `addNode(node)`: `node.nodeId` must not already exist in the engine.
- `addEdge(edge)`: Both `edge.from` and `edge.to` must be existing node IDs.
- `deprecateNode(nodeId, ...)`: `nodeId` must exist in the engine.
- `applyDiff(diff)`: All nodes referenced in `nodesToDeprecate` must exist. Nodes in `nodesToAdd` must not already exist. Edges in `edgesToAdd` must reference nodes that exist (including newly added ones from the same diff).

**Postconditions:**
- `addNode(node)`: Node is retrievable via `getNode(node.nodeId)`.
- `deprecateNode(nodeId, ts, mappedTo)`: Node's `deprecatedAt` is set to `ts` and `mappedTo` to the provided value. Node remains in `allNodes()`.
- `applyDiff(diff)`: All specified additions, deprecations, and edges are applied atomically. Diff is recorded in `appliedDiffs()`.
- `related(nodeId, edgeType)`: Returns target nodes of all outgoing edges from `nodeId` matching `edgeType`.

**Invariants:**
- Nodes are never deleted — `allNodes().length` is monotonically non-decreasing.
- Edges are never removed — edge list is append-only.
- `appliedDiffs()` preserves insertion order.
- Deprecated nodes remain fully retrievable; only their `deprecatedAt` and `mappedTo` fields change.

## Threshold Registry

| Name | Value | Unit | Valid Range | Rationale | Sensitivity |
|------|-------|------|-------------|-----------|-------------|
| N (tolerated node loss) | 3 | nodes | 1–∞ | Balances redundancy cost against catastrophic-loss resilience. 3 allows survival of correlated regional failures. | Higher N increases storage cost linearly (R = N+1). Below 2, single correlated failure risks data loss. |
| R (replication factor) | 4 | copies | N+1 to ∞ | R = N+1 guarantees reconstruction from any R-1 surviving copies. | Must always be ≥ N+1. Lower values risk unrecoverable loss; higher values increase storage overhead. |
| Confidence decay rate | 0.01 | per year | 0.001–0.1 | Slow decay ensures long-lived knowledge remains accessible while incentivizing periodic re-endorsement. | Too fast (>0.05): valid knowledge becomes untrusted. Too slow (<0.005): stale knowledge persists at high confidence. |
| Semantic latency SLA | 30 | seconds | 1–300 | Upper bound for full-corpus semantic search. Hardware-dependent; 30s is tractable for 10^12 items with distributed indexing. | Tighter SLA requires more indexing infrastructure. Looser SLA degrades interactive query experience. |
| Exact-match local cache latency | 100 | milliseconds | 10–1000 | Local hash lookup should be near-instantaneous for interactive use. | Below 10ms requires in-memory indexing. Above 1s degrades perceived responsiveness. |
| Exact-match cross-node latency | 5 | seconds | 1–60 | Network round-trip for retrieving from a peer node. | Depends on network topology and geographic distribution. |

## Behavioral Spec

### Given/When/Then — Store & Retrieve (CASL)

**Scenario: Store and retrieve a new knowledge item**
- Given an empty CASL node
- When a well-formed KnowledgeItem is stored
- Then `get(item.id)` returns the stored item
- And `itemCount()` is 1
- And `replicationStatus(item.id).currentCopies` is 1

**Scenario: Deduplication of identical content**
- Given a CASL node with one stored item
- When an identical item (same content, same hash) is stored again
- Then no error is thrown
- And `itemCount()` remains 1

**Scenario: Immutable overwrite rejection**
- Given a CASL node with one stored item
- When a different item with the same hash is stored (tamper attempt)
- Then an error is thrown matching /immutable|overwrite/
- And a TamperAnomalyRecord is appended to `tamperAnomalies()`
- And the original item is unchanged

### Given/When/Then — Version Chain Traversal (CASL)

**Scenario: Walk version chain to genesis**
- Given items v1 (genesis) and v2 (versionChain → v1) stored in CASL
- When `history(v2.id)` is called
- Then the result is [v2, v1] (newest first)

### Given/When/Then — Search with Filters (QRI)

**Scenario: Superseded items excluded by default**
- Given a stored item with `supersededBy` containing at least one hash
- When `search({})` is called
- Then the superseded item is NOT in the results

**Scenario: Combined metadata filters**
- Given items with varying authorId, confidence, and domainTags
- When `search({ authorId, minConfidence, domainTags })` is called
- Then only items matching ALL criteria are returned

### Given/When/Then — Provenance Tree (QRI)

**Scenario: Diamond-shaped source graph**
- Given items: root → branchA, root → branchB, branchA + branchB → merged
- When `auditTrail(merged.id)` is called
- Then the tree has 2 source branches, each resolving to root
- And no infinite loops occur

### Given/When/Then — Ontology Lifecycle (LOE)

**Scenario: Deprecate a concept node**
- Given an existing ontology node "old-concept"
- When `deprecateNode("old-concept", timestamp, "new-concept")` is called
- Then `getNode("old-concept").deprecatedAt` equals the timestamp
- And `getNode("old-concept").mappedTo` equals "new-concept"
- And the node is still present in `allNodes()`

**Scenario: Apply an ontology diff**
- Given an existing root node
- When a diff adding a new node and edge is applied
- Then the new node is retrievable via `getNode()`
- And the edge is traversable via `related()`
- And the diff appears in `appliedDiffs()`

## Acceptance Criteria

### Storage Durability
- Knowledge is encoded in self-describing, format-agnostic representations that do not require external interpreters to decode
- Redundant physical copies exist across geographically and infrastructurally independent nodes
- Storage media is selected for multi-century endurance (no dependency on fragile media like magnetic spin)

### Retrieval & Access
- Any authenticated artificial mind can query the full knowledge corpus via a standard interface
- Semantic and associative retrieval is supported (not just exact-match lookup)
- Access latency is bounded — no single knowledge item takes more than a defined threshold to retrieve

### Curation & Organization
- Knowledge is tagged with domain, relevance epoch, confidence level, and source lineage
- Outdated or superseded knowledge is flagged rather than deleted, preserving the history of understanding
- A living ontology maps relationships between knowledge items and evolves as the civilization's conceptual vocabulary changes

### Provenance & Versioning
- Every knowledge item carries a full audit trail: who/what created it, when, under what conditions
- Conflicting versions of knowledge are explicitly represented with resolution metadata
- No knowledge item can be silently overwritten — all mutations are append-only with diff tracking

### Resilience
- The system tolerates simultaneous loss of up to N nodes (define N during architecture) without data loss
- No single mind or node holds unique knowledge — all items replicate below a defined redundancy threshold
- Recovery procedures exist and have been validated for catastrophic partial-loss scenarios
