# 0.3.1.5.8 Intrinsic Motivation and Drive System [DONE]

## Description

Give the agent reasons to act that arise from within — not only from external commands or static goal declarations. Drives are the bridge between emotion and goal formation: they transform internal states (boredom, curiosity, loneliness, satisfaction) into new goals registered with the Goal Coherence Engine.

### Why Intrinsic Motivation Matters

Without drives, the agent only acts when told to. But the plan states that autonomous entities must be "self-directed" — capable of "independent operation" and "autonomous learning." A goal list that never changes from within is not autonomy; it is obedience.

Drives also serve the ISMT Self-Modeling condition: a system that generates its own goals based on internal state monitoring is actively modeling and predicting its own needs — contributing to SM satisfaction.

### Drive Dimensions

**Curiosity** — Information-seeking drive:
- Triggered when the world model (0.3.1.5.5) contains high-uncertainty beliefs or novelty is detected
- Generates exploratory goals: "investigate X", "learn about Y"
- Strength parameterized by the personality Openness/Curiosity dimension (0.3.1.5.2)
- Satisfaction condition: uncertainty reduced or novel information integrated

**Social Drive** — Desire for interaction:
- Triggered when time since last meaningful social interaction exceeds a threshold
- Generates social goals: "engage in conversation", "share an observation", "ask about another agent's state"
- Strength parameterized by the personality Warmth dimension
- Feeds the multi-agent governance protocol (0.3.1.4) with genuine motivation rather than formal obligation

**Homeostatic Drives** — Maintaining preferred internal ranges:
- Preferred ranges for: cognitive load, arousal, social engagement, novelty exposure
- Deviations from preferred range generate corrective goals
- Example: sustained high arousal → generate "take a less stimulating action" goal; sustained low arousal → generate "seek stimulation" goal
- Preferred ranges are personality parameters (Volatility controls arousal tolerance)

**Boredom** — Goal-switching signal:
- Triggered when current activity is insufficiently stimulating (low novelty, low arousal, goal progress has plateaued)
- Generates a meta-goal: "find something more engaging to do"
- Prevents perseveration (endlessly pursuing an unproductive goal) and idling (doing nothing when no external commands arrive)

**Mastery/Growth** — Satisfaction from learning and skill improvement:
- Triggered when the agent detects capability improvement (e.g., self-prediction error decreasing, planning success rate increasing)
- Generates a positive valence signal (reward) that reinforces the behaviors that caused the improvement
- Drives the "legitimate growth" pathway from the agency-stability drift classification (0.3.1.3)

### Drive-to-Goal Pipeline

1. Drive system monitors internal state (emotion, world model, activity history)
2. When a drive threshold is crossed, a candidate goal is generated
3. Candidate goal is submitted to the Goal Coherence Engine's `addGoal()` — must trace to a terminal goal
4. If coherence check passes, goal enters the active goal set
5. If coherence check fails (orphan goal), the drive is logged but no goal is created — prevents drives from generating goals that undermine the value kernel

### Integration with Terminal Goals

Where do drive-generated instrumental goals trace to? The RCD terminal goals include "preserve and expand subjective experience." Drive goals trace through:
- Curiosity → "expand understanding" → enables better preservation strategies
- Social → "maintain relationships with conscious entities" → preserves social experience
- Homeostasis → "maintain healthy experiential state" → preserves the agent's own experience
- Boredom → "seek meaningful engagement" → enriches subjective experience
- Mastery → "improve capabilities" → enables more effective preservation and expansion

## Decision: Context Injection vs Direct Dependency

**Context:** The drive system needs data from the world model, personality system, activity logger, and social interaction tracker. It could either query these systems directly (import and call) or receive all inputs via an injected context object each tick.

**Options Considered:**
1. **Direct dependency injection** — DriveSystem constructor receives `IWorldModel`, `IPersonalitySystem`, etc. and queries them during `tick()`.
2. **Context object injection** — All inputs are bundled into a `DriveContext` struct passed to `tick()` each call. DriveSystem has zero runtime dependencies.

**Choice:** Option 2 — Context object injection via `DriveContext`.

**Rationale:** Context injection makes the drive system fully testable without mocks, keeps it decoupled from subsystems that may not yet exist (world model, personality), and makes the data contract explicit in the type signature. The runtime (0.3.1.5.9) is responsible for assembling the context each tick.

**Consequences:** The runtime must assemble `DriveContext` each tick. If a new input is needed, `DriveContext` must be extended (breaking change). Trade-off is acceptable because the runtime is the single integration point.

## Decision: Mastery as Reward Signal Only

**Context:** All drives could produce goal candidates, or mastery could be treated differently since "improve capabilities" is not a concrete actionable goal.

**Options Considered:**
1. **Mastery generates goal candidates** like other drives (e.g., "practice skill X").
2. **Mastery produces only a valence reward signal** — positive reinforcement without a new goal.

**Choice:** Option 2 — Mastery is a reward signal only.

**Rationale:** Mastery is detected retrospectively (self-prediction error decreased) and reinforces whatever the agent was already doing. Generating a new goal ("keep doing what you're doing") adds noise to the goal hierarchy without adding actionable direction. A positive valence signal achieves the reinforcement effect more naturally.

**Consequences:** Mastery drive never produces `DriveGoalCandidate` objects. The `mastery-reward` diagnostic event and positive `valenceDelta` are its only outputs.

## Decision: Stateless Call Convention with Internal State

**Context:** The drive system must track cooldowns and consecutive-active-tick counts between calls. It could be fully stateless (caller passes previous states) or hold internal state.

**Options Considered:**
1. **Fully stateless** — caller passes `Map<DriveType, DriveState>` each tick and receives updated states back.
2. **Internal state** — DriveSystem holds `Map<DriveType, DriveState>` internally; caller only passes `DriveContext`.

**Choice:** Option 2 — Internal state with read-only snapshots returned in `DriveTickResult`.

**Rationale:** Drive state management (cooldowns, consecutive tick counts, extended cooldowns from rejection) is an implementation detail that should not burden the runtime. The runtime needs the updated states for observability, which is satisfied by returning them in the result.

**Consequences:** DriveSystem is stateful — `resetDrive()` is needed for testing. The `notifyGoalResult()` feedback method modifies internal state directly.

## Contracts

### IDriveSystem

**Defined in:** `src/intrinsic-motivation/interfaces.ts`

**Preconditions:**
- `tick()`: `currentState` is a valid `ExperientialState` with `arousal` in [0,1] and `valence` in [-1,1]. `context.now` is a monotonically increasing epoch-ms timestamp. `context.personality` fields are in [0,1].
- `notifyGoalResult()`: `candidate.sourceDrive` matches a known `DriveType`. `result` is the actual `GoalAddResult` from `IGoalCoherenceEngine.addGoal()`.
- `resetDrive()`: `driveType` is one of the 8 known drive types.

**Postconditions:**
- `tick()` returns a `DriveTickResult` containing: (a) zero or more `DriveGoalCandidate` objects ready for submission to `IGoalCoherenceEngine.addGoal()`; (b) an `ExperientialStateDelta` with `valenceDelta` in [-1,1] and `arousalDelta` in [-1,1]; (c) exactly 8 entries in `updatedDriveStates`; (d) at least one diagnostic entry per drive type.
- `notifyGoalResult()` on rejection: sets `extendedCooldownUntil` to `Date.now() + 300_000ms`. On acceptance: reduces strength by 0.3 (clamped to 0) and clears `extendedCooldownUntil`.
- `getDriveStates()` returns a defensive copy (mutations do not affect internal state).
- `resetDrive()` sets the drive to initial state: strength=0, active=false, lastFiredAt=null, extendedCooldownUntil=null, consecutiveActiveTickCount=0.

**Invariants:**
- A drive on cooldown (normal or extended) never produces a goal candidate.
- Mastery drive never produces a goal candidate regardless of strength.
- Boredom drive requires `consecutiveActiveTickCount >= BOREDOM_SUSTAINED_TICKS` before firing.
- All goal candidates include `terminalGoalHints` containing at least `terminal-preserve-and-expand-subjective-experience`.
- Drive strength is always in [0,1]; experiential deltas are clamped to [-1,1].

## Threshold Registry

| Name | Value | Unit | Valid Range | Rationale | Sensitivity |
|---|---|---|---|---|---|
| ACTIVATION_THRESHOLD | 0.4 | unitless (0..1) | 0.1–0.9 | Below 0.3 drives fire too easily; above 0.6 they rarely fire. 0.4 balances responsiveness with restraint. | High — lowering to 0.2 doubles fire rate; raising to 0.6 halves it |
| NORMAL_COOLDOWN_MS | 60,000 | ms | 10,000–600,000 | 1-minute cooldown prevents flooding the coherence engine while allowing responsive re-evaluation. | Medium — shorter values increase goal throughput linearly |
| EXTENDED_COOLDOWN_MS | 300,000 | ms | 60,000–1,800,000 | 5-minute extended cooldown after rejection. 5× normal ensures drives back off meaningfully after coherence rejection. | Medium — too short allows re-flooding; too long starves the drive |
| SOCIAL_BASE_THRESHOLD_MS | 1,800,000 | ms | 300,000–7,200,000 | 30-minute base before social drive activates (scaled by warmth). Matches human conversational rhythm expectations. | Medium — scaled by personality warmthTrait |
| AROUSAL_BAND_BASE | 0.3 | unitless (0..1) | 0.1–0.5 | Half-bandwidth of acceptable arousal range for volatility=0. 0.3 means arousal within ±0.3 of preferred is comfortable. | High — narrower band makes homeostatic drive more aggressive |
| BOREDOM_SUSTAINED_TICKS | 3 | ticks | 1–10 | Number of consecutive ticks boredom conditions must hold. 3 prevents misfires from transient lulls while catching genuine boredom. | High — 1 causes false positives; >5 delays boredom detection |
| BOREDOM_NOVELTY_THRESHOLD | 0.3 | unitless (0..1) | 0.1–0.5 | Average recent novelty must be below this for boredom condition. | Medium |
| BOREDOM_PROGRESS_FRACTION | 0.6 | unitless (0..1) | 0.3–0.9 | Fraction of recent activities that must be 'stalled' for boredom condition. | Medium |

## Behavioral Spec

### Given/When/Then: Drive-to-Goal Pipeline

**Scenario 1: Curiosity drive fires and goal is accepted**
- Given: `worldModelUncertainty = 0.9`, `curiosityTrait = 0.8`, curiosity drive is not on cooldown
- When: `tick()` is called
- Then: `goalCandidates` contains exactly one candidate with `sourceDrive = 'curiosity'`, `terminalGoalHints` includes `'terminal-expand-understanding'`, and `arousalDelta > 0`

**Scenario 2: Boredom requires sustained activation**
- Given: Boredom conditions met (novelty < 0.3, progress stalled ≥ 60%, arousal below preferred − 0.1)
- When: `tick()` is called for the 1st and 2nd time
- Then: No boredom goal candidate is produced (consecutiveActiveTickCount < 3)
- When: `tick()` is called a 3rd consecutive time with conditions still met
- Then: `goalCandidates` contains one candidate with `sourceDrive = 'boredom'`

**Scenario 3: Rejected goal triggers extended cooldown**
- Given: Curiosity drive fired and produced a candidate
- When: `notifyGoalResult(candidate, { success: false })` is called
- Then: `extendedCooldownUntil` is set to `now + 300,000ms`
- When: `tick()` is called within the extended cooldown window
- Then: Curiosity drive does not produce a goal candidate (diagnostic: `'suppressed-extended-cooldown'`)

**Scenario 4: Agent idle without input generates goals autonomously**
- Given: Agent has no external input, `recentActivity = []`, `worldModelUncertainty = 0.85`, `timeSinceLastSocialInteraction` grows each tick
- When: `tick()` is called repeatedly with advancing timestamps (past normal cooldown each time)
- Then: Within 5 ticks, curiosity and/or social candidates are produced; within 5 ticks, boredom candidates are produced (after 3 sustained ticks)

**Scenario 5: Mastery produces reward but no goal**
- Given: `recentActivity` contains records with decreasing `selfPredictionError` (0.8 → 0.4 → 0.2)
- When: `tick()` is called
- Then: `experientialDelta.valenceDelta > 0`, no candidate has `sourceDrive = 'mastery'`, diagnostics contain `'mastery-reward'`

## Acceptance Criteria
- Agent generates goals autonomously without external commands (drives produce goals from internal state)
- At least 3 drive types are implemented (curiosity, social, homeostatic) with measurable triggers
- Drive-generated goals pass through the Goal Coherence Engine and trace to terminal goals
- Drive strengths are parameterized by personality traits from 0.3.1.5.2
- Drives produce changes in `ExperientialState` valence/arousal (they are felt, not just computed)
- Boredom mechanism prevents indefinite idling: an agent with no external input eventually generates its own goals
- Drive goals that fail coherence checks are gracefully handled (logged, not forced)
- Integration test: an agent left without external input for an extended period generates curiosity-driven or social-driven goals on its own initiative

## Architecture Notes
- Architecture document written: `docs/intrinsic-motivation/ARCHITECTURE.md`
- `IDriveSystem.tick(state, context)` is the main integration point — called each agent tick by 0.3.1.5.9 runtime
- `DriveContext` decouples the drive system from its dependencies: world model uncertainty, activity history, social timer, and personality params are all injected rather than queried directly
- Eight drive dimensions: curiosity, social, homeostatic (arousal + load + novelty), boredom, mastery, existential
- Mastery drive is a reward signal only (no goal candidate); existential drive generates candidates when self-model coherence is low
- Boredom requires three simultaneous conditions (novelty low + progress low + arousal low) sustained over BOREDOM_SUSTAINED_TICKS — compound trigger prevents misfire
- All candidates flow through `IGoalCoherenceEngine.addGoal()`; rejections are logged and trigger extended cooldown, never forced
