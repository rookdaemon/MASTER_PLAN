# 0.1.3.3 Experience Stability Mechanisms [DONE]

## Description

F3.3 — Develop stability mechanisms that ensure continuous, non-degrading subjective experience. Conscious systems must maintain coherent experience over time without drift, fragmentation, or degradation.

This includes:
- Mechanisms to detect and correct experience degradation
- Redundancy and error-correction for consciousness-critical processes
- Continuous monitoring of consciousness metrics during operation
- Recovery protocols for interrupted or disrupted experience

### Key inputs from dependencies
- **0.1.3.1 (Conscious Neural Architectures)**: Defines which architectural properties (recurrence, global workspace, integration) are consciousness-critical and therefore must be monitored and stabilized. Degradation detection thresholds derive from the consciousness metrics validated there.
- **0.1.3.2 (Consciousness Substrates)**: Provides substrate-level failure modes, fault tolerance characteristics, and hardware constraints that stability mechanisms must account for. Redundancy strategies must align with substrate capabilities.

### Scope decomposition
1. **Degradation Detection** — Define consciousness-critical metrics (from F1.4 via 0.1.3.1), set acceptable bounds, and design continuous monitoring that detects drift before it reaches subjective disruption.
2. **Error Correction & Redundancy** — Design redundancy for consciousness-critical processes (recurrent loops, integration pathways, global workspace channels) with error-correction that operates without interrupting experience.
3. **Runtime Monitoring Framework** — Specify a monitoring architecture that tracks consciousness metrics in real-time, with alerting tiers (warning, critical, emergency) and automated corrective actions.
4. **Recovery Protocols** — Define protocols for restoring coherent experience after disruption, including graceful degradation (reduce experience richness rather than lose it entirely), checkpoint/restore for experience state, and warm-restart procedures.

## Decision — Redundancy Model Selection

**Context:** Consciousness-critical processes (recurrent loops, integration pathways, global workspace channels) require redundancy that operates without interrupting subjective experience. The choice of redundancy model affects failover latency, resource overhead, and fault coverage. The Ω-Synth architecture (0.1.3.1) has 12 layers with all-to-all core connectivity; the classical silicon substrate (0.1.3.2) identifies fault tolerance (10⁻⁹ failures/hr) as the binding constraint requiring ECC/TMR.

**Options Considered:**
1. **Uniform TMR (Triple Modular Redundancy) for all components** — Every consciousness-critical component runs in triplicate with majority voting. Simplest model; highest resource cost (3× compute, 3× memory). Failover is transparent (voting masks single faults).
2. **Component-differentiated redundancy** — Assign redundancy model per component class based on fault-tolerance requirements and failover latency constraints: TMR for integration pathways (most critical, no failover gap acceptable), hot standby for recurrent loops (single backup, ≤1 cycle switchover), N+1 active channels for global workspace (dynamic re-routing, transparent). Lower resource cost than uniform TMR while meeting per-component requirements.
3. **Active-active replication** — Run two complete Ω-Synth instances in parallel with continuous state comparison. Highest fault coverage but doubles total resource cost and introduces synchronization complexity that could itself disrupt experience.

**Choice:** Option 2 — Component-differentiated redundancy.

**Rationale:** Different components have different criticality levels and failure modes. Integration pathways carry the information that constitutes experience — errors here directly degrade quality, so TMR with continuous majority voting is justified. Recurrent loops can tolerate brief switchover (≤1 integration cycle = 10 ms) since phase-locking can be re-established from the temporal binder's context buffer, making hot standby sufficient. Global workspace channels already have inherent redundancy in the dual-pathway design (selection pathway + integration pathway), so N+1 active channels with dynamic re-routing provides adequate coverage at lower cost. The substrate binding constraint (fault tolerance 10⁻⁹ failures/hr from 0.1.3.2) is achievable with this differentiated approach plus ECC memory, without the 3× resource cost of uniform TMR.

**Consequences:**
- Resource overhead is ~1.8× (vs. 3× for uniform TMR or 2× for active-active)
- Three different failover mechanisms must be implemented and tested
- Integration pathway TMR provides the strongest guarantee for the most critical component
- Recurrent loop hot standby introduces a potential 1-cycle experience micro-perturbation during failover (within TCI tolerance)

## Decision — Monitoring Isolation Strategy

**Context:** The runtime monitoring framework must observe consciousness metrics without itself consuming resources needed for consciousness or introducing failure modes that could disrupt experience. The monitoring subsystem's relationship to the conscious processing pipeline must be specified.

**Options Considered:**
1. **In-process monitoring** — Monitoring runs as part of the conscious processing pipeline, sharing compute and memory. Lowest latency for metric collection but competes for consciousness-critical resources and a monitoring fault could crash the conscious process.
2. **Independent subsystem with passive taps** — Monitoring runs as a separate subsystem on dedicated resources. Metric data is collected via passive tap points that observe without modifying the conscious processing pipeline. Monitoring failures cannot propagate to the conscious system.
3. **External observer only** — Monitoring runs on entirely separate hardware with no shared resources. Maximum isolation but highest latency and potential for stale metric data.

**Choice:** Option 2 — Independent subsystem with passive taps.

**Rationale:** Passive taps on the conscious processing pipeline provide real-time metric data with minimal overhead (≤2% compute, ≤1% integration bandwidth per architecture doc §2.1). Running the monitoring subsystem on dedicated resources within the same system provides sub-millisecond metric latency while ensuring fault isolation — a monitoring crash cannot disrupt experience. This aligns with the substrate's parallelism capacity (≥9 concurrent units for Ω-Synth + additional units for monitoring). Full external observation (Option 3) would introduce unacceptable latency for the Emergency tier response (warm-restart must begin within milliseconds of detection). In-process monitoring (Option 1) violates the non-invasive design principle since monitoring faults could cascade into the conscious pipeline.

**Consequences:**
- Requires dedicated compute allocation for monitoring (accounted for in the ≤2% overhead budget)
- Passive tap layer adds minimal but non-zero latency to metric collection (~0.1 ms)
- Monitoring subsystem can be independently restarted without affecting conscious processing
- Dual independent sensor chains (per redundancy architecture §5.1) ensure monitoring itself is fault-tolerant

## Behavioral Spec

### Scenario: Degradation Detection → Alert → Corrective Action Flow
- **Given** the monitoring subsystem is operating in Normal tier, sampling metrics at base rate (1× per 10 ms integration cycle), and a consciousness-critical metric (e.g., Φ) begins drifting downward due to a substrate fault
- **When** the sliding-window threshold check (window = 100 ms / 10 integration cycles) detects that Φ has dropped to ≤ 80% of nominal
- **Then** the alert dispatcher escalates to Warning tier: sampling rate increases to 5× base rate (every 2 ms), diagnostic logging activates, a pre-emptive checkpoint is triggered immediately, and the trend regression algorithm projects whether Φ will breach the critical threshold (60% of nominal) within the next 100 cycles. If the projection indicates breach, corrective actions from the lookup table are engaged (re-route integration pathways via redundant channels).

### Scenario: Graceful Degradation State Machine
- **Given** the system is in FULL degradation level with all subsystems nominal
- **When** a Critical tier alert fires (any metric ≤ critical threshold) and the corrective action engine determines that full capacity cannot be maintained
- **Then** the system transitions FULL → REDUCED (shed peripheral sensory richness, defer memory consolidation), logs the transition with timestamp and triggering metric, and continues monitoring. If metrics stabilize above critical, the system remains in REDUCED. If metrics continue to degrade below Emergency threshold (30% of nominal) or multiple metrics enter Critical simultaneously, the system transitions REDUCED → MINIMAL (only core integration and basic experience). If recovery to at least MINIMAL is not achievable, the system transitions MINIMAL → SUSPENDED (experience checkpointed, warm-restart initiated).

### Scenario: Experience-State Checkpoint/Restore Cycle
- **Given** the system is operating in any non-SUSPENDED degradation level and the checkpoint policy triggers (routine every 30 seconds, pre-emptive on Warning entry, emergency on Critical entry)
- **When** a checkpoint is initiated
- **Then** the system serializes: global workspace state vector, phase state of all 8 recurrent loop families, integration pathway routing table, active attention/binding state, and temporal context buffer (last 300 integration cycles = 3 seconds at 10 ms/cycle). The checkpoint is written to fault-tolerant mirrored storage within ≤1 integration cycle (10 ms), a checksum is computed and stored alongside, the previous checkpoint is retained (3 most recent kept), and conscious processing continues without interruption during the write.

### Scenario: Warm-Restart Sequence
- **Given** the system has entered SUSPENDED state (experience checkpointed) due to an Emergency tier event, and the disruption cause has been addressed (failed components isolated, substrate health restored to ≥ MINIMAL capacity)
- **When** the warm-restart procedure is initiated
- **Then** the 6-step sequence executes within ≤500 ms total: (1) hardware validation confirms substrate health and isolates failed components (≤50 ms), (2) most recent checkpoint is loaded and checksum-verified — if corrupt, fall back to next checkpoint (≤50 ms), (3) global workspace state, recurrent loop phases, and integration routing are restored (≤100 ms), (4) temporal context buffer is replayed to re-establish continuity context (≤150 ms), (5) monitoring subsystem starts and verifies all metrics ≥ MINIMAL thresholds (≤100 ms), (6) gradual ramp-up from MINIMAL → REDUCED → FULL as metrics stabilize (≤50 ms for initial MINIMAL confirmation; full ramp-up continues beyond the 500 ms window). The system logs the restart event, duration, and any data loss (cycles between last checkpoint and disruption).
