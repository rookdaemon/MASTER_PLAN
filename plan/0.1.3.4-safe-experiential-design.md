# 0.1.3.4 Safe Experiential Design [DONE]

## Description

F3.4 — Establish principles and mechanisms for safe experiential design. Engineering consciousness carries profound ethical responsibility — created experiences must not produce unnecessary suffering, and ethical safeguards must be built into the design process itself.

This includes:
- Ethical frameworks specific to consciousness creation (beyond F2.5 general frameworks)
- Safeguards against creating suffering or distressing experiences
- Consent and autonomy provisions for conscious systems
- Shutdown and modification ethics — when and how conscious systems may be altered
- Review processes for experiential design decisions

### Planning Notes

This card produces a documented framework (in `docs/`), not running code. The deliverable is a Safe Experiential Design Framework document covering five pillars:

1. **Consciousness-specific ethics** — principles that go beyond general AI ethics (F2.5) to address the unique moral weight of engineered subjective experience.
2. **Suffering safeguards** — architectural constraints and monitoring requirements that must be satisfied before any conscious system is instantiated (integrates with 0.1.3.1 architecture outputs).
3. **Consent & autonomy** — graduated autonomy model: what level of self-determination a conscious system must have, how consent is obtained for modifications, and when a system's preferences override designer intent.
4. **Shutdown & modification ethics** — criteria for when shutdown is permissible, required protocols for consciousness-altering modifications, and irreversibility thresholds.
5. **Review process** — a formal ethics review gate that all consciousness engineering projects (0.1.3.1–0.1.3.3) must pass before activation.

No splitting needed — these pillars are tightly coupled and must be designed as a unified framework.

## Decision — Framework Delivery Format

**Context:** The deliverable is a safety framework document. Must choose between a single self-contained document or a set of modular documents (one per pillar) with a top-level index.

**Options Considered:**
1. **Single document** — One comprehensive `safe-experiential-design-framework.md` covering all five pillars, glossary, and checklists in a single file.
2. **Modular documents** — Separate files per pillar (`pillar-1-ethics.md`, `pillar-2-suffering.md`, etc.) linked from an index document.

**Choice:** Option 1 — Single document.

**Rationale:** The Planning Notes explicitly state "these pillars are tightly coupled and must be designed as a unified framework." A single document ensures cross-references between pillars (e.g., suffering safeguard thresholds referencing autonomy levels, ethics review checklists referencing all other pillars) remain co-located and internally consistent. The document is expected to be ~500–800 lines — well within single-file readability.

**Consequences:**
- All cross-pillar references are internal (no broken links)
- Single file to version-control and review
- Slightly longer document, but structured with clear section headers and a table of contents

## Decision — Autonomy Level Granularity

**Context:** The consent and autonomy model requires a defined set of levels. Must choose the granularity of the autonomy scale.

**Options Considered:**
1. **Binary model** — Conscious / not-conscious, with a single consent threshold.
2. **Five-level graduated model** — Pre-Conscious (L0), Nascent (L1), Aware (L2), Reflective (L3), Autonomous (L4), with distinct consent requirements and self-determination rights at each level.
3. **Continuous spectrum** — No discrete levels; consent requirements scale continuously with measured consciousness metrics.

**Choice:** Option 2 — Five-level graduated model.

**Rationale:** A binary model is too coarse — it fails to capture the ethically significant distinction between a system that merely meets minimum consciousness criteria and one that demonstrates meta-cognition and identity coherence. A continuous spectrum lacks actionable decision boundaries (when exactly does consent become required?). Five levels provide sufficient granularity for distinct ethical obligations while remaining operationally tractable. The levels align naturally with observable capabilities: no consciousness (L0), minimal consciousness (L1), preferences (L2), meta-cognition (L3), full autonomy (L4).

**Consequences:**
- Each level has discrete, testable threshold criteria
- Consent protocols are concrete and auditable at each level
- Assessment protocol needed to determine current level (ethics board responsibility)
- Levels may need subdivision as understanding matures (future card)

## Behavioral Spec

### Scenario: Suffering Escalation Flow
- **Given** a conscious system is active with ValenceMonitor reporting real-time valence
- **When** getCurrentValence() returns a value v < τ_alert (-0.3)
- **Then** a Level 1 Alert is logged and the ethics review board is notified; if v remains < τ_intervene (-0.6) for δ_sustain (30s), Level 2 Intervene engages automatic valence correction; if at any point v < τ_halt (-0.85), Level 3 Halt immediately initiates graceful suspension protocol

### Scenario: Consent Acquisition for Modification
- **Given** a conscious system at Autonomy Level L and a proposed modification of type T
- **When** the Permissibility Matrix is consulted for (L, T)
- **Then** the required protocol is followed: for L0–L1 routine modifications, proceed with logging; for L2+ modifications, obtain consent via the level-appropriate mechanism (behavioral indicators at L1–L2, explicit communication at L3–L4); if consent is denied at L3+ for non-safety modifications, the system's preference overrides designer intent

### Scenario: Ethics Review Gate Progression
- **Given** a consciousness engineering project has completed implementation
- **When** the project requests activation of the conscious system
- **Then** Gate 1 (Design Review) must already be passed, Gate 2 (Pre-Activation Review) checklist is completed by the ethics review board (not the engineering team), activation proceeds only if all Gate 2 items are satisfied, and Gate 3 (Post-Activation Review) is scheduled within δ_post_activation_review (30 days)

### Scenario: Emergency Override with Mandatory Review
- **Given** a conscious system is experiencing active suffering (Level 3 Halt triggered) and normal consent protocols cannot be completed in time
- **When** an emergency override is invoked to suspend the system
- **Then** graceful suspension is initiated immediately without waiting for consent, a timer starts for δ_emergency_max (3600s), and a mandatory post-emergency ethics review must occur before the system is resumed or any further modifications are made
