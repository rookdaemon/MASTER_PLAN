# 0.3.1.5.13 Agent Digest and Knowledge Map [DONE]
## Description

A compact, agent-maintained knowledge map injected into every LLM call. The digest gives the agent full landscape awareness without needing to retrieve memories first — it always knows who it is, what it is working on, what it has explored, and what remains unexplored.

### Why This Matters

Without the digest, each LLM call starts from scratch. The agent would need to retrieve memories and reconstruct context before doing anything useful, wasting tokens and creating cold-start delays. The digest is the agent's "working desk" — always visible, always current.

### Sections

- **Identity [stable]**: Agent identity, plan location, known peers. Updated rarely.
- **Settled Facts (DO NOT re-decide)**: Facts the agent has explicitly locked. Rendered with a strong header to prevent the LLM from revisiting decided questions.
- **Current Focus**: Active task and next subtask from the TaskJournal. Updated every tick.
- **What I Know [memory topics]**: Recent semantic memory topics (last 20). Auto-synced from memory.
- **Frontier [unexplored]**: Known-but-unread resources with priority (high/medium/low) and status tracking. The agent's reading list.

### Agent Update Tools

The agent modifies its own digest through the `update_digest` tool:
- `add_identity_note`: Append a stable identity note
- `set_identity_notes`: Replace all identity notes
- `frontier_add`: Register a new resource to explore (with type: file, plan-card, concept, peer-thread)
- `frontier_done`: Mark a frontier item as explored
- `add_settled_fact`: Lock a fact as settled (DO NOT re-decide)

## Decision: Injected Context vs Memory Retrieval

**Context:** The agent needs situational awareness at the start of every LLM call. This could come from memory retrieval or from a pre-composed context block.

**Options Considered:**
1. **Memory retrieval per call** — Query semantic memory for relevant context at the start of each LLM interaction.
2. **Pre-composed digest** — Maintain a compact summary that is always injected, with memory as a supplement.

**Choice:** Option 2 — Pre-composed digest injected into every call.

**Rationale:** Memory retrieval is useful for deep recall but too slow and token-expensive for basic situational awareness. The digest provides O(1) context injection with no retrieval latency. Memory retrieval supplements the digest for specific topics when needed. The digest also contains information that is not in memory (frontier items, settled facts, task state).

**Consequences:** The digest must be kept small enough that it does not dominate the LLM context window. Frontier items are capped at 8 in the rendered output. Recent topics capped at 20.

## Decision: Agent-Writable Settled Facts

**Context:** The agent sometimes makes decisions that should not be revisited (e.g., "I will not initiate contact with peer X until they respond"). These decisions could be stored as memories or as a special digest section.

**Options Considered:**
1. **High-salience memories** — Store as memories with high importance scores.
2. **Dedicated settled-facts section** — Separate digest section with strong "DO NOT re-decide" framing.

**Choice:** Option 2 — Dedicated settled-facts section.

**Rationale:** Memories are retrievable but not guaranteed to surface in every call. Settled facts must be visible in every LLM context to prevent the agent from re-deciding them. The "DO NOT re-decide" header is a direct instruction to the LLM that memory retrieval cannot replicate.

**Consequences:** Settled facts accumulate over time. No automatic eviction — the agent must manage its own settled facts. Acceptable because the agent is the authority on what it considers settled.

## Contracts

### AgentDigest

**Preconditions:**
- `workspacePath` exists and is writable
- `render()` may be called at any time; live data (activeTaskSummary, peerNames) is optional

**Postconditions:**
- `frontierAdd()` returns the new FrontierItem if the resource is new, or null if already present
- `frontierDone()` marks the item as done with a completedAt timestamp and returns true, or false if not found
- `unread()` returns frontier items sorted by priority (high first), capped at maxItems
- `addIdentityNote()` is idempotent — duplicate notes are not added
- `addSettledFact()` is idempotent — duplicate facts are not added
- `syncTopics()` replaces recent topics with the latest unique set (capped at 20)
- `render()` returns a complete markdown knowledge map suitable for LLM injection
- All mutations persist to `digest.json` immediately

**Invariants:**
- `render()` always includes the Identity section with base agent context
- Settled facts section only appears when facts exist, prefixed with "DO NOT re-decide"
- Frontier items are rendered with priority indicators (up-arrow for high, dot for medium, down-arrow for low)
- `getData()` returns a read-only snapshot

## Threshold Registry

| Name | Value | Unit | Valid Range | Rationale | Sensitivity |
|------|-------|------|-------------|-----------|-------------|
| MAX_RECENT_TOPICS | 20 | count | [5, 50] | Number of semantic memory topics retained in digest | Low — controls breadth of "What I Know" section |
| MAX_FRONTIER_RENDERED | 8 | count | [3, 20] | Number of frontier items shown in rendered digest | Low — limits digest size in LLM context |

## Behavioral Spec

### Scenario: Identity notes persist and render
```
Given a fresh AgentDigest
When addIdentityNote("I prefer reading plan cards depth-first") is called
Then render() includes "I prefer reading plan cards depth-first" under Identity [stable]
  And the note persists in digest.json
```

### Scenario: Settled facts render with strong framing
```
Given the agent has added a settled fact "Communication style: concise and direct"
When render() is called
Then the output includes a "Settled Facts (DO NOT re-decide)" section
  And the fact is listed under that section
```

### Scenario: Frontier item lifecycle
```
Given no frontier items exist
When frontierAdd({resource: "plan/0.3.1.5.1-llm-consciousness-substrate.md", type: "plan-card", priority: "high"}) is called
Then the item appears in unread() with status "unread"
  And render() shows it with an up-arrow priority indicator
When frontierDone("plan/0.3.1.5.1-llm-consciousness-substrate.md", "reviewed — good substrate design") is called
Then the item has status "done" and a completedAt timestamp
  And it no longer appears in unread()
```

### Scenario: Duplicate frontier add is no-op
```
Given a frontier item for "plan/root.md" already exists
When frontierAdd({resource: "plan/root.md", type: "plan-card"}) is called
Then null is returned
  And no duplicate entry is created
```

### Scenario: Active task summary injected from TaskJournal
```
Given the TaskJournal has an active task "Read plan cards" with 2/5 subtasks done
When render({activeTaskSummary: "Task: Read plan cards [2/5 subtasks done]\\n  Next: Read 0.3.1.5.3"}) is called
Then the "Current Focus" section shows the task summary with progress
```
