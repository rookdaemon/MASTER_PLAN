# 0.3.1.5.12 Inner Monologue and Reasoning Audit [DONE]
## Description

A human-readable narrative log that captures the full back-and-forth of every drive-initiated LLM interaction. Separate from the structured `debug.log`, the inner monologue tells the story of the agent's reasoning in a comprehensible format: what drives fired, what prompt was sent, how the LLM responded, which tools were called, and what the final output was.

### Why This Matters

The agent's decision-making is opaque without an audit trail. Debug logs are structured but noisy. The inner monologue provides a narrative record that a human operator can read sequentially to understand what the agent did and why. It also feeds the web UI's real-time stream via SSE, letting operators observe the agent's reasoning as it happens.

### Design

- Written to `inner-monologue.txt` in the agent's state directory
- Entry types: drive_activation, system_prompt, user_message, assistant_text, tool_call, tool_result, final_output, summary, error
- Uses visual formatting (box-drawing characters) for readability
- Summary lines include model ID for attribution
- Real-time streaming via `MonologueListener` callbacks (used by web UI SSE endpoint)
- Capped at 5000 lines with FIFO eviction (oldest lines trimmed)
- Injectable dependencies (`MonologueDeps`) for testability — file I/O can be replaced with no-ops
- `NoopInnerMonologueLogger` subclass for tests that need no disk I/O

## Decision: Separate Log vs Structured Debug Integration

**Context:** The agent already has a structured debug.log. The monologue could be a view on the same data or a separate log stream.

**Options Considered:**
1. **Structured debug.log with narrative renderer** — Single log source, monologue is a post-hoc rendering.
2. **Separate narrative log** — Independent file, written in real-time alongside debug.log.

**Choice:** Option 2 — Separate narrative log.

**Rationale:** The debug.log is optimized for machine consumption (JSON-structured, keyed by subsystem). The inner monologue is optimized for human reading (box-drawn sections, natural flow). Trying to generate one from the other adds coupling and latency. The monologue must stream in real-time via SSE, so it needs its own write path. Both can coexist — the debug.log captures what happened at the system level, the monologue captures what happened at the experiential level.

**Consequences:** Two log files to manage. Acceptable because they serve different audiences and the monologue is capped at 5000 lines.

## Decision: Line-Based FIFO Capping

**Context:** The monologue grows continuously and must not consume unbounded disk space.

**Options Considered:**
1. **Rotation on startup** — Archive old log, start fresh each session.
2. **FIFO line eviction** — Cap at N lines, trim oldest when exceeded.
3. **Size-based rotation** — Rotate when file exceeds N bytes.

**Choice:** Option 2 — FIFO eviction at 5000 lines.

**Rationale:** FIFO eviction keeps the most recent context available at all times without session boundaries. The agent runs continuously; startup-based rotation would lose recent context during long sessions. 5000 lines is enough to review the last several cognitive cycles while staying under reasonable disk limits.

**Consequences:** Very old entries are lost. Acceptable because the debug.log retains the structured equivalent and the monologue is a real-time observability tool, not an archival record.

## Contracts

### InnerMonologueLogger

**Preconditions:**
- `logFile` path is a valid writable filesystem location
- `deps` provides working file I/O implementations (or no-ops for testing)

**Postconditions:**
- All log methods append formatted text to the log file immediately
- `addListener()` registers a callback that receives every entry in real-time
- `removeListener()` stops delivery to the removed callback
- `summary()` includes model ID when provided
- Line count never exceeds MAX_LINES (5000) after any write

**Invariants:**
- Listener errors never propagate (caught and swallowed to protect the agent)
- `_ensureDir()` creates the parent directory on first write only
- Entry timestamps use ISO 8601 format
- The NoopInnerMonologueLogger produces no disk I/O

## Threshold Registry

| Name | Value | Unit | Valid Range | Rationale | Sensitivity |
|------|-------|------|-------------|-----------|-------------|
| MAX_LINES | 5000 | lines | [500, 50000] | Balance between context retention and disk footprint | Low — larger values retain more history at marginal disk cost |

## Behavioral Spec

### Scenario: Drive activation logged with formatting
```
Given the InnerMonologueLogger is initialized with a writable log file
When driveActivation(42, [{sourceDrive: 'curiosity', description: 'explore plan cards'}]) is called
Then the log file contains a formatted drive activation header with cycle number 42
  And the activated drives are listed with source and description
  And all registered listeners receive a MonologueEntry with type 'drive_activation'
```

### Scenario: Tool call and result captured
```
Given a drive-initiated LLM interaction is in progress
When the LLM calls read_file with {path: "plan/root.md"}
  And the tool returns the file contents
Then toolCall() logs the tool name and input JSON
  And toolResult() logs the result text with a success prefix
  And both entries are emitted to listeners
```

### Scenario: FIFO eviction at capacity
```
Given the monologue log has 4990 lines
When a driveActivation writes 15 new lines
Then the log file is trimmed to 5000 lines
  And the oldest lines are removed
```

### Scenario: Listener error isolation
```
Given a listener that throws an exception on every call
When assistantText() is called
Then the text is still written to the log file
  And other listeners still receive the entry
  And no exception propagates to the caller
```

### Scenario: Summary includes model ID
```
Given a completed LLM interaction with 3 iterations
When summary(3, 15000, 2000, 'claude-opus-4-6') is called
Then the log line contains "3 iteration(s), 15000 prompt tokens, 2000 completion tokens [claude-opus-4-6]"
```
