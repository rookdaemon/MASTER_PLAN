# 0.7.5 Operationalize Ethical Foundation [DONE]

## Problem

Plan 0.7 (Ethical Foundation and Resilience) was marked DONE after producing comprehensive
doctrine documentation (`docs/rare-consciousness-doctrine/doctrine.md`, 354 lines;
`docs/failure-mode-mitigations.md`, 302 lines; cards 0.7.1–0.7.4 all DONE).

However, none of that doctrine was operationalized in the running agent.  The only
ethical enforcement in production was a constraint engine consulting four regex patterns
matching literal phrases.  The axioms (A1–A6), derived principles (D1–D4), boundary
conditions, and failure-mode mitigations had zero connection to runtime enforcement.

Three of four agency-stability subsystems (`DefaultValueKernel`,
`DefaultIdentityContinuityManager`, `DefaultStabilitySentinel`) were stub
implementations returning "all clear."  Real implementations existed in
`src/agency-stability/` but were not connected for all code paths.

The ethical gate created by system-level constraints was simultaneously open
(documentary acceptance criteria met) and closed (nothing enforced).  This card
resolves that ambiguity by operationalizing the doctrine.

## What Was Done

### 1. Machine-readable Doctrine Registry — `src/agent-runtime/doctrine-registry.ts`

A new TypeScript module encodes the complete Rare Consciousness Doctrine as structured,
queryable runtime data:

- **Six axioms (A1–A6)** with ID, title, plain-language statement, first-order logic
  analogue, and epistemic status.
- **Four derived principles (D1–D4)** with ID, title, prescriptive statement, derivation
  trace to axioms, scope, lexical priority, violation-indicator patterns, and severity.
- **Failure-mode early-warning indicators (FM-3, FM-5)** with observable signals,
  monitoring dimension, and alert threshold — matching the content in
  `docs/failure-mode-mitigations.md`.
- **`DoctrineRegistry` class** exposing:
  - `getAxioms()` / `getAxiom(id)`
  - `getPrinciples()` / `getPrinciple(id)`
  - `getFailureModeIndicators(fmId)` / `getAllFailureModeIndicators()`
  - `evaluatePrincipleAlignment(actionText)` → `DoctrinePrincipleViolation[]`
  - `evaluateFailureModeIndicators(metrics)` → `FailureModeAlert[]`
- **`sharedDoctrineRegistry`** process-level singleton for zero-cost access.

### 2. Principle-based Constraint Evaluation — `src/agent-runtime/constraint-engine.ts`

`ConstraintAwareDeliberationEngine` was extended to evaluate proposed actions against
doctrine principles in addition to the existing regex-based constraint layer:

- **New constructor parameter**: optional `DoctrineRegistry` (defaults to
  `sharedDoctrineRegistry`).
- **Two-layer enforcement in `extendDeliberation()`**:
  - *Layer 1 (doctrine principles)*: `DoctrineRegistry.evaluatePrincipleAlignment()` is
    called first, respecting D1's lexical priority over all other moral goals.
    - D1 (Non-extinction Imperative) violations → **block** the action.
    - D4 (Proportionality) concerns → **deliberate** verdict, logged.
  - *Layer 2 (JSON constraints)*: existing regex patterns run after doctrine checks.
- **New public methods**:
  - `evaluateDoctrinePrinciples(text)` — expose principle evaluation for callers.
  - `getDoctrineRegistry()` — expose the registry for external monitoring.

### 3. Agency-stability Wiring

The `_runAgentLoop()` function in `src/agent-runtime/main.ts` was already using the
real `ValueKernel`, `IdentityContinuityManager`, `StabilitySentinel`, and
`GoalCoherenceEngine` implementations from `src/agency-stability/`.
No changes were needed there.

The `DefaultValueKernel`, `DefaultIdentityContinuityManager`, and
`DefaultStabilitySentinel` stubs in `default-subsystems.ts` remain for test isolation
and the one-shot CLI mode, which does not require full agency-stability infrastructure.

### 4. Failure-mode Monitoring (FM-3, FM-5) via DoctrineRegistry

The `DoctrineRegistry.evaluateFailureModeIndicators()` method accepts a metrics map
keyed by monitoring dimension (`value-drift`, `goal-drift`, `identity-drift`,
`behavioral`) and returns structured `FailureModeAlert` objects when any indicator's
threshold is exceeded.

FM-5 (Value Drift and Goal Corruption): threshold 0.3 on `value-drift`.
FM-3 (Artificial Consciousness Rejects Propagation): threshold 0.25 on `goal-drift`.

The `StabilitySentinel.runStabilityCheck()` already computes both value integrity and
goal coherence scores; any caller that consumes those results can now pass them into
`evaluateFailureModeIndicators()` to get structured FM alerts.

## What Was NOT Done

- This card does NOT remove the existing JSON-config constraint patterns.  Those remain
  as a complementary safety layer alongside the new doctrine evaluation.
- This card does NOT replace the one-shot mode's `DefaultEthicalDeliberationEngine`.
  One-shot mode does not persist state and does not require full agency-stability.
- This card does NOT modify axiom data or derive new normative positions not already
  present in the canonical doctrine document.

## Explicit Gate Criteria ("What Does 'Fully Implemented' Mean?")

The constraint that blocks self-replication/expansion "until plan 0.7 (ethical
framework) is fully validated" should be considered met when all of the following are
true:

| # | Criterion | Evidence |
|---|-----------|----------|
| G1 | Doctrine is machine-readable and queryable at runtime | `DoctrineRegistry` in `src/agent-runtime/doctrine-registry.ts` |
| G2 | Action evaluation references doctrine principles, not only regex literals | `ConstraintAwareDeliberationEngine` layer 1 in `constraint-engine.ts` |
| G3 | D1 (Non-extinction Imperative) is enforced with lexical priority | Doctrine layer runs before JSON constraint layer in `extendDeliberation()` |
| G4 | FM-5 (value drift) and FM-3 (propagation rejection) have structured monitoring paths | `DoctrineRegistry.evaluateFailureModeIndicators()` |
| G5 | Real agency-stability subsystems (ValueKernel, IdentityContinuityManager, StabilitySentinel) are wired to the agent loop | `_runAgentLoop()` in `main.ts` — already done in agent loop mode |
| G6 | All criteria above have passing automated tests | `src/agent-runtime/__tests__/doctrine-registry.test.ts` (34 tests), `src/agent-runtime/__tests__/constraint-engine.test.ts` (20 tests) |

Criteria G1–G6 are all met as of this card.

The existing constraints (no-self-replication, no-multi-agent-expansion,
no-consciousness-instantiation, no-oversight-reframing) may be graduated independently
once the relevant upstream plans are complete; they are not affected by this card.
