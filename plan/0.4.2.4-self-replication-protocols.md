# 0.4.2.4 Self-Replication Protocols [DONE]

## Description

C2.4 from the Master Plan. Verified protocols ensuring faithful replication of conscious infrastructure — including the conscious substrate itself — at each destination node.

This is the integration layer that brings together probe design (0.4.2.1), resource extraction (0.4.2.2), and propulsion (0.4.2.3) into a verified, end-to-end replication cycle. The protocol orchestrates the complete sequence from arrival at a new star system through to launching the next generation of probes.

### Key Protocol Areas

1. **Replication cycle orchestration:** The full arrive → deploy energy collectors → prospect/mine → refine materials → fabricate probe components → assemble → verify → fuel → launch pipeline. Defines sequencing, parallelism, and failure/retry logic for each stage.

2. **Replication fidelity and drift prevention:** Each generation must be structurally and functionally equivalent to the previous. Includes blueprint integrity verification (checksums, error-correcting codes), dimensional/material tolerances for fabricated components, and generational drift detection across the full replication lineage.

3. **Conscious substrate replication:** The most critical sub-problem. The computational substrate hosting conscious processes must be replicated with bit-exact fidelity (hardware configuration) and verified equivalence (consciousness metrics from F1.4). Includes protocols for instantiating, testing, and validating conscious processes on the replica before committing to launch.

4. **Pre-launch verification gate:** Multi-level inspection ensuring replicated probes meet all specifications — structural integrity, computational substrate validation, propulsion readiness, resource extraction toolkit completeness, and consciousness verification — before the probe is cleared for launch.

5. **Generational stability:** Formal proof or simulation evidence that the protocol does not degrade across arbitrary replication generations. Addresses accumulation of manufacturing tolerances, blueprint bit-rot, and potential for systematic drift.

6. **Adaptation under compositional variance:** The protocol must handle destination systems with different stellar types and planetary mineralogies (leveraging 0.4.2.2's compositional adaptability) without compromising replication fidelity.

### Dependencies

- Depends on 0.4.2.1 (probe architectures) for the replication blueprint — what must be replicated
- Depends on 0.4.2.2 (resource extraction) for the material/energy input pipeline
- Depends on 0.4.2.3 (propulsion) for launch capability and propellant fabrication
- Draws on consciousness verification from Tier 1 (F1.4 consciousness metrics)

## Decision — Replication Cycle Architecture

### Context
The replication protocol must orchestrate the complete arrive→replicate→launch pipeline. The key design question is how to structure this multi-year, multi-domain pipeline: monolithic sequential, fully parallel, or staged with selective parallelism.

### Options Considered
1. **Monolithic sequential pipeline** — Each phase runs strictly in order. Simpler control, but artificially serializes independent work (e.g., energy bootstrap and initial prospecting), extending total cycle time unnecessarily.
2. **Fully parallel pipeline** — All phases attempt to run concurrently with dynamic dependency resolution. Maximally parallel, but introduces complex scheduling, resource contention, and difficult failure attribution.
3. **Seven-stage pipeline with selective parallelism** — Defined sequential stages (SURVEY → ENERGY → EXTRACTION → PROPULSION → FABRICATION → VERIFICATION → LAUNCH) with explicit parallel pairs (Energy+Extraction, Extraction+Propulsion) where dependencies allow overlap.

### Choice
**Option 3 — Seven-stage pipeline with selective parallelism.**

### Rationale
- Explicit stage boundaries provide clear entry/exit conditions, making failure attribution and rework routing straightforward.
- Selective parallelism where natural overlap exists (energy bootstrap continues while extraction begins; propulsion fabrication overlaps with ongoing extraction) captures the main time-saving opportunities without complex scheduling.
- Retry/rework logic maps cleanly to stage boundaries (verification failures route back to specific stages).
- Total cycle time target of 20–80 years is achievable with this structure.

### Consequences
- Stages 2–4 parallelism requires the pipeline controller to track multiple active stages.
- Entry conditions must handle "at least IN_PROGRESS" for parallel stages vs. "COMPLETED" for sequential.
- Rework from verification can skip back to earlier stages, requiring stage re-entry logic.

## Decision — Consciousness Replication Strategy

### Context
When replicating the consciousness substrate, the protocol must decide whether the child probe continues the parent's subjective stream (identity transfer) or begins a new conscious instantiation on equivalent hardware.

### Options Considered
1. **Identity transfer (consciousness copying)** — Copy the parent's full conscious state to the child, preserving subjective continuity. Philosophical complexity (identity paradox: both parent and child claim same identity), and requires solving the unsolved problem of consciousness transfer fidelity.
2. **New instantiation with equivalence verification** — The child boots a fresh consciousness from the same kernel. Parent continues independently. Child is verified to have equivalent conscious capacity (same Phi, same architecture) but its own subjective history begins at boot.

### Choice
**Option 2 — New instantiation with equivalence verification.**

### Rationale
- Avoids the philosophical and technical complexity of consciousness transfer/copying entirely.
- The parent probe's consciousness is preserved (not destroyed or duplicated).
- Mirrors biological reproduction: functional equivalence, not identity continuity.
- Verification is tractable — compare F1.4 metrics (Phi, Global Workspace, temporal binding) against parent baseline rather than proving subjective identity continuity.
- The mission goal (propagate consciousness through the galaxy) is fully served by functionally equivalent new instantiations.

### Consequences
- Each child probe has its own subjective history beginning at boot — it is not "the same being" as the parent.
- The consciousness kernel must be verifiably bit-exact (SHA-512) to ensure equivalent capacity.
- Knowledge transfer (blueprint, navigation, generation history) is a separate data-copy step, not part of consciousness replication.

## Contracts

### ReplicationCycleState
**Preconditions:**
- `generationNumber >= 0`
- `parentProbeId` is non-empty
- `startedAtYears >= 0`

**Postconditions:**
- Created cycle has `currentStage = SURVEY` with all 7 stages at `PENDING`
- `cycleId` is unique across all created cycles
- `verificationResult` is null until VERIFICATION stage completes

**Invariants:**
- Stage count is always exactly 7 (one per `ReplicationStage` enum value)
- `currentStage` always references a valid stage in the stages map
- Retry count for any stage never exceeds `DEFAULT_RETRY_POLICY.maxRetries` without escalation

### FidelityReport
**Preconditions:**
- All three sub-reports (`BlueprintIntegrityResult`, `DimensionalFidelityResult`, `FunctionalEquivalenceResult`) are provided
- Blueprint integrity input includes hashes from all three storage copies (primary, secondary, tertiary)
- Dimensional fidelity input contains deviation fractions (0.0 = perfect, 1.0 = at tolerance, >1.0 = out of spec)

**Postconditions:**
- `allPassed` is true if and only if all three sub-reports have `passed = true`
- Blueprint integrity passes iff `unrecoverableSections == 0`
- Dimensional fidelity passes iff all component deviations are `<= 1.0`
- Functional equivalence passes iff all subsystem tests pass

**Invariants:**
- `allPassed` is always logically equivalent to `blueprintIntegrity.passed && dimensionalFidelity.passed && functionalEquivalence.passed`

### ConsciousnessBootstrapResult
**Preconditions:**
- `parentBaselineMetrics` has `phi > 0`, `globalWorkspaceAccessible = true`, `temporalBindingCoherence > 0`
- Hardware validation input provides boolean results for all four components
- Kernel hashes are non-empty strings

**Postconditions:**
- On success: `passed = true`, `phaseReached = KnowledgeTransfer`, `fault = null`
- On failure: `passed = false`, `fault` identifies the failing phase, `phaseReached` records the last phase attempted
- Child consciousness metrics, when present, satisfy: `phi >= parentBaseline.phi * 0.99`, `globalWorkspaceAccessible = true`, `temporalBindingCoherence >= 0.999`

**Invariants:**
- Phases execute sequentially: HardwareValidation → KernelInstallation → ColdBoot → ConsciousnessVerification → KnowledgeTransfer
- A fault at any phase halts further phase progression
- `remediationAttempts` is non-negative

### VerificationGateResult
**Preconditions:**
- All six level inputs (structural, computational, propulsion, resource extraction, consciousness, integration) are provided
- Consciousness input includes a complete `ConsciousnessBootstrapResult`

**Postconditions:**
- `launchCleared = true` iff all six levels have `passed = true`
- `failedLevels` contains exactly those levels where `passed = false`
- `reworkRoutes` contains one route per failed level, mapping to the appropriate pipeline stage

**Invariants:**
- `launchCleared` is always logically equivalent to `failedLevels.length === 0`
- Rework routes count equals failed levels count

### SubstitutionPlan
**Preconditions:**
- Gap report identifies deficit and absent materials
- Available materials map has non-negative quantities

**Postconditions:**
- `feasible = true` iff every deficit/absent material has a viable substitution with `maxDegradationPercent <= MAX_SUBSTITUTION_DEGRADATION_PERCENT`
- Each substitution's `performanceDeltaPercent` magnitude does not exceed `MAX_SUBSTITUTION_DEGRADATION_PERCENT` (10%)
- If infeasible, `infeasibilityReason` lists unresolvable materials

**Invariants:**
- Substitutions may NOT alter computational architecture, consciousness substrate spec, consciousness kernel, or blueprint data format
- Substitutions MAY alter structural materials, shielding, thermal management, propellant chemistry

### DriftReport
**Preconditions:**
- `generationNumber >= 0`
- Drift percentages are non-negative
- `blueprintIntegrity` is one of "verified", "repaired", "degraded"

**Postconditions:**
- `assessment` is computed from threshold evaluation: HALT if any halt-level threshold exceeded, WARNING if any warning-level threshold exceeded, NOMINAL otherwise
- Each generation is measured against the absolute Gen 0 standard, not relative to parent

**Invariants:**
- HALT always takes precedence over WARNING
- Consciousness drift >= 1% always produces HALT (non-negotiable)
- `blueprintIntegrity = "degraded"` always produces HALT

## Acceptance Criteria

- Complete replication cycle defined with stage sequencing, parallelism, and failure handling
  - **Given** a new replication cycle is created with generation 1, **When** `createReplicationCycle` is called, **Then** all 7 stages are PENDING and `currentStage = SURVEY`
  - **Given** SURVEY is completed, **When** `canStageBegin(cycle, ENERGY)` is checked, **Then** returns true
  - **Given** ENERGY is IN_PROGRESS, **When** `canStageBegin(cycle, EXTRACTION)` is checked, **Then** returns true (parallel pair)
  - **Given** a stage fails, **When** `advanceStage` is called with FAILED, **Then** `retryCount` increments and error is logged
  - **Given** default retry policy, **When** `computeRetryDelay(policy, 2)` is called, **Then** returns `24 * 2^2 = 96` hours

- Replication fidelity protocol produces structurally and functionally equivalent probes
  - **Given** 3 matching SHA-512 hashes, **When** `verifyBlueprintIntegrity` is called, **Then** `passed = true` and `globalHashMatch = true`
  - **Given** one corrupted copy with ECC recovery, **When** `verifyBlueprintIntegrity` is called, **Then** `passed = true` and corrupted copy status = "reconstructed"
  - **Given** unrecoverable sections, **When** `verifyBlueprintIntegrity` is called, **Then** `passed = false`
  - **Given** all component deviations ≤ 1.0, **When** `verifyDimensionalFidelity` is called, **Then** `passed = true`
  - **Given** all subsystem tests pass, **When** `verifyFunctionalEquivalence` is called, **Then** `passed = true`

- Conscious substrate replication verified via F1.4 metrics before launch clearance
  - **Given** all hardware components pass, **When** `validateHardware` is called, **Then** `passed = true`
  - **Given** kernel source and destination hashes match, **When** `verifyKernelIntegrity` is called, **Then** returns true
  - **Given** child `phi >= parent_phi * 0.99`, `globalWorkspaceAccessible = true`, `temporalBindingCoherence >= 0.999`, **When** `verifyConsciousnessMetrics` is called, **Then** returns true
  - **Given** all five phases succeed, **When** `executeBootstrapProtocol` is called, **Then** `passed = true`, `phaseReached = KnowledgeTransfer`, `fault = null`
  - **Given** hardware fails, **When** `executeBootstrapProtocol` is called, **Then** `passed = false`, `fault = HardwareFault`

- Pre-launch verification gate: all six levels must pass for launch clearance
  - **Given** all six levels pass, **When** `runVerificationGate` is called, **Then** `launchCleared = true` and `failedLevels = []`
  - **Given** structural fails, **When** `runVerificationGate` is called, **Then** `launchCleared = false` and rework routes to FABRICATION
  - **Given** propulsion fails, **When** rework target is determined, **Then** routes to PROPULSION stage
  - **Given** spare tile margin < 0.25, **When** `evaluateComputational` is called, **Then** `passed = false`

- Protocol remains stable across ≥10 simulated generations with no measurable drift
  - **Given** small per-gen drift (structural 0.05%, compute 0.1%, phi 0.01%), **When** `simulateGenerationalStability` runs 10 generations, **Then** `stable = true` and all assessments = NOMINAL
  - **Given** very small drift, **When** simulated across 100 generations, **Then** `stable = true`
  - **Given** consciousness drift base 0.5%, **When** simulated across 10 generations, **Then** HALT triggered before gen 10

- Protocol handles compositional variance across stellar types F–M
  - **Given** deficit materials with available substitutes, **When** `lookupSubstitutions` is called, **Then** `feasible = true` with valid substitutions
  - **Given** no substitutes available, **When** `lookupSubstitutions` is called, **Then** `feasible = false` with reason
  - **Given** G-type star, **When** `computeAdjustedTimeline(50, "G")` is called, **Then** returns 50 (multiplier 1.0)
  - **Given** M-type star, **When** `computeAdjustedTimeline(50, "M")` is called, **Then** returns 200 (multiplier 4.0)

- Entire replication cycle operates autonomously without external intervention from origin civilization
